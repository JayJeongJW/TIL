# 1. 마비노기 모바일 메인 퀘스트 제작기

## 강연 주제

**숏폼 시대와 스킵 시대에서도 메인 퀘스트 스토리를 어떻게 제작할 것인가**에 대한 강연이다.

마비노기 모바일의 메인 퀘스트는 스토리팀과 내러티브 제작팀이 협업하며 제작되었다.
핵심은 이야기를 단순히 읽게 만드는 것이 아니라, **플레이를 통해 경험하게 만드는 것**이었다.

---

## 숏폼 시대의 변화

요즘 콘텐츠는 더 빠르고 짧게 소비된다.

모바일 환경은 접근성이 높지만, 지속성은 낮다.

| 구분    | 내용                            |
| ----- | ----------------------------- |
| 장점    | 핸드폰만 있으면 언제 어디서든 부담 없이 플레이 가능 |
| 단점    | 플레이 환경이 제각각이고 경쟁 콘텐츠가 많음      |
| 핵심 문제 | 접근성은 높지만 지속성은 낮음              |

따라서 모바일 메인 퀘스트는 긴 설명보다 짧은 시간 안에 몰입할 수 있는 구조가 필요하다.

---

## 메인 퀘스트 제작 방향

```text
쉽게 읽히지만 내용은 풍부하게
플레이가 이야기가 되도록
```

마비노기 세계관의 큰 틀은 유지하되, 모바일 환경에 맞게 다시 구성했다.

IP를 활용하면 긴 설명 없이도 빠르게 몰입시킬 수 있다.
다만 원작 요소를 그대로 옮기는 것이 아니라, 모바일 환경에서 이해하기 쉬운 방식으로 조정했다.

---

## 스토리 역할 변화

기존 이야기에서 플레이어가 수동적인 조력자에 가까웠다면, 모바일에서는 더 능동적인 조력자로 느껴지도록 구성했다.

```text
기존 역할: 수동적 조력자
변화 방향: 능동적 조력자
```

오마주 요소와 아이템 재창조를 통해 익숙하면서도 낯선 즐거움을 주는 방식도 사용되었다.

---

## 글자 수와 정보량 줄이기

모바일 환경에서는 대사와 정보량을 줄이는 것이 중요했다.

| 항목     | 정리                       |
| ------ | ------------------------ |
| 말풍선 길이 | 평균 18자 수준                |
| 줄 수    | 2줄을 넘지 않도록 구성            |
| 문장 구조  | 완결된 문장 사용                |
| 정보 구성  | 하나의 말풍선에 하나의 주제          |
| 목표     | 대사가 짧아도 핵심 키워드는 전달되도록 구성 |

핵심은 단순히 글자 수를 줄이는 것이 아니라, 짧은 문장 안에서도 필요한 키워드를 전달하는 것이다.

---

## UI와 시각 정보 활용

텍스트만으로 모든 내용을 설명하지 않고, 여러 전달 방식을 함께 사용했다.

| 전달 방식     | 역할                   |
| --------- | -------------------- |
| 버튼 텍스트    | 눌러서 읽는 방식, 주제 강조와 환기 |
| 선택지       | 다양한 반응 제공, 전달 목표 강조  |
| 중앙 대화 이미지 | 긴 설명 대신 시각 정보로 내용 전달 |
| 하단 메시지    | 상황 설명 보조             |
| 배경        | 공간과 맥락 전달            |

---

## 고유명사와 표현 정리

주요 타겟층은 남녀노소, 게임 초보자, 판타지 입문자까지 포함된다.

그래서 이해하기 어려운 고유명사를 줄이고 쉬운 표현으로 바꾸었다.

| 기존 표현           | 변경 방향 |
| --------------- | ----- |
| 포보르 / 포모르 / 포워르 | 마족    |
| 고대의 지혜          | 리아 파르 |
| 깨지지 않는 광석       | 아다만티움 |

---

## 무엇을 더하고 무엇을 뺄 것인가

원작의 모든 요소를 그대로 가져오는 것이 아니라, 모바일 메인 퀘스트에서 전달할 핵심 경험을 기준으로 내용을 선택했다.

| 방향     | 내용                    |
| ------ | --------------------- |
| 더한 요소  | 플레이어가 팔라딘으로 각성하는 경험   |
| 덜어낸 요소 | 기사단을 둘러싼 정치 세력 음모     |
| 덜어낸 요소 | 정령과 인간의 사랑 이야기        |
| 강조한 요소 | 빛의 기사에 대한 신념, 성취와 보람감 |

---

## 사건 중심과 인물 중심의 딜레마

메인 퀘스트 구성에서는 사건 중심과 인물 중심 사이의 딜레마가 있었다.

| 구성 방식     | 특징           |
| --------- | ------------ |
| 사건 중심     | 템포 조절이 쉬움    |
| 사건 중심의 한계 | 몰입이 방해될 수 있음 |
| 인물 중심     | 캐릭터 몰입에 유리함  |
| 인물 중심의 한계 | 분량 조정이 어려움   |

---

## 텍스트 한계 극복 방식

캐릭터가 직접 말하지 않아도 이야기를 전달할 수 있는 방법이 사용되었다.

```text
선택해서 읽기
긴 설명을 체계적으로 분산하기
발견과 해석의 즐거움 만들기
주변 살펴보기
배경을 통해 맥락 전달하기
공간을 탐색하는 재미 만들기
NPC 모션으로 감정 전달하기
전략적으로 컷신 배치하기
배경음과 효과음 활용하기
```

이야기를 하나의 방식으로 강제하기보다, 유저가 각자의 방식대로 경험할 수 있도록 구성했다.

---

## 텍스트를 플레이로, 이야기를 경험으로

이야기를 플레이 시퀀스로 만들기 위해 세 가지 축이 사용되었다.

| 축      | 내용               |
| ------ | ---------------- |
| 서사 연출  | 이야기와 캐릭터 감정 전달   |
| 플레이 템포 | 반복과 피로도 조절       |
| 플레이 테마 | 이야기 주제에 맞는 행동 설계 |

---

## 예시: 여신강림 3장 팔라딘 모험조

짧은 만남 안에서도 인물들을 매력적으로 전달하기 위해 다음 장치들이 사용되었다.

| 구성            | 목적              |
| ------------- | --------------- |
| 본격 합류 전 사전 노출 | 호의적 관계 형성       |
| 서약의 잔         | 결의를 다지는 장면 구성   |
| 도플갱어 추적 임무    | 캐릭터성 전달         |
| 탈단 구도 빌드업     | 이후 전개를 위한 사전 준비 |

---

## 플레이 템포 보완

퀘스트 내 일상 플레이가 반복되면 피로도가 생긴다.

이를 줄이기 위해 반복 사이에 환기성 플레이를 추가했다.

```text
수동 조작으로 주도성 부여
미니게임으로 세계관의 생활감 전달
연출을 통해 호기심과 긴장감 부여
반복을 끊고 새로운 경험 제공
```

같은 선택지가 등장하더라도 전후 맥락이 다르면 플레이 체감은 달라질 수 있다.

---

## 플레이 테마 선정

이야기 주제를 더 잘 느끼게 하기 위해 플레이 테마를 설정했다.

```text
선택
추적
정화
```

선택지를 통해 순간적인 몰입을 유도하고, 유저가 추적하는 주체가 되도록 장면을 배치했다.

또한 다양한 탐색 시퀀스를 추가해 사건을 경험하고 추적하는 흐름을 반복적으로 학습하게 만들었다.

---

## 선택지의 효과

가장 인상적인 효과를 낸 요소는 선택지였다.

정해진 답이 없는 딜레마형 선택지는 유저의 성향과 가치관이 반영될 수 있게 했다.

또한 다른 선택지의 보상도 별도 획득 경로를 제공하여, 선택에 따른 박탈감을 줄였다.

```text
정답 없는 선택지
→ 유저 성향과 가치관 반영
→ 자발적 토론 발생
→ 재플레이 확장
```

---

## 의도한 효과와 의도하지 않은 효과

플레이 변주를 추가했을 때 의도한 효과와 의도하지 않은 효과가 모두 있었다.

| 구분         | 내용                     |
| ---------- | ---------------------- |
| 의도한 효과     | 유사 플레이 반복 피로 감소        |
| 의도한 효과     | 경험 환기 유도               |
| 의도하지 않은 효과 | 서사적 긴장감이 높은 순간에 몰입이 깨짐 |

서사 긴장감이 높은 구간에서는 짧은 환기보다 몰입 유지가 우선이라는 결과가 있었다.

---

## 분량과 편의성 개선

메인 퀘스트는 플레이어가 부담 없이 몰입할 수 있도록 적정한 플레이 타임 설계가 필요했다.

먼저 이야기를 담는 그릇의 크기를 정하고, 그 안에서 이야기 밀도를 유지하는 방식이 사용되었다.

```text
1. 시퀀스마다 규모 등급 매기기
2. 등급 비율 정하기
3. 규모와 일정 예측하기
4. 기준을 먼저 적용하기
```

이 과정을 통해 비슷한 수준의 경험을 제공하고, 협업 효율도 높일 수 있었다.

---

## 스킵에 대한 접근

스킵은 자연스러운 플레이 방식 중 하나로 보았다.

| 유저 유형      |
| ---------- |
| 전부 스킵하는 유저 |
| 일부 스킵하는 유저 |
| 스킵하지 않는 유저 |

스킵을 막는 것이 아니라, 스킵하려던 순간에도 멈칫하게 만드는 이야기 구성이 목표였다.

```text
스킵해도 이해 가능하게
스킵하지 않으면 더 깊게 경험하게
스킵하려던 순간을 멈칫하게
```

# 2. SPICA: 쌍성에서 우주까지 게임 서버의 진화

## 강연 주제

루트 슈터, 멀티플레이 환경에서 많은 몬스터와 액터를 처리하기 위한 서버 구조에 대한 강연이다.

몬스터 수가 증가하면 서버 처리량도 함께 증가한다.
SPICA는 서버와 클라이언트가 하나처럼 동작하도록 구성된 구조로 설명되었다.

---

## SPICA의 기본 구성

SPICA는 크게 다음 요소로 구성된다.

| 요소        | 내용                  |
| --------- | ------------------- |
| 서브 네트워크   | 특정 액터나 범위를 별도로 처리   |
| 액터 커맨드    | 몬스터 같은 개별 단위의 동작 명령 |
| SPICA RPC | 서버와 액터 간 호출 구조      |

액터는 몬스터와 같은 개별 단위로 볼 수 있다.

---

## 라이브 서비스를 향한 구조

라이브 환경을 위해 멀티 월드 구조가 도입되었다.

```text
멀티 월드
→ 한 프로세스에 여러 월드를 띄우는 구조
```

추가로 다음 요소들이 지원되었다.

```text
코어 고정
로드맵 단계화
월드별 로그 파일 지원
```

---

## 빠른 맵 이동

같은 종류의 맵 이동을 빠르게 처리하기 위해 불필요한 unload / load를 스킵하고, 이동 위치를 미리 스트리밍하는 방식이 사용되었다.

```text
기존 방식
unload → load → 이동

개선 방향
불필요한 unload/load 스킵
이동 위치 미리 스트리밍
```

---

## SPICA 설정 조건

SPICA에는 시작 조건, 위임 조건, 반환 조건, 안정장치가 있었다.

| 구분    | 내용                     |
| ----- | ---------------------- |
| 시작 조건 | 윈도우 환경, CPU 점수 기준치 이상  |
| 위임 조건 | 150m 이내 일반 몬스터         |
| 반환 조건 | 150m 범위 이탈, 피격, 서브넷 파괴 |
| 안정장치  | 서브넷 파괴 실시간 on/off 킬스위치 |

---

## 라이브 결과

미션 완료 기준으로 기록했을 때, 1인 기준 약 39% 동작했다.

서버 부하는 약 5~10% 감소했다.

```text
1인 기준 동작 비율: 39%
서버 부하 감소: 5~10%
SPICA 관련 이슈: 0건
킬스위치 사용: 0회
```

광역 스킬 위주의 플레이 스타일로 인해 몬스터가 점점 더 빨리 피격되는 경향도 확인되었다.

---

## SPICA Satellite

SPICA Satellite는 별도 프로세스를 띄워서 일을 시키고, 작업이 끝나면 내리는 구조다.

```text
필요할 때 프로세스 실행
→ 작업 처리
→ 작업 완료 후 종료
```

---

## SPICA Cosmos

한 서버가 아니라 여러 서버를 연결해야 하는 상황을 위한 확장 구조다.

퍼스트 디센던트는 최대 4인 플레이 구조였지만, 수십 명 또는 수백 명이 함께 플레이하는 구조라면 하나의 서버만으로 처리하기 어렵다.

SPICA Cosmos는 서버 간 정보를 주고받고, 권한을 가진 서버가 정보를 관리하며, 필요할 때 권한을 넘길 수 있는 구조로 설명되었다.

---

## SPICA Cosmos 구조

SPICA Cosmos에서는 새로운 NetDriver를 생성하고 다른 Dedicated Server에 접속한다.

연결은 한 방향으로만 진행된다.

```text
새로운 NetDriver 생성
→ 다른 Dedicated Server에 접속
→ 한 방향 연결
→ SPICA RPC 그대로 사용
```

액터 생성 시점에 소유한 NetDriver를 세팅하고, 플레이어 캐릭터 정보가 있을 때 자연스러운 이동이 가능하도록 구성된다.

---

## Cosmos Assistant

Cosmos Assistant는 핸드오버 전용 모드로 설명되었다.

SPICA Assistant와 동일한 역할을 수행하며, 서버 간 자연스러운 이동과 권한 이전을 돕는 구조로 정리된다.

---

## SPICA Cosmos 활용 가능성

SPICA Cosmos는 다양한 방식으로 확장이 가능하다.


# 3. 시네마틱 제작 파이프라인에서 실험한 AI 리뷰

## 강연 주제

시네마틱 제작 파이프라인에서 AI 리뷰를 실험한 내용을 다룬 강연이다.

발표 내용은 실무 적용 완료 사례라기보다 R&D 학습 기록에 가까운 형태로 소개되었다.

---

## 문제 인식: 리뷰의 시간 전쟁

시네마틱이나 애니메이션 제작 과정에서는 리뷰 시간이 부족하다.

작업자는 같은 장면을 오래 보다 보면 이상한 움직임에도 익숙해질 수 있다.

```text
작업자 몰입
→ 이상한 움직임에도 익숙해짐
→ 제3의 눈 필요
```

이런 상황에서 AI가 일부 객관적인 체크를 도울 수 있는지 실험했다.

---

## AI와 사람의 역할 비율

강연에서는 AI와 사람의 역할을 다음처럼 구분했다.

| 구분 | 내용               |
| -- | ---------------- |
| AI | 약 60%의 객관적 체크 가능 |
| 사람 | 약 40%의 개인 판단 필요  |

가능성은 명확했지만, 이 상태 그대로 작업자에게 전달하기에는 문제가 있었다.

---

## AI 자유 리뷰의 문제

AI에게 자유롭게 리뷰를 맡기면 피드백이 나오지만, 작업자가 바로 활용하기 어려운 경우가 있었다.

예시로는 다음과 같은 형태다.

```text
총기의 물리적인 리액션이 조금 더 섬세하게 조율될 필요가 있습니다.
```

이런 리뷰는 그럴듯하지만, 실제 작업자 입장에서는 무엇을 어떻게 수정해야 하는지 알기 어렵다.

---

## AI 자유 리뷰의 한계

AI 자유 리뷰에는 다음 한계가 있었다.

```text
결과가 산발적임
분석 기준이 일정하지 않음
정확도를 판단하기 어려움
수정 지시로 사용하기 어려움
```

그래서 분석을 리뷰로 바꾸는 플랫폼이 필요했다.

---

## 리뷰 허브

AI가 자유롭게 말하는 방식이 아니라, 정해진 항목을 기준으로 리뷰하도록 만드는 플랫폼이 필요했다.

리뷰 허브에서는 영역별 체크 항목을 나눴다.

| 영역      | 항목 수 |
| ------- | ---: |
| Anim 전반 |   9개 |
| Anim 후반 |   9개 |
| FX      |  12개 |

---

## AI는 문지기가 아니다

AI는 최종 승인자가 아니라, 측정받는 대상이다.

AI의 리뷰 결과도 다시 측정되어야 하며, 그 측정의 주체는 컨펌 권한을 가진 사람이다.

```text
AI는 문지기가 아니다.
AI 자체도 측정받는 대상이다.
측정의 주체는 컨펌권자다.
```

---

## 입력 데이터와 컨텍스트 문제

애니메이션 리뷰는 영상만으로 부족한 경우가 있었다.

FBX 데이터를 함께 넣으면 더 많은 정보를 제공할 수 있지만, AI가 영상을 오히려 덜 보는 문제가 발생했다.

따라서 AI가 전제 데이터 설정을 제대로 읽는 구조가 필요했다.

```text
영상만 입력
→ 정보 부족

FBX 추가 입력
→ AI가 영상을 덜 보는 문제 발생

필요한 것
→ AI가 전제 데이터와 컨텍스트를 함께 읽는 구조
```

## 레퍼런스 영상의 역할

레퍼런스 영상은 최종 결과물의 천장이 아니라 커뮤니케이션 수단으로 설명되었다.

즉, 레퍼런스는 똑같이 만들기 위한 기준이 아니라, 의도와 방향을 맞추기 위한 도구다.

작업 영상의 프레임을 참고해 같은 캐릭터, 같은 컨텍스트에서 출발하는 레퍼런스를 생성하는 방식도 언급되었다.

# 4. ⭐ 퍼스트 디센던트 오토 플레이 테스트 툴 제작기

> **2일차 정리에서 가장 중요하게 표시한 세션**
> 최적화와 성능 검증을 위해, 사람이 직접 반복 플레이하지 않고도 게임을 자동으로 플레이하며 성능 데이터를 수집하는 도구를 제작한 사례다.

---

## ⭐ 세션 한 줄 요약

```text
최적화 검증에는 반복 테스트가 필요하고,
반복 테스트에는 동일 조건의 자동 플레이 수집 도구가 필요하다.
```

---

## 4-1. M1 Auto Play Test 소개

최적화 작업은 반복 테스트가 필수다.

```text
문제 지점 파악
→ 최적화 진행
→ 결과 확인
```

이 과정을 정확히 비교하려면 동일한 조건으로 여러 번 플레이해야 한다.

하지만 실제 테스트 환경은 플랫폼과 옵션 조합이 많아 사람이 직접 반복하기 어렵다.

---

## 테스트 조합 문제

테스트해야 할 조합은 매우 많았다.

| 플랫폼 / 환경          | 조합 수 |
| ----------------- | ---: |
| PS5               |  7가지 |
| Xbox Series X / S | 14가지 |
| Windows           |  9가지 |
| 총합                | 30가지 |

이 많은 조합을 사람이 직접 동일 조건으로 반복 플레이하는 것은 부담이 크다.

---

## M1 Auto Play Test의 목적

M1 Auto Play Test는 게임을 직접 플레이하며 성능 데이터를 수집하는 자동 툴이다.

```text
사람이 직접 반복 플레이
→ 조건 유지가 어렵고 시간이 오래 걸림

자동 플레이 테스트 툴
→ 동일 경로 반복
→ 성능 데이터 자동 수집
→ 결과 비교 가능
```

---

## 4-2. 핵심 철학

이 툴의 핵심 철학은 다음 문장으로 정리된다.

```text
변수가 많은 게임 앞에서
완벽한 재현보다 중요한 것은
멈추지 않는 것이다.
```

자동 플레이가 항상 완벽한 플레이를 할 수는 없다.

하지만 성능 데이터를 수집하는 목적에서는 중간에 멈추지 않고 계속 진행되는 것이 더 중요하다.

---

## 4-3. 핵심 기술 구조

강연에서는 핵심 기술 딥다이브로 경로 탐색과 멈춤 감지 조건이 소개되었다.

자동 플레이 중 캐릭터가 끼거나 목표 지점에 도달하지 못할 수 있기 때문에, 이를 감지하고 탈출하는 조건이 필요했다.

---

## 멈춤 감지 조건

자동 플레이가 멈춘 상태인지 판단하기 위해 다음 조건을 사용했다.

| 조건명                     | 의미                            |
| ----------------------- | ----------------------------- |
| `NoMoveButAccelTime`    | 이동은 없지만 가속도가 10cm/s 미만인 누적 시간 |
| `NonDecreasingDistTime` | 타겟 지점까지 남은 거리가 줄어들지 않은 누적 시간  |

두 조건을 만족하면 텔레포트로 탈출한다.

```text
이동 없음
+ 목표 지점까지 거리 감소 없음
= 멈춤 상태로 판단
→ 텔레포트로 탈출
→ 테스트 계속 진행
```

---

## 4-4. 자동 플레이 흐름

```text
자동 플레이 시작
→ 지정 경로 이동
→ 성능 데이터 수집
→ 이동 상태 확인
→ 정상 이동이면 계속 진행
→ 멈춤 감지 시 텔레포트 탈출
→ 테스트 재개
→ 결과 CSV 기록
→ 리포트 생성
```

---

## 4-5. 데이터 수집

M1 Auto Play Test는 매 프레임마다 성능 데이터를 수집한다.

수집한 데이터는 CSV로 자동 기록된다.

| 수집 항목 예시   |
| ---------- |
| 메모리 사용량    |
| 렌더링 지표     |
| 프레임 관련 지표  |
| 성능 비교용 데이터 |

PC 플랫폼에서는 정기적으로 무인 수집도 진행 중이었다.

---

## 4-6. 자동 결과 리포팅

수집된 데이터는 결과 리포팅으로 이어진다.

같은 경로를 기준으로 결과를 비교하고, 색상 변화를 통해 주의 깊게 봐야 할 대상을 구분할 수 있었다.

```text
같은 경로
→ 다른 색상으로 결과 표시
→ 노랑 / 빨강 / 흰색 구간은 주의 깊게 확인
→ 최적화 전후 변화를 시각적으로 비교
```

---

## 4-7. 성과 검증

성능 오버헤드 검증과 데이터 수집을 통해 실제 최적화 협업에 활용되었다.

렌더러 스레드 타임에서 일관된 개선을 확인할 수 있었고, 색상 변화가 최적화 효과를 시각적으로 보여주는 방식으로 사용되었다.

---

# 5. 블루 아카이브 보스 개선 포스트모템

## 강연 주제

**「랜덤에서 재미를 추구하면 안 되는 걸까? Re: 트라이부터 시작하는 보스전」**

블루 아카이브의 총력전 콘텐츠에 등장하는 보스 개선 사례를 다룬 강연이다.

이 발표는 2025년에 진행된 블루 아카이브의 보스 개선을 중심으로, 왜 개선을 제안하게 되었는지, 어떤 전투 트렌드를 참고했는지, 실제로 어떤 보스들이 어떻게 수정되었는지를 설명했다.

---

## 발표에서 다룬 핵심 범위

```text
블루 아카이브 전투 구조
→ 택틱 중심 전투
→ 랜덤 요소로 인한 반복 리트라이
→ 이차원 게임 전투 트렌드 분석
→ 보스별 랜덤 요소 개선
→ 개선 이후 결과 확인
→ 전투와 캐릭터 매력의 관계 정리
```

---

## 5. 블루 아카이브의 전투 구조

블루 아카이브의 전투는 유저가 모든 행동을 직접 조작하는 방식은 아니다.

기본 전투는 자동으로 진행되고, 유저는 제한된 자원인 코스트를 사용해 EX 스킬을 선택적으로 사용한다.

| 요소    | 내용                          |
| ----- | --------------------------- |
| 기본 전투 | 학생들이 자동으로 전투 진행             |
| 유저 개입 | 코스트를 사용해 EX 스킬 발동           |
| 전투 공략 | 파티 조합, 스킬 순서, 타이밍이 중요       |
| 공유 방식 | 공략 영상이나 택틱을 보고 따라 하는 문화가 형성 |

---

## 택틱 중심 전투

블루 아카이브에서는 동일한 파티 조합과 스펙을 맞춘 뒤, 정해진 스킬 순서와 타이밍을 따라 하면 같은 보스를 클리어할 수 있는 형태가 많다.

이런 공략 흐름은 보통 **택틱**으로 공유된다.

```text
동일한 파티 조합
+ 비슷한 스펙
+ 정해진 스킬 순서
+ 정해진 타이밍
= 보스 클리어 가능
```

유저 입장에서는 공략을 따라 할 수 있다는 장점이 있지만, 택틱을 정확히 따라도 랜덤 요소 때문에 실패하는 경우가 있었다.

---

## 반복 리트라이가 발생하는 구조

택틱을 성공시키기 위해서는 여러 조건이 맞아야 했다.

발표에서는 반복 리트라이를 유발하는 요소를 다음과 같이 정리했다.

```text
패턴
치명타
회피
그로기
```

이 중 하나라도 원하는 흐름대로 나오지 않으면 전투를 다시 시작해야 하는 상황이 생겼다.

---

## 랜덤 요소와 경쟁 콘텐츠

일부 보스는 보스 자체가 보유한 랜덤 요소가 많았다.

또한 총력전은 경쟁 콘텐츠이기 때문에, 상위 랭크를 노리는 유저는 이런 랜덤 요소까지 돌파해야 했다.

```text
보스의 랜덤 요소
+ 경쟁 콘텐츠 구조
= 상위 랭크를 위해 반복 리트라이 필요
```

결과적으로 랜덤은 재미보다 피로와 스트레스로 작동하는 경우가 있었다.

---

## 개선 제안의 계기

개선 이전에는 특정 보스 전투에서 운이 나쁘면 같은 흐름을 오랫동안 반복해야 했다.

발표에서는 개선 이전 전투 흐름을 다음과 같은 형태로 설명했다.

```text
의도한 쪽에 본체가 등장했는가?
→ 패턴이 유리하게 나왔는가?
→ 유의미한 치명타가 발생했는가?
→ 캐릭터 이동에 문제가 없었는가?
→ 그 외 의도대로 되었는가?
→ 택틱 성공
```

중간 조건 중 하나라도 어긋나면 리트라이가 발생했다.

라이브 기간이 길어질수록 더 많은 학생을 편성할 수 있게 되었지만, 동시에 더 운이 좋은 유저가 높은 랭크를 기록하는 상황이 반복될 수 있었다.

---

## 전투 피로도 문제

라이브가 길어질수록 전투 콘텐츠의 피로도는 리텐션에 영향을 줄 수 있다.

특히 신규 난이도인 **루나틱**이 출시되면서 기존 난이도보다 더 많은 파티 운용이 필요해졌고, 더 많은 파티 운용은 더 많은 랜덤과의 싸움을 의미했다.

```text
더 많은 파티 운용
→ 더 많은 랜덤 요소 발생
→ 더 많은 리트라이
→ 전투 피로도 증가
```

그래서 보스가 보유한 랜덤성이 주는 피로도를 기준으로 개선 규모를 정했다.

---

## 이차원 게임 전투 트렌드 분석

보스 개선을 진행하기 전, 최근 이차원 게임의 전투 트렌드를 분석했다.

발표에서 주목한 흐름은 크게 두 가지였다.

```text
버스트 윈도우
수동적 랜덤의 축소
```

---

## 버스트 윈도우

버스트 윈도우는 짧은 시간 동안 공격 능력을 극대화해 폭발적인 피해를 입히는 전략적 시간대를 의미한다.

이차원 게임에서는 제한된 버프 시간, 디버프 시간, 제한된 자원을 어떻게 운용하는지에 따라 데미지 편차가 크게 발생한다.

| 구성 요소           | 설명                        |
| --------------- | ------------------------- |
| 제한된 버프 / 디버프 시간 | 정해진 시간 안에 최대 화력을 넣어야 함    |
| 제한된 자원          | 스킬이나 에너지를 언제 사용할지 선택해야 함  |
| 전략적 운용          | 버프, 디버프, 딜러 스킬을 맞춰 사용해야 함 |

버스트 윈도우는 유저가 선택하는 전투 타이밍을 만든다는 점에서 중요한 전투 구조로 다뤄졌다.

---

## 수동적 랜덤의 축소

이차원 게임에서는 유저가 통제하기 어려운 랜덤 요소를 긍정적으로 설계하기 어렵다.

이미 전투를 진행하기 위해 여러 허들을 넘는 구조가 있기 때문에, 유저가 통제할 수 없는 랜덤이 추가되면 재미보다 피로로 이어지기 쉽다.

예시로는 다음과 같은 상황이 있다.

```text
디버프가 Miss가 되는 경우
그로기가 가능한 공격이 Miss가 되는 경우
Miss 때문에 공격 준비가 되지 않는 경우
보스가 갑자기 무적 패턴을 사용하는 경우
```

이런 상황은 전투 사이클을 끊거나, 유저가 준비한 택틱을 의미 없게 만들 수 있다.

---

## 블루 아카이브에서 이미 사용 중인 수동적 랜덤

블루 아카이브는 이미 명중치, 회피치 등 수동적 랜덤과 연결된 스탯을 활용하고 있었다.

다만 이런 스탯은 학생의 능력과 연결되어 있기 때문에, 학생을 직접 수정하기보다는 보스 쪽을 수정하는 방식으로 간접적인 개선을 진행했다.

```text
명중치
회피치
치명 저항
안정치
보스 패턴
```

---

## 보스 개선 방향

분석을 바탕으로 블루 아카이브의 보스 전투 개선은 다음 방향으로 진행되었다.

| 개선 구분            | 대상 보스            |
| ---------------- | ---------------- |
| 랜덤 스탯 개선         | 그레고리오            |
| 랜덤 스탯 + 랜덤 패턴 개선 | 호드, 고즈, 카이텐저, 비나 |
| 랜덤 패턴 개선         | 쿠로카게, 페로로지라      |

개선의 주요 목표는 다음과 같았다.

```text
리트라이를 줄이는 것
기존 공략을 크게 바꾸지 않는 것
전투 피로도를 줄이는 것
```

---

## 랜덤 스탯 개선: 그레고리오 / 카이텐저

그레고리오와 카이텐저는 적의 회피 수치를 줄이는 방향으로 개선되었다.

| 보스    | 개선 내용          |
| ----- | -------------- |
| 그레고리오 | 적의 회피 수치 감소    |
| 카이텐저  | 일부 적의 회피 수치 감소 |

그레고리오의 경우 루나틱 클리어 파티 중 명중치가 낮아 활약하지 못하던 학생이 다시 활용될 수 있도록 조정되었다.

카이텐저는 다른 적보다 회피 수치가 높은 대상 때문에 리트라이가 발생할 가능성이 있었고, 해당 부분을 줄이는 방향으로 개선되었다.

---

## 랜덤 스탯 개선: 비나

비나의 경우 특정 디버프와 엄폐 관련 요소가 전투 결과에 영향을 주었다.

기존에는 같은 상황에서도 데미지 편차를 만들던 명중치와 안정치 관련 요소가 있었고, 엄폐 성공 여부에 따라 엄폐물 파괴와 학생 이동이 달라져 리트라이가 발생할 수 있었다.

개선 방향은 다음과 같다.

```text
명중치 / 안정치 디버프 제거
확률적 엄폐로 인한 리트라이 제거
엄폐 학생이 엄폐 자체로 이점을 얻을 수 있도록 수정
```

---

## 랜덤 스탯 개선: 호드 / 고즈

호드와 고즈는 치명타 관련 리트라이를 줄이는 방향으로 개선되었다.

| 보스 | 기존 문제                       | 개선 방향                 |
| -- | --------------------------- | --------------------- |
| 호드 | 치명 저항이 높아 치명타 관련 리트라이 발생 가능 | 기믹 수행 시 치명 저항 수치 감소   |
| 고즈 | 치명 저항과 리트라이 강도가 높음          | 적의 회피 수치와 치명 저항 수치 감소 |

고즈는 특히 리트라이 강도가 높았던 보스였기 때문에, 전반적으로 적의 회피 수치와 치명 저항 수치를 낮추는 방향으로 수정되었다.

---

## 랜덤 패턴 개선: 카이텐저 / 비나 / 호드

일부 보스는 특정 스킬을 랜덤 타이밍에 사용하거나, 난이도에 따라 스킬 사용 흐름을 예측하기 어려웠다.

이를 개선하기 위해 난이도와 관계없이 중요한 스킬을 고정된 타이밍에 사용하도록 수정했다.

| 보스   | 기존 문제                      | 개선 방향              |
| ---- | -------------------------- | ------------------ |
| 카이텐저 | 랜덤 타이밍에 방패 스킬 사용           | 고정된 타이밍에 핵심 스킬 사용  |
| 비나   | 하위 난이도에서 대부분의 스킬을 확률적으로 사용 | 주요 스킬 타이밍 고정       |
| 호드   | 첫 스킬에 따라 유리한 상황이 달라짐       | 지정된 타이밍에 지정된 스킬 사용 |

기존보다 전투 난이도가 올라갈 수는 있지만, 리트라이 자체를 줄이는 것이 먼저라고 판단해 난이도를 조정했다.

---

## 랜덤 패턴 개선: 페로로지라

페로로지라는 전투 환경으로 인해 학생 이동이 랜덤해지는 문제가 있었다.

기존에는 랜덤한 이동 로직과 특정 학생의 생존 문제가 맞물려 많은 리트라이가 필요했다.

개선 방향은 다음과 같다.

```text
학생의 활용도를 유지하는 것보다 리트라이 요소 감소를 우선
특정 타겟 구조를 변경
파티 편성 선택지에서 문제가 되는 요소를 줄이도록 수정
```

즉, 해당 학생을 계속 활용하게 두는 것보다 리트라이 피로를 줄이는 쪽을 선택했다.

---

## 랜덤 패턴 개선: 고즈

고즈는 본체 위치와 스킬 회피 위치가 랜덤하게 결정되는 구조가 있었다.

기존에는 본체가 원하는 위치에 등장할 때까지 리트라이하거나, 스킬을 피하지 않아도 되는 흐름이 나올 때까지 리트라이하는 상황이 반복되었다.

개선 방향은 다음과 같다.

```text
본체가 나타나는 위치 순서 고정
확률적인 패턴 사용 제거
지정된 타이밍에 지정된 스킬 사용
```

기존 마술사 콘셉트를 일부 포기하더라도, 전투 피로도를 줄이는 데 집중했다.

---

## 고즈의 그로기 구조 개선

고즈는 그로기와 관련된 랜덤성도 함께 개선되었다.

기존에는 그로기 타이밍과 본체 위치가 맞물리면서 랜덤성이 커질 수 있었다.

개선 방향은 다음과 같다.

```text
그로기 누적 방식 변경
그로기 타이밍이 일정해지도록 수정
본체 위치를 주기적으로 랜덤하게 변경
일부 스킬에 Cost 회복 추가
버스트를 노릴 수 있는 구조 제공
```

즉, 단순히 랜덤을 제거하는 것에서 끝나지 않고, 랜덤 대신 전투 흐름에 변곡점을 줄 수 있는 구조를 추가했다.

---

## 랜덤 패턴 개선: 쿠로카게

쿠로카게는 랜덤 패턴과 대상 선택 구조가 문제로 다뤄졌다.

기존에는 초음파와 관련된 패턴에서 내부 로직이 랜덤하게 동작해, 유저 입장에서 해법을 알아도 실제 전투 흐름이 꼬일 수 있었다.

개선 방향은 다음과 같다.

```text
대상 선택 규칙 정리
랜덤하게 느껴지던 대상 지정 방식 개선
패턴 해석 과정의 부담 감소
```

쿠로카게는 단순히 랜덤을 줄이는 것뿐 아니라, 유저가 패턴을 이해하고 대응할 수 있도록 구조를 정리하는 방향으로 개선되었다.

---

## 개선 이후 반응

보스 개선 이후 커뮤니티에서는 긍정적인 반응이 관측되었다.

특히 고즈와 그레고리오 관련 개선은 유저들이 바로 체감할 수 있는 형태였다.

```text
패턴 고정
회피 정상화
전투 피로도 감소
개발팀이 문제를 인지하고 있다는 반응
```

발표에서는 개선 이후 전투 피로도와 관련해 긍정적인 동향이 관측되었다고 정리했다.

---

## 실제 플레이 데이터 결과

실제 플레이 데이터에서도 평균 클리어 타임과 전투 실패 비율이 유의미하게 변화했다.

### 고즈

| 구분        |    인세인 |    토먼트 |
| --------- | -----: | -----: |
| 평균 클리어 타임 | 20% 감소 | 34% 감소 |
| 전투 실패 비율  | 31% 감소 | 42% 감소 |

### 쿠로카게

| 구분        |    인세인 |    토먼트 |
| --------- | -----: | -----: |
| 평균 클리어 타임 | 17% 감소 | 24% 감소 |
| 전투 실패 비율  | 27% 감소 | 34% 감소 |

데이터상으로도 리트라이가 줄어들었고, 평균 클리어 타임과 전투 실패 비율이 감소한 것으로 확인되었다.

---

## 개선의 성공과 남은 문제

개선 이후 전투 피로도와 관련해서는 긍정적인 변화가 있었다.

하지만 보스 개선이 모든 문제를 해결한 것은 아니었다.

기존 공략과 충돌하거나, 새로운 난이도와 보스 구조에서 다시 조정이 필요한 부분도 남았다.

```text
기존 공략과 충돌하는 요소 발생 가능
새로운 난이도에서 전투 피로도 재발 가능
보스 기억과 이해 난이도 문제
경쟁 콘텐츠 특성상 피로도 완전 제거는 어려움
```

따라서 이후에도 신규 난이도와 보스 설계에서 랜덤 요소와 전투 피로도를 계속 고려해야 하는 흐름으로 정리되었다.

---

## 전투는 캐릭터의 매력을 보여주는 무대

강연의 결론은 다음 문장으로 정리할 수 있다.

```text
전투는 캐릭터의 매력을 보여주는 무대다.
```

이차원 게임에서는 유저가 캐릭터를 획득하고, 육성하고, 애정을 쌓은 뒤 전투에 진입한다.

따라서 전투가 지나치게 불합리하거나 스트레스를 유발하면, 캐릭터의 매력을 보여주는 무대가 아니라 피로를 주는 장치가 될 수 있다.

---

## 이차원 게임 전투의 방향

발표에서는 최근 이차원 게임 전투가 다음 방향으로 이동하고 있다고 정리했다.

```text
사라져가는 Miss
강화되는 버스트 윈도우
캐릭터와의 교감 유지
```

유저가 캐릭터와 교감하고 애정을 쌓는 구조가 중요하기 때문에, 전투는 그 경험을 해치지 않는 방향으로 설계되어야 한다.

---

## 랜덤에 대한 결론

발표의 핵심 질문은 다음과 같았다.

```text
랜덤에서 재미를 추구하면 안 되는 걸까?
```

결론은 랜덤 자체가 항상 나쁜 것은 아니지만, 유저가 통제할 수 없는 수동적 랜덤은 이차원 게임 전투에서 피로로 작동하기 쉽다는 것이다.

정리하면 다음과 같다.

| 구분      | 설명                                       |
| ------- | ---------------------------------------- |
| 수동적 랜덤  | 유저가 회피하거나 통제하기 어려운 랜덤                    |
| 능동적 플레이 | 유저가 예측하고 대응하고 조정할 수 있는 플레이               |
| 개선 방향   | 보이지 않는 불쾌한 랜덤을 줄이고, 유저가 대응 가능한 전투 상황을 제공 |

전투 상황을 변화시키는 랜덤은 재미를 줄 수 있다.
하지만 유저가 통제할 수 없는 변수를 통해 재미를 만들려는 방식은 이차원 게임에서는 주의가 필요하다는 내용으로 정리된다.

## 블루 아카이브 보스 개선 핵심 정리

```text
문제
→ 택틱을 따라 해도 랜덤 요소 때문에 반복 리트라이가 발생함
→ 경쟁 콘텐츠 구조에서는 랜덤을 돌파해야 상위 랭크를 노릴 수 있음
→ 랜덤이 재미보다 피로와 스트레스로 작동하는 경우가 있었음

분석
→ 최근 이차원 게임 전투 트렌드 확인
→ 버스트 윈도우와 수동적 랜덤 축소에 주목
→ 블루 아카이브의 명중, 회피, 치명 저항, 보스 패턴을 점검

개선
→ 회피 수치 감소
→ 치명 저항 수치 감소
→ 확률적 패턴 사용 제거
→ 핵심 스킬 타이밍 고정
→ 그로기 구조 조정
→ 보스별 랜덤 패턴 개선

결과
→ 평균 클리어 타임 감소
→ 전투 실패 비율 감소
→ 전투 피로도 관련 긍정 반응 확인

결론
→ 전투는 캐릭터의 매력을 보여주는 무대
→ 유저가 통제할 수 없는 랜덤은 피로로 이어질 수 있음
→ 랜덤을 줄이고, 대응 가능한 전투 흐름을 만드는 방향으로 개선
```
