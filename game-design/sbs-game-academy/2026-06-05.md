## 시뮬레이션 RPG 시스템 기획서 - 데이터 테이블 정리

## 오늘 진행한 내용

오늘은 시뮬레이션 RPG 시스템 기획서에서 `데이터 테이블` 파트를 정리했다.

이전까지는 유닛, 지형, 이동력, 전투 보정 같은 내용을 문장으로 설명하는 방식이 많았다.
하지만 시스템 기획서에서는 단순히 “이런 규칙이 있다”라고 쓰는 것만으로는 부족하다는 것을 느꼈다.

실제 개발에서 사용하려면 규칙이 데이터로 관리될 수 있어야 한다.
그래서 이번에는 기획서에 작성한 내용을 기준으로 `지형 데이터 테이블`과 `유닛 데이터 테이블`을 나누어 정리했다.

---

## 데이터 테이블을 사용하는 이유

데이터 테이블은 게임에 들어가는 수치를 표로 관리하기 위한 방식이다.

예를 들어 지형마다 이동력 소모량이 다르고, 유닛마다 체력이나 공격력, 이동력 같은 수치가 다르다.
이런 값을 문장으로만 적어두면 나중에 수정하거나 비교하기 어렵다.

반대로 데이터 테이블로 정리하면 어떤 값이 어디에 쓰이는지 한눈에 확인할 수 있다.

이번 기획서에서는 데이터 테이블을 다음과 같이 정의했다.

1. 유닛과 지형의 수치를 관리하기 위한 표
2. 이동력, 능력치, 지형효과 값을 데이터로 정리하는 공간
3. 수치를 변경할 때 기획서 문장을 고치는 것이 아니라 테이블 값을 수정하는 방식
4. 개발자가 실제로 참고할 수 있는 기준표

---

## 데이터 테이블 작성 기준

데이터 테이블을 만들 때는 단순히 표를 예쁘게 만드는 것이 중요한 게 아니었다.
각 컬럼이 어떤 값을 의미하는지 명확해야 했다.

이번에 정리한 기준은 다음과 같다.

1. `번호`는 데이터를 구분하기 위한 고유 값으로 사용한다.
2. `타입`은 데이터의 큰 분류를 의미한다.
3. `명칭`은 실제 게임 화면에 표시되는 이름이다.
4. 이동력, 공격력, 방어력 같은 값은 숫자로 입력한다.
5. 적용 여부는 `TRUE / FALSE`로 구분한다.
6. 사용하지 않는 값은 빈칸으로 두지 않고 `없음` 또는 `-1`로 입력한다.

특히 `-1`은 이동 불가처럼 규칙상 사용할 수 없는 값을 표시할 때 사용했다.
빈칸으로 두면 값이 빠진 것인지, 일부러 사용하지 않는 것인지 구분하기 어렵기 때문이다.

## 지형 데이터 테이블

지형 데이터 테이블은 지형별 이동력 소모량과 전투 보정 값을 관리하기 위한 표이다.

이번 기획서에서는 평지, 다리, 숲, 사막, 물, 산, 설산, 늪지대를 지형으로 사용했다.
각 지형은 병종별로 이동력 소모량이 다르게 적용된다.

예를 들어 숲은 보병에게는 이동하기 쉬울 수 있지만, 기병에게는 이동이 불리하게 작동할 수 있다.
물 지형은 수병에게 유리하고, 산은 일반 지상 병종에게 진입이 제한될 수 있다.

지형 데이터 테이블에는 다음 항목을 넣었다.

| 항목     | 설명                          |
| ------ | --------------------------- |
| 번호     | 지형을 구분하기 위한 고유 ID           |
| 타입     | 지형의 큰 분류                    |
| 명칭     | 게임 화면에 표시되는 지형 이름           |
| 보병이동   | 보병이 해당 지형을 이동할 때 소모하는 이동력   |
| 기병이동   | 기병이 해당 지형을 이동할 때 소모하는 이동력   |
| 비병이동   | 비병이 해당 지형을 이동할 때 소모하는 이동력   |
| 수병이동   | 수병이 해당 지형을 이동할 때 소모하는 이동력   |
| 스키병이동  | 스키병이 해당 지형을 이동할 때 소모하는 이동력  |
| 조사병단이동 | 조사병단이 해당 지형을 이동할 때 소모하는 이동력 |
| 효과종류   | 지형이 전투에 주는 보정 종류            |
| 효과수치   | 지형효과가 적용되는 수치               |
| 공격통과   | 해당 지형을 사이에 두고 공격이 가능한지 여부   |

### 지형 데이터 테이블을 정리하면서 배운 점

지형은 단순히 배경 이미지가 아니라 전투 규칙에 직접 영향을 주는 요소라는 것을 알게 되었다.

처음에는 지형을 “평지, 숲, 물, 산”처럼 종류만 나누면 된다고 생각했다.
하지만 실제 시스템으로 만들려면 각 지형이 어떤 병종에게 어떤 이동력 소모를 주는지까지 정해야 했다.

또한 `공격통과` 같은 항목도 필요했다.
예를 들어 산이나 벽 같은 지형이 공격 경로를 막는다면, 단순히 이동만 막는 것이 아니라 공격 판정에도 영향을 줄 수 있다.

그래서 지형 데이터는 이동과 전투 양쪽에 모두 연결된다는 것을 이해했다.

## 유닛 데이터 테이블

유닛 데이터 테이블은 유닛의 기본 능력치와 전투 관련 수치를 관리하기 위한 표이다.

처음에는 아군, 적군, 중립 몬스터를 한 표에 넣는 것이 헷갈렸다.
특히 중립 몬스터를 병종으로 처리해야 하는지, 별도 유형으로 처리해야 하는지 고민했다.

정리하면서 내린 기준은 다음과 같다.

* 아군과 적군은 병종을 사용한다.
* 중립 몬스터는 병종 대신 몬스터 유형으로 구분한다.
* 중립 몬스터는 포획형과 보상형으로 나눈다.

이렇게 나누면 아군/적군은 전투 병종 규칙을 적용하고, 중립 몬스터는 포획이나 보상 처리 규칙을 따로 관리할 수 있다.

유닛 데이터 테이블에는 다음 항목을 넣었다.

| 항목     | 설명                        |
| ------ | ------------------------- |
| 번호     | 유닛을 구분하기 위한 고유 ID         |
| 이름     | 게임 화면에 표시되는 유닛 이름         |
| 진영     | 아군, 적군, 중립 구분             |
| 병종     | 아군과 적군에게 적용되는 전투 병종       |
| 몬스터유형  | 중립 몬스터일 때 포획형/보상형을 구분     |
| 이동타입   | 지상이동, 공중이동, 수상이동 구분       |
| 체력     | 유닛의 최대 체력                 |
| 공격력    | 기본 공격력                    |
| 물리방어   | 물리 공격을 받을 때 사용하는 방어 수치    |
| 마법방어   | 마법 공격을 받을 때 사용하는 방어 수치    |
| 공격타입   | 물리 또는 마법                  |
| 사정거리   | 기본 공격 가능 거리               |
| 이동력    | 한 턴에 이동할 수 있는 기본 범위       |
| 치명타율   | 기본 치명타 발생 확률              |
| 특수능력ID | 유닛이 사용하는 특수능력 데이터와 연결되는 값 |

## 중립 몬스터 정리

이번에 가장 헷갈렸던 부분은 중립 몬스터였다.

처음에는 중립 몬스터도 병종 중 하나로 넣을 수 있을 것 같았다.
하지만 그렇게 하면 포획형 몬스터가 왜 보병이나 조사병단 같은 병종을 갖는지 애매해진다.

그래서 중립 몬스터는 병종이 아니라 `몬스터유형`으로 구분하는 것이 더 맞다고 판단했다.

정리한 기준은 다음과 같다.

| 진영     | 병종 사용 여부 | 추가 구분                     |
| ------ | -------- | ------------------------- |
| 아군     | 사용       | 보병, 기병, 비병, 수병, 스키병, 조사병단 |
| 적군     | 사용       | 보병, 기병, 비병, 수병, 스키병, 조사병단 |
| 중립 몬스터 | 사용하지 않음  | 포획형, 보상형                  |

이렇게 정리하니 아군/적군/중립 몬스터의 역할이 더 명확해졌다.

## 데이터 테이블을 기획서에 적용하면서 느낀 점

이번 작업을 하면서 시스템 기획서가 단순히 설명문을 쓰는 문서가 아니라는 것을 느꼈다.

처음에는 문장으로 규칙을 정리하면 충분하다고 생각했다.
하지만 강사님 피드백을 받고 보니, 개발자가 실제로 구현하려면 “어떤 값이 어디에 들어가는지”가 필요했다.

예를 들어 “숲은 기병이 이동하기 불리하다”라고만 쓰면 부족하다.
이 문장은 실제 수치로 바꾸면 “숲에서 기병 이동 소모량은 2”처럼 표현되어야 한다.

또 “공중이동 유닛은 ZOC를 무시한다”라는 규칙도 유닛 데이터 테이블의 이동타입 값과 연결되어야 한다.

이번 작업을 통해 기획서의 문장과 데이터 테이블은 따로 떨어진 것이 아니라 서로 연결되어야 한다는 것을 배웠다.

## 오늘 배운 핵심

1. 데이터 테이블은 게임 수치를 관리하기 위한 표이다.
2. 문장으로 작성한 규칙은 데이터 값으로 내려와야 한다.
3. 지형 데이터는 이동력 소모량과 전투 보정을 관리한다.
4. 유닛 데이터는 병종, 이동타입, 능력치, 사정거리 등을 관리한다.
5. 중립 몬스터는 병종이 아니라 몬스터유형으로 따로 구분하는 것이 더 자연스럽다.
6. 빈칸은 최대한 피하고, 없음 / -1 / TRUE / FALSE 같은 값으로 명확하게 표시해야 한다.
7. 데이터 테이블은 개발자가 참고할 수 있도록 값의 의미가 분명해야 한다.
