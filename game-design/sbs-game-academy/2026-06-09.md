## 시스템 기획서 피드백 및 데이터 테이블 보완

## 오늘 진행한 내용

시뮬레이션 RPG 이동 및 전투 시스템 기획서를 완성한 뒤 강사님께 피드백을 받았다.

이번 피드백에서는 내용을 더 많이 작성하는 것보다, 시스템 구현에 필요하지 않은 설명을 줄이고 실제 작업자가 확인해야 하는 규칙과 수치만 남기는 것이 중요하다는 점을 배웠다.

또한 데이터 테이블은 단순히 현재 능력치를 입력하는 표가 아니라, 유닛이 성장했을 때 능력치가 어떻게 변하는지까지 고려해야 한다는 것을 알게 되었다.

---

## 1. 시스템 기획서에 필요하지 않은 설명

기획서에 다음 문장을 작성했었다.

> 스테이지 종류에 따라 특정 지형의 비중이 높아질 수 있다.

처음에는 스테이지마다 어떤 지형이 많이 나올 수 있는지 설명하면 이해에 도움이 된다고 생각했다.

하지만 이 내용은 스테이지 콘셉트나 레벨 디자인에 가까운 설명이며, 지형 시스템이 실제로 어떻게 작동하는지를 설명하는 내용은 아니었다.

시스템 기획서에서는 다음과 같은 내용이 더 중요하다.

* 해당 지형에 진입할 수 있는가
* 이동력이 얼마나 소모되는가
* 전투 능력치에 어떤 보정이 적용되는가
* 공격이나 이동 경로를 막는가

이번 피드백을 통해 시스템 기획서에는 관련된 내용을 모두 넣는 것이 아니라, 실제 기능 구현에 필요한 내용만 남겨야 한다는 것을 배웠다.

---

## 2. 이미 정해진 규칙을 다시 설명하지 않기

대기 규칙에 다음 내용을 작성했었다.

> PvE 스테이지에서는 제한 턴 안에 목표를 달성해야 하므로, 대기만 반복하면 실패로 이어질 수 있다.

이 문장은 틀린 내용은 아니지만, 시스템 기획서에는 굳이 필요하지 않다는 피드백을 받았다.

이미 승패 규칙에서 제한 턴을 초과하면 패배한다고 정의했다면, 대기 페이지에서 같은 결과를 다시 설명할 필요가 없다.

작업자는 다음 규칙만 알면 된다.

* 대기를 선택하면 해당 유닛의 행동이 종료된다.
* 대기에는 추가 능력치 보정이 없다.
* 다음 자신의 턴까지 다시 행동할 수 없다.

대기를 반복했을 때 제한 턴을 초과하는 결과는 기존 승패 규칙에 따라 자동으로 처리된다.

이번 피드백을 통해 설명을 많이 작성하는 것이 항상 좋은 것은 아니며, 이미 다른 항목에서 정의한 내용은 반복하지 않는 것이 문서를 더 읽기 쉽게 만든다는 것을 알게 되었다.

## 3. 지형 적응력 F등급 문제

기존 지형 적응력은 다음과 같이 구성했다.

* A등급: 기본 능력치 적용
* B등급: 일부 능력치 감소
* C등급: 큰 능력치 감소
* F등급: 진입 불가

하지만 강사님 피드백을 통해 F등급을 전투 보정 등급에 넣는 것이 어색하다는 점을 알게 되었다.

F등급 지형은 유닛이 진입할 수 없기 때문에 해당 타일 위에서 전투가 발생하지 않는다.
전투가 발생하지 않는다면 명중률, 회피율, 공격력, 방어력 보정을 정할 필요도 없다.

현재 구조에서는 두 가지 정보가 섞여 있었다.

* 이동 가능 여부
* 지형 위에서 전투할 때 받는 능력치 보정

앞으로는 다음과 같이 구분해야 한다.

### 이동 가능 여부

지형별 이동력 소모량에서 관리한다.

* 숫자: 이동 시 소비하는 이동력
* `-1`: 해당 지형 진입 불가

### 지형 적응력

실제로 진입할 수 있는 지형에서 전투 능력치가 어떻게 달라지는지 관리한다.

* A등급
* B등급
* C등급

F등급을 삭제하고, 진입 가능한 지형 중 가장 불리한 경우를 C등급으로 처리하는 방향이 더 자연스럽다고 생각했다.

## 4. 작업자가 신경 쓰지 않아도 되는 내용 줄이기

이번 수업에서 가장 크게 느낀 부분이다.

기획서를 작성하면서 내용을 자세히 적을수록 좋은 문서가 된다고 생각했다.
하지만 작업자가 구현할 때 따로 확인하지 않아도 되는 내용까지 계속 작성하면 오히려 중요한 규칙을 찾기 어려워질 수 있다.

예를 들어 다음과 같은 문장은 기능 설명보다 부가 설명에 가깝다.

* 특정 스테이지에서는 어떤 지형의 비중이 높아질 수 있다.
* 대기를 반복하면 제한 턴을 넘길 수 있다.
* 상황에 따라 값이 변경될 수 있다.

시스템 기획서에서는 다음과 같이 확정된 기준을 먼저 보여줘야 한다.

* 어떤 조건에서 작동하는가
* 어떤 값이 적용되는가
* 작동 결과가 무엇인가
* 예외 조건은 무엇인가

앞으로는 문장을 추가하기 전에 개발자나 작업자가 이 내용을 실제로 확인할 필요가 있는지 먼저 생각해야겠다.

## 5. 데이터 테이블 컬럼명 작성 기준

기존 데이터 테이블은 컬럼명을 한글로 작성했다.

예시:

* 번호
* 이름
* 체력
* 공격력
* 이동력
* 특수능력 ID

하지만 실제 데이터 테이블에서는 한글 컬럼명을 사용하지 않는 것이 좋다는 피드백을 받았다.

컬럼명은 개발 과정에서 직접 사용될 가능성이 있기 때문에 영문으로 작성하는 것이 관리하기 쉽다.

예를 들면 다음과 같이 변경할 수 있다.

| 기존 한글 컬럼 | 변경할 영문 컬럼     |
| -------- | ------------- |
| 번호       | `UnitID`      |
| 이름       | `UnitName`    |
| 진영       | `FactionType` |
| 병종       | `ClassType`   |
| 이동타입     | `MoveType`    |
| 체력       | `HP`          |
| 공격력      | `ATK`         |
| 물리방어력    | `PDEF`        |
| 마법방어력    | `MDEF`        |
| 사정거리     | `AttackRange` |
| 이동력      | `MoveRange`   |
| 특수능력 ID  | `SkillID`     |

게임 화면에 표시되는 실제 이름이나 설명은 한글 데이터를 사용할 수 있지만, 데이터를 구분하는 컬럼명은 영문으로 작성하는 방향이 더 적절하다.

## 6. 유닛의 레벨 성장 데이터

현재 작성한 유닛 데이터 테이블에는 유닛의 기본 능력치만 들어가 있었다.

예시:

* 체력
* 공격력
* 물리방어력
* 마법방어력
* 이동력
* 사정거리

하지만 이 값이 1레벨 기준이라면, 레벨이 올랐을 때 능력치가 얼마나 증가하는지도 정해야 한다.

기본 능력치만 있으면 레벨이 올라갔을 때의 능력치를 계산할 수 없기 때문이다.

추가로 필요한 컬럼은 다음과 같다.

| 컬럼명            | 설명           |
| -------------- | ------------ |
| `BaseHP`       | 1레벨 기준 체력    |
| `HPPerLevel`   | 레벨당 증가 체력    |
| `BaseATK`      | 1레벨 기준 공격력   |
| `ATKPerLevel`  | 레벨당 증가 공격력   |
| `BasePDEF`     | 1레벨 기준 물리방어력 |
| `PDEFPerLevel` | 레벨당 증가 물리방어력 |
| `BaseMDEF`     | 1레벨 기준 마법방어력 |
| `MDEFPerLevel` | 레벨당 증가 마법방어력 |

능력치는 다음과 같은 방식으로 계산할 수 있다.

```text
현재 체력 = BaseHP + HPPerLevel × (현재 레벨 - 1)
현재 공격력 = BaseATK + ATKPerLevel × (현재 레벨 - 1)
```

예를 들어 다음 값을 가진 유닛이 있다고 가정한다.

```text
BaseHP = 1000
HPPerLevel = 50
현재 레벨 = 5
```

계산 결과는 다음과 같다.

```text
현재 체력 = 1000 + 50 × (5 - 1)
현재 체력 = 1200
```

이번 피드백을 통해 데이터 테이블은 현재 값만 저장하는 표가 아니라, 게임 안에서 값이 어떻게 변화하는지까지 생각해야 한다는 것을 배웠다.

## 이번 피드백을 통해 배운 점

1. 시스템 기획서에는 구현과 직접 관련된 내용만 남겨야 한다.
2. 다른 항목에서 이미 정의한 규칙은 반복해서 설명하지 않아도 된다.
3. 이동 가능 여부와 전투 능력치 보정은 서로 다른 규칙으로 나누어야 한다.
4. 진입할 수 없는 지형에는 전투 보정 등급이 필요하지 않다.
5. 데이터 테이블의 컬럼명은 영문으로 작성하는 것이 좋다.
6. 유닛 데이터에는 1레벨 능력치뿐만 아니라 레벨당 증가 수치도 필요하다.
7. 문서를 자세하게 만드는 것보다, 작업자가 필요한 내용을 빠르게 찾을 수 있게 만드는 것이 중요하다.
