## 1. 시스템 기획서 피드백 정리

이번에 작성한 시뮬레이션 RPG 시스템 기획서에서는 지형, 유닛, 이동, 전투, 승패 처리, 데이터 테이블까지 정리했다.

다만 피드백을 받으면서 내가 작성한 내용 중 일부는 시스템 기획서에 꼭 필요한 내용이 아니라는 것을 알게 되었다.

### 필요 없는 설명은 줄여야 한다

기존 문서에는 다음과 같은 문장이 있었다.

> 스테이지 종류에 따라 특정 지형의 비중이 높아질 수 있다.

처음에는 스테이지마다 지형 구성이 달라질 수 있다는 설명이 필요하다고 생각했다.

하지만 이 내용은 시스템 규칙이라기보다는 스테이지 콘셉트나 레벨 디자인에 가까운 설명이다.
시스템 기획서에서는 지형이 어떤 비중으로 등장하는지보다, 지형이 실제로 어떤 규칙으로 작동하는지가 더 중요하다.

예를 들면 다음과 같은 내용이 더 필요하다.

* 해당 지형에 진입할 수 있는가
* 이동력이 얼마나 소모되는가
* 전투 보정이 어떻게 적용되는가
* 공격 경로를 막는가

이번 피드백을 통해 시스템 기획서에는 “설명하고 싶은 내용”보다 “작업자가 구현할 때 필요한 내용”을 우선해야 한다는 것을 배웠다.

## 2. 이미 정의된 규칙은 반복해서 쓰지 않기

대기 항목에는 다음과 같은 내용을 작성했었다.

> PvE 스테이지에서는 제한 턴 안에 목표를 달성해야 하므로, 대기만 반복하면 실패로 이어질 수 있다.

이 문장 자체가 틀린 것은 아니다.
하지만 이미 승패 처리 규칙에서 제한 턴 초과 시 패배라고 정의되어 있다면, 대기 항목에서 다시 설명할 필요는 없다.

대기 항목에서는 다음 정도만 정의하면 된다.

* 대기를 선택하면 해당 유닛의 행동이 종료된다.
* 대기한 유닛은 다음 턴까지 다시 행동할 수 없다.
* 대기에는 방어력 증가나 선제공격 같은 추가 효과를 부여하지 않는다.

대기를 반복해서 제한 턴을 넘기는 상황은 승패 처리 규칙에서 처리하면 된다.

이번 피드백을 통해 같은 내용을 여러 페이지에서 반복하면 문서가 길어지고, 오히려 핵심 규칙을 찾기 어려워질 수 있다는 것을 알게 되었다.

## 3. 지형 적응력 F등급 문제

기존 지형 적응력 표에서는 F등급을 `진입 불가`로 설정했다.

하지만 피드백을 통해 전투 보정표에 `진입 불가`를 넣는 것이 어색하다는 것을 알게 되었다.

이유는 단순하다.

> 진입할 수 없는 지형에서는 전투가 발생하지 않는다.
> 전투가 발생하지 않는다면 명중률, 회피율, 공격력, 방어력 보정을 적용할 상황도 없다.

즉, 이동 가능 여부와 전투 보정은 따로 관리하는 것이 더 맞다.

앞으로는 다음과 같이 구분하는 것이 좋다.

### 이동 가능 여부

지형별 이동력 소모량에서 관리한다.

* `1`, `2`, `3` 같은 숫자: 이동 시 소비하는 이동력
* `-1`: 진입 불가

### 지형 적응력

진입 가능한 지형에서 전투할 때 받는 보정을 관리한다.

* A등급: 기본 능력치 유지
* B등급: 일부 능력치 감소
* C등급: 큰 능력치 감소

F등급은 삭제하거나, 최소한 전투 보정표가 아니라 이동 가능 여부 쪽에서 처리하는 것이 더 자연스럽다고 느꼈다.

## 4. 데이터 테이블 피드백

데이터 테이블에서도 중요한 피드백을 받았다.

기존에는 컬럼명을 한글로 작성했다.

예시:

* 번호
* 이름
* 체력
* 공격력
* 이동력

하지만 실제 데이터 테이블에서는 한글 컬럼명을 사용하는 것보다 영문 컬럼명을 사용하는 것이 좋다는 피드백을 받았다.

예를 들면 다음과 같이 바꿀 수 있다.

| 기존 컬럼명 | 변경 컬럼명        |
| ------ | ------------- |
| 번호     | `UnitID`      |
| 이름     | `UnitName`    |
| 진영     | `FactionType` |
| 병종     | `ClassType`   |
| 이동타입   | `MoveType`    |
| 체력     | `HP`          |
| 공격력    | `ATK`         |
| 물리방어   | `PDEF`        |
| 마법방어   | `MDEF`        |
| 사정거리   | `AttackRange` |
| 이동력    | `MoveRange`   |
| 특수능력ID | `SkillID`     |

게임 화면에 표시되는 이름은 한글이어도 되지만, 데이터 컬럼명은 개발 과정에서 직접 사용될 가능성이 있으므로 영문으로 작성하는 것이 더 좋다.

## 5. 레벨 성장 데이터

유닛 데이터 테이블에는 1레벨 기준 능력치만 넣었었다.

하지만 유닛이 레벨업하는 구조라면, 레벨당 능력치가 얼마나 증가하는지도 필요하다.

예를 들어 1레벨 체력만 있으면 5레벨, 10레벨 능력치를 계산할 수 없다.

그래서 다음과 같은 컬럼이 추가로 필요하다.

| 컬럼명            | 설명          |
| -------------- | ----------- |
| `BaseHP`       | 1레벨 기준 체력   |
| `HPPerLevel`   | 레벨당 증가 체력   |
| `BaseATK`      | 1레벨 기준 공격력  |
| `ATKPerLevel`  | 레벨당 증가 공격력  |
| `BasePDEF`     | 1레벨 기준 물리방어 |
| `PDEFPerLevel` | 레벨당 증가 물리방어 |
| `BaseMDEF`     | 1레벨 기준 마법방어 |
| `MDEFPerLevel` | 레벨당 증가 마법방어 |

계산 방식은 다음처럼 잡을 수 있다.

```text
현재 체력 = BaseHP + HPPerLevel × (현재 레벨 - 1)
현재 공격력 = BaseATK + ATKPerLevel × (현재 레벨 - 1)
```

예를 들어 다음 값이 있다고 하면,

```text
BaseHP = 1000
HPPerLevel = 50
현재 레벨 = 5
```

계산 결과는 다음과 같다.

```text
현재 체력 = 1000 + 50 × (5 - 1)
현재 체력 = 1200
```

이번 피드백을 통해 데이터 테이블은 현재 수치만 입력하는 표가 아니라, 게임 안에서 값이 어떻게 변화하는지까지 고려해야 한다는 것을 배웠다.

## 6. 컨텐츠 기획이란 무엇인가?

오늘 수업에서는 컨텐츠 기획에 대해서도 배웠다.

컨텐츠는 한마디로 게임 안에서 유저가 즐길 수 있는 요소이다.
뷔페에 여러 먹거리가 있는 것처럼, 게임에도 유저가 계속 플레이할 수 있게 만드는 여러 즐길거리가 필요하다.

하지만 단순히 많은 컨텐츠를 넣는다고 좋은 게임이 되는 것은 아니다.

여러 장르의 요소를 아무 기준 없이 섞으면 오히려 이상한 게임이 될 수 있다.
수업에서는 이것을 민트초코파인애플피자 같은 느낌으로 비유했다.

즉, 중요한 것은 “무엇을 넣을까?”보다 “왜 넣는가?”이다.

## 7. 3매치 퍼즐 예시

3매치 퍼즐은 같은 블록을 맞춰 없애는 방식의 게임이다.

대표적으로 캔디 크러쉬 같은 게임이 있다.

하지만 3매치 퍼즐은 구조가 단순하기 때문에 오래 플레이하면 매너리즘이 생길 수 있다.
계속 같은 방식으로 퍼즐을 클리어하는 것만 반복하면 유저가 쉽게 지루함을 느낄 수 있다.

이때 단순히 퍼즐 규칙을 더 복잡하게 만드는 방법도 있지만, 다른 방식으로 재미를 확장할 수도 있다.

수업에서는 꿈의 정원 예시를 들었다.

기존 3매치 퍼즐은 스테이지를 클리어하는 것이 목적이었다.
하지만 꿈의 정원은 퍼즐을 클리어해서 별을 모으고, 그 별을 사용해서 정원을 꾸밀 수 있게 만들었다.

즉, 퍼즐 클리어 이후의 목적을 추가한 것이다.

이 예시를 통해 같은 장르라도 컨텐츠를 어떻게 연결하느냐에 따라 플레이 목적이 달라질 수 있다는 것을 알게 되었다.

## 8. 컨텐츠를 넣을 때 생각해야 할 점

컨텐츠를 추가할 때는 단순히 재미있어 보이는 요소를 넣는 것이 아니라, 우리 게임에 맞는지 생각해야 한다.

생각해야 할 질문은 다음과 같다.

* 이 컨텐츠를 왜 넣는가?
* 이 컨텐츠를 넣으면 유저가 무엇을 얻는가?
* 우리 게임의 핵심 재미와 어울리는가?
* 유저가 좋아할 가능성이 있는가?
* 반대로 유저가 싫어할 가능성은 없는가?
* 기존 시스템과 충돌하지 않는가?

게임은 여러 요소를 넣을 수 있지만, 모든 요소가 다 어울리는 것은 아니다.
컨텐츠 기획은 단순히 많은 기능을 넣는 것이 아니라, 게임의 방향에 맞는 즐길거리를 고르는 작업이라고 느꼈다.

## 9. 기획 의도 찾기

수업에서 가장 기억에 남은 부분은 기획 의도에 대한 이야기였다.

기획 의도가 바로 보이지 않는 경우에도 의도를 찾으려는 태도가 필요하다고 했다.

예를 들어 어떤 시스템이나 컨텐츠가 왜 들어갔는지 설명이 부족할 수 있다.
또는 내가 보기에는 납득하기 어려운 이유로 들어간 것처럼 보일 수도 있다.

그래도 기획자는 다음 질문을 해야 한다.

* 이걸 왜 넣고 싶었을까?
* 이걸 넣으면 어떤 효과를 기대했을까?
* 유저에게 어떤 행동을 유도하려고 했을까?
* 이 컨텐츠가 없으면 어떤 문제가 생길까?
* 이 컨텐츠가 있으면 게임의 흐름이 어떻게 바뀔까?

이번 수업을 통해 기획 의도는 누가 항상 친절하게 설명해주는 것이 아니라, 기획자가 스스로 찾아내고 정리해야 하는 것이라는 생각이 들었다.

## 오늘 배운 핵심

오늘 배운 내용은 크게 두 가지로 정리할 수 있다.

첫 번째는 시스템 기획서에서는 구현에 필요한 규칙과 수치만 남겨야 한다는 것이다.
설명이 많다고 좋은 문서가 되는 것이 아니라, 작업자가 필요한 내용을 빠르게 찾을 수 있어야 한다.

두 번째는 컨텐츠 기획에서는 기능을 많이 넣는 것보다, 왜 그 컨텐츠가 필요한지 생각하는 것이 중요하다는 것이다.

결국 시스템 기획과 컨텐츠 기획은 방향은 다르지만 공통점이 있다.

* 시스템 기획은 규칙의 이유와 작동 방식을 명확히 해야 한다.
* 컨텐츠 기획은 유저가 즐길 이유와 목적을 명확히 해야 한다.

둘 다 “그냥 넣었다”가 아니라, “왜 필요한가”를 설명할 수 있어야 한다.

## 느낀 점

오늘 수업을 들으면서 기획서는 많이 쓰는 것이 중요한 게 아니라, 필요한 내용을 정확히 쓰는 것이 중요하다는 생각이 들었다.

나는 아직 기획서를 작성할 때 설명을 많이 넣어야 안전하다고 생각하는 경향이 있다.
하지만 이번 피드백을 통해 설명이 많으면 오히려 중요한 규칙이 흐려질 수 있다는 것을 느꼈다.

앞으로는 문서를 작성할 때 다음 기준을 먼저 생각해야겠다.

* 이 문장이 실제 구현에 필요한가?
* 이미 다른 항목에서 설명한 내용을 반복하고 있지는 않은가?
* 수치로 표현해야 하는 부분을 말로만 적고 있지는 않은가?
* 이 컨텐츠를 넣는 의도가 분명한가?

이번 TIL은 시스템 기획서 수정 방향과 컨텐츠 기획의 기본 개념을 같이 정리한 기록이다.
