# 1. 컨텐츠 기획

## 컨텐츠 기획의 기본 개념

컨텐츠 기획은 게임 안에서 유저가 즐길 수 있는 요소를 만드는 작업이다.

컨텐츠는 단순히 기능을 추가하는 것이 아니라, 유저가 게임 안에서 계속 행동할 이유를 만드는 역할을 한다.

예시로는 다음과 같은 것들이 있다.

* 길드 컨텐츠
* 생산 컨텐츠
* 퀘스트 컨텐츠
* 거래 컨텐츠
* NPC 호감도 컨텐츠
* PvP 컨텐츠
* 하우징 / 건축물 컨텐츠
* 컬렉션 컨텐츠

각 컨텐츠는 단독으로 존재하기보다, 메인 게임에 어떤 영향을 주는지까지 함께 고려해야 한다.

---

## 컨텐츠와 시스템의 연결

수업 자료에서는 컨텐츠가 시스템으로 확장되는 예시가 나왔다.

### 길드 컨텐츠

길드 컨텐츠 예시:

* 길드 채팅
* 길드전
* 길드 버프

이 요소들이 모이면 `길드 시스템`이 된다.

### 생산 컨텐츠

생산 컨텐츠 예시:

* 생산 레시피
* 생산 스킬
* 대량 납품

이 요소들이 모이면 `생산 시스템`이 된다.

즉, 컨텐츠는 하나의 즐길거리이고, 그 컨텐츠가 규칙과 구조를 가지면 시스템으로 확장될 수 있다.

---

## 컨텐츠 기획에서 생각할 점

컨텐츠를 기획할 때는 단순히 넣고 싶은 기능을 나열하는 것이 아니라, 메인 게임과 어떤 관계를 가지는지 생각해야 한다.

수업 자료에서는 다음 흐름이 나왔다.

```text
길드 / 생산 컨텐츠
↓
시스템화
↓
메인 게임에 영향
```

즉, 컨텐츠는 메인 게임과 분리된 부가 기능이 아니라, 게임의 플레이 흐름에 영향을 줄 수 있다.

예를 들어 생산 컨텐츠라면 다음과 같은 질문이 필요하다.

* 생산 레시피는 어떤 방식으로 얻는가?
* 생산 스킬은 어떻게 성장하는가?
* 대량 납품은 어떤 보상과 연결되는가?
* 생산 시스템이 메인 게임에 어떤 영향을 주는가?

---

## 컨텐츠 기획에서 고려할 대상

수업 메모에는 다음 내용도 있었다.

> 싫어하는 사람들도 좋아하게 만드는 컨텐츠 기획도 만들어야 함

이 말은 특정 컨텐츠를 좋아하는 유저만 보는 것이 아니라, 그 컨텐츠를 좋아하지 않는 유저도 어느 정도 받아들일 수 있는 구조를 고민해야 한다는 의미로 정리할 수 있다.

예를 들어 PvP를 좋아하지 않는 유저가 있다면, PvP를 강제로 시키는 방식은 반발을 만들 수 있다.
반대로 PvP를 하지 않아도 다른 방식으로 보상을 얻을 수 있다면 부담이 줄어든다.

---

## 컨셉 기획 예시

수업 메모에는 컨셉 기획 예시로 다음 내용이 있었다.

* K-웹소설
* 어반 판타지
* 헌터물 장르
* 게임으로 만들기에 적합한 장르

헌터물 장르는 퀘스트, 던전, 성장, 파티, 장비, 길드 같은 게임 요소와 연결하기 쉽다.

# 2. 길드 컨텐츠

## 길드 컨텐츠 개요

길드 컨텐츠는 마음이 맞는 플레이어끼리 길드를 결성하고, 길드 전용 컨텐츠를 즐길 수 있게 만드는 컨텐츠이다.

자료에 나온 길드 컨텐츠의 주요 구성은 다음과 같다.

* 길드 설립
* 길드 유지
* 길드 등급
* 길드 채팅
* 길드 전용 생산
* 길드 버프
* 길드 컨텐츠 UI
* 길드 컨텐츠 연출

---

## 길드 설립

길드는 플레이어끼리 커뮤니티를 형성하는 기능이다.

길드 설립 조건은 다음과 같이 정리할 수 있다.

* 마음이 맞는 플레이어끼리 길드를 결성한다.
* 길드 설립 시 대량의 골드를 지불한다.
* 최소 인원이 필요하다.
* 캐릭터는 하나의 길드에만 소속될 수 있다.
* 일정 기간 길드 멤버 중 아무도 접속하지 않으면 길드가 비활성화된다.
* 비활성화 상태에서 일정 기간이 지나면 길드가 해체될 수 있다.

---

## 길드 가입 제한

자료에는 길드 가입 제한도 있었다.

* 길드 가입을 위해서는 게임 내 추천서 아이템이 필요하다.
* 추천서는 반복 퀘스트 보상으로 얻을 수 있다.
* 길드는 가입 인원이 제한된다.

이 구조는 길드 가입을 단순한 클릭 가입으로 만들지 않고, 게임 내 플레이와 연결하는 방식이다.

---

## 길드원에게 주어지는 혜택

길드에 소속된 캐릭터는 길드 관련 혜택을 받을 수 있다.

자료에 나온 예시는 다음과 같다.

* 길드 채팅
* 길드 전용 생산 레시피
* 경험치 버프
* 추가 창고칸

길드 컨텐츠는 단순히 소속만 제공하는 것이 아니라, 길드에 소속되어 있을 이유를 제공해야 한다.

---

## 길드 등급

길드에는 등급이 존재한다.

자료에 나온 등급 구조는 다음과 같다.

* 길드 등급: 1~10

길드 등급 상승에는 일정량의 골드가 필요하다.

등급 상승 시 제공되는 혜택은 다음과 같다.

* 길드 멤버 수 증가
* 레시피 종류 증가
* 버프 종류 증가
* 추가 창고칸 증가

반대로 길드 유지 조건을 만족하지 못하면 등급이 하락할 수 있다.

자료에 나온 조건은 다음과 같다.

* 1개월에 1회
* 일정량의 골드 지불
* 과제 수행 필요
* 조건 미달 시 길드 등급 하락

---

## 길드 유지 비용과 문제점

길드 컨텐츠는 유지 비용이 존재한다.

자료에서는 길드 등급을 유지하기 위해 골드와 과제를 요구하는 구조가 나왔다.

이 구조는 길드 유지와 접속을 유도할 수 있지만, 유저에게 부담이 될 수도 있다.

자료에 나온 반발 가능성은 다음과 같다.

* 길드 설립 및 등급 유지에 골드와 과제를 요구하면 유저 반발이 있을 수 있음

해결책 예시는 다음과 같이 제시되었다.

* 1~3초 길드 개설 비용은 저렴하게 책정
* 1레벨 길드에는 유지비나 과제, 각종 버프를 제공하지 않음
* 등급 하락 시 보관중이던 아이템 반환

---

## 길드 컨텐츠 기획 의도

길드 컨텐츠의 기획 의도는 다음과 같이 정리할 수 있다.

* 플레이어 간 커뮤니케이션 활성화
* 접속 빈도 증가
* 접속 시간 증가
* 길드 유지와 등급 상승을 위한 지속적인 골드 소모 유도
* 무의미한 길드의 과다 생성을 줄여 서버 부담 감소

길드 컨텐츠는 단순한 커뮤니티 기능이 아니라, 접속 유지, 재화 소모, 유저 관계 형성과 연결된다.

---

## 길드 컨텐츠 문서 구성

자료에는 길드 컨텐츠 기획서와 길드 컨텐츠 UI 기획서가 분리되어 있었다.

# 3. 추가 컨텐츠 예시

자료에는 `좀비 아포칼립스 도시운영 게임 : 추가 컨텐츠` 예시가 있었다.

이 예시는 하나의 게임에 여러 추가 컨텐츠를 붙였을 때 각 컨텐츠가 어떤 역할을 하는지 보여준다.

## 플레이어 간 거래

플레이어 간 거래의 역할은 다음과 같다.

* 유저 간 커뮤니티 활성화
* 거래 수수료를 통한 골드 회수
* 초반 진입 장벽 하락
* 유저의 게임 정착률 증가

거래 컨텐츠는 유저끼리 아이템을 주고받게 만들고, 거래 수수료를 통해 게임 내 재화를 회수하는 역할을 한다.

---

## NPC 호감도 컨텐츠

NPC 호감도 컨텐츠의 역할은 다음과 같다.

* 데드엔딩 1회 회피 기능 제공
* 유저 이탈률 억제
* 꾸준한 접속 유도
* 캐릭터 판매 활성화
* 2차 창작 활성화

NPC 호감도는 단순한 대화 컨텐츠가 아니라, 유저가 특정 캐릭터에게 애정을 느끼고 계속 접속하게 만드는 역할을 할 수 있다.

---

## PvP 컨텐츠

PvP 컨텐츠의 역할은 다음과 같다.

* 랭킹 컨텐츠를 통한 접속 빈도 향상
* 간접 PvP로 돌발 이벤트 대처 능력 강화
* 유저 이탈률 억제

자료에서는 직접 PvP가 아니라 간접 PvP 형태로 언급되었다.

간접 PvP는 실시간 전투 부담을 줄이면서도 순위 경쟁과 대처 능력을 만들 수 있다.

---

## 유명 건축물 DLC

유명 건축물 DLC의 역할은 다음과 같다.

* 간접적인 Pay2Win 요소로 PvE 난이도 하락
* 유저 정착률 증가
* 꾸미기 요소를 좋아하는 유저층에게 추가 수익 기대

이 컨텐츠는 전투력과 직접 연결되지 않더라도, 도시 운영 게임에서 꾸미기와 수집 욕구를 자극하는 방식으로 활용될 수 있다.

# 4. 퀘스트편

## 퀘스트 컨텐츠와 데이터 테이블

퀘스트편에서는 컨텐츠 기획에서 데이터 테이블을 활용하는 사례가 나왔다.

핵심은 다음과 같다.

> 기획 의도 필수 → 퀘스트 목적

즉, 퀘스트를 만들 때는 단순히 보상과 조건만 넣는 것이 아니라, 이 퀘스트가 왜 필요한지 목적을 먼저 정해야 한다.

---

## 퀘스트 테이블 구성

자료에 나온 퀘스트 테이블은 다음 항목으로 구성되어 있었다.

| 항목          | 내용               |
| ----------- | ---------------- |
| 퀘스트ID       | 퀘스트의 고유 번호       |
| 발동 오브젝트     | 퀘스트를 시작시키는 대상    |
| 발생 전제조건     | 퀘스트 발생 조건        |
| 퀘스트 스크립트 ID | 퀘스트 대화와 진행 연결    |
| 보상          | 퀘스트 완료 시 지급되는 보상 |
| 반복가능        | 퀘스트 반복 여부        |

---

## 퀘스트 ID

퀘스트 ID는 퀘스트의 고유 번호이다.

하나의 퀘스트에는 하나의 번호만 지정된다.

퀘스트 ID는 데이터 테이블에서 퀘스트를 구분하기 위한 기준값이다.

---

## 발동 오브젝트

발동 오브젝트는 퀘스트를 시작시키는 대상이다.

자료에는 세 가지 발동 방식이 나왔다.

### NPC ID

지정된 ID의 NPC를 클릭하면 퀘스트 발동 판정을 시작한다.

* 각 NPC의 ID는 NPC 테이블을 참조한다.

### 오브젝트 ID

맵상에 지정된 ID의 오브젝트를 클릭하면 퀘스트 발동 판정을 시작한다.

* 각 오브젝트의 ID는 오브젝트 테이블을 참조한다.

### 지역 ID

정해진 ID 지역에 플레이어가 진입하면 퀘스트 발동 판정을 시작한다.

* 지역 ID와 범위는 지역 테이블을 참조한다.

---

## 발생 전제조건

퀘스트는 아무 때나 발생하는 것이 아니라, 조건을 만족해야 발생할 수 있다.

자료에 나온 발생 전제조건은 다음과 같다.

* 선행 퀘스트 ID
* 레벨
* 아이템
* 1일 1회 여부

예를 들어 특정 퀘스트를 완료해야 다음 퀘스트가 열리거나, 특정 아이템을 가지고 있어야 퀘스트가 발생할 수 있다.

---

## 퀘스트 스크립트 ID

퀘스트 스크립트 ID는 퀘스트의 대화, 선택지, 컷신, 보상 지급과 연결된다.

자료에서는 퀘스트 테이블의 `퀘스트 스크립트 ID`가 별도의 `퀘스트 스크립트 테이블`과 연결되어 있었다.

즉, 퀘스트 테이블은 퀘스트의 기본 조건을 관리하고, 스크립트 테이블은 퀘스트 진행 내용을 관리한다.

---

## 퀘스트 스크립트 테이블

퀘스트 스크립트 테이블에는 다음 항목이 있었다.

| 항목          | 내용                                 |
| ----------- | ---------------------------------- |
| 퀘스트 스크립트 ID | 스크립트 고유 번호                         |
| 순서          | 퀘스트 진행 순서                          |
| 종류          | 대사, 컷신, 선택지, 아이템, 결과 등             |
| 실행주체        | Player, NPC ID, Item ID 등          |
| 실행대상        | Text ID, Scene ID, 순서 번호, 클리어/중단 등 |

이 테이블은 퀘스트가 어떤 순서로 진행되는지 정의한다.

예시 흐름은 다음과 같다.

```text
플레이어 대사
↓
NPC 대사
↓
컷신
↓
선택지
↓
아이템 지급
↓
결과 처리
```

---

## 텍스트 테이블

텍스트 테이블은 실제 대사 문장을 관리한다.

자료에서는 다음 항목이 있었다.

| 항목     | 내용        |
| ------ | --------- |
| TextID | 텍스트 고유 번호 |
| 한국어    | 한국어 대사    |
| 영어     | 영어 대사     |

퀘스트 스크립트 테이블에서 `Text ID`를 참조하면, 텍스트 테이블에 있는 실제 문장이 출력된다.

예시:

| TextID    | 한국어       | 영어                   |
| --------- | --------- | -------------------- |
| 100010101 | 안녕!       | Hello!               |
| 100010102 | 내 이름은 코난. | My name is Conan!    |
| 100010105 | 탐정이죠.     | I’m the Private Eye! |

텍스트 테이블을 따로 두면 언어별 대사를 관리하기 쉬워진다.

---

## 보상 항목

퀘스트 테이블에는 보상 항목이 있었다.

예시 보상은 다음과 같다.

* Gold 5000
* ItemID
* 콘텐츠ID

보상은 하나만 있는 것이 아니라 여러 개를 넣을 수 있다.

자료에서는 `보상 1`, `보상 2`, `보상 3`처럼 구분되어 있었다.

보상은 퀘스트를 완료했을 때 지급되는 결과이다.

---

## 반복 가능 여부

퀘스트 테이블에는 반복 가능 여부가 있었다.

반복 가능 값은 다음과 같이 표시되어 있었다.

* O
* X

O는 반복 가능, X는 반복 불가로 볼 수 있다.

자료 예시에서는 1일 1회 조건과 반복 가능 여부가 함께 사용되었다.

즉, 퀘스트가 반복 가능한지, 하루에 한 번만 가능한지 등을 데이터로 관리한다.

예시로 보이는 퀘스트는 다음과 같은 구조였다.

# 6. 연습과제

## 과제 내용

이번 연습과제는 리그 오브 레전드에 퀘스트 컨텐츠를 기획하는 것이다.

과제 성격은 컨셉 및 창작 기획이다.

AOS 게임인 리그 오브 레전드에 다른 장르의 게임에서 나올 법한 퀘스트 컨텐츠를 구성해서 만드는 과제이다.

작성해야 할 퀘스트 유형은 네 가지이다.

1. 스토리 퀘스트
2. 단일형 서브 퀘스트
3. 반복형 퀘스트
4. 일일 퀘스트

현재는 과제 내용을 확인한 단계이며, 실제 기획 작성은 아직 진행하지 않았다.

---

## 오늘 정리한 핵심

오늘 내용은 컨텐츠 기획을 실제 문서와 데이터 테이블로 연결하는 방식에 가까웠다.

핵심은 다음과 같다.

1. 컨텐츠는 메인 게임에 어떤 영향을 주는지까지 고려해야 한다.
2. 길드 컨텐츠는 커뮤니티, 접속 유지, 재화 소모, 길드 성장과 연결된다.
3. 길드 컨텐츠 기획서는 시스템 기획서와 UI 기획서로 나눌 수 있다.
4. 추가 컨텐츠는 유저 정착, 접속 빈도, 수익 구조, 커뮤니티 활성화와 연결된다.
5. 퀘스트는 기획 의도와 목적이 먼저 정리되어야 한다.
6. 퀘스트는 퀘스트 테이블, 스크립트 테이블, 텍스트 테이블로 나누어 관리할 수 있다.
7. 퀘스트 데이터는 발동 조건, 전제 조건, 보상, 반복 여부를 명확하게 관리해야 한다.
8. 연습과제는 리그 오브 레전드에 4종류의 퀘스트 컨텐츠를 기획하는 것이다.

---

## 느낀 점

이번 내용은 컨텐츠를 단순한 아이디어로만 보는 것이 아니라, 실제 문서와 데이터로 어떻게 정리하는지를 보는 수업이었다.

특히 퀘스트 테이블을 보면서 퀘스트 하나에도 생각보다 많은 데이터가 필요하다는 것을 알게 되었다.

퀘스트는 단순히 “NPC가 말을 걸고 보상을 준다”가 아니라, 다음 요소들이 모두 연결되어야 한다.

* 어떤 대상이 퀘스트를 발동시키는가
* 어떤 조건을 만족해야 발생하는가
* 어떤 순서로 대화와 선택지가 진행되는가
* 어떤 보상을 지급하는가
* 반복 가능한가
* 어떤 텍스트를 출력하는가

또 길드 컨텐츠를 보면서 컨텐츠 기획서와 UI 기획서를 나누는 방식도 이해할 수 있었다.

앞으로 컨텐츠를 기획할 때는 아이디어만 적는 것이 아니라, 해당 컨텐츠가 어떤 시스템과 연결되는지, 어떤 데이터로 관리되는지까지 같이 생각해야겠다.
