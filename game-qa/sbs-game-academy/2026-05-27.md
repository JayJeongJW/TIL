## 2. SQL이란?

SQL은 **Structured Query Language**의 약자로, 데이터베이스에 저장된 데이터를 조회하거나 조건에 맞게 필터링할 때 사용하는 질의 언어다.

수업에서는 게임 QA 실무에서 서버나 DB에 저장된 유저 데이터, 주문 데이터, 결제 데이터, 로그 데이터를 확인할 때 SQL 쿼리를 사용할 수 있다고 배웠다.

실무에서는 다음과 같은 표현도 자주 쓰일 수 있다.

* DB 쿼리
* 서버 쿼리
* 쿼리문을 날린다
* 쿼리로 데이터를 확인한다

즉, SQL을 배운다는 것은 단순히 개발 문법을 외우는 것이 아니라, **QA가 필요한 데이터를 빠르게 조회하고 검증하는 방법을 배우는 것**에 가깝다.

## 3. SQL 실습에 사용한 테이블 구조

수업 실습에서는 크게 `users` 테이블과 `orders` 테이블을 사용했다.

### 3.1 users 테이블

`users`는 회원 정보를 관리하는 테이블이다.

| 컬럼명           | 의미    |
| ------------- | ----- |
| `id`          | 회원 번호 |
| `name`        | 회원 이름 |
| `email`       | 이메일   |
| `signup_date` | 가입일   |

### 3.2 orders 테이블

`orders`는 주문/결제 정보를 관리하는 테이블이다.

| 컬럼명            | 의미    |
| -------------- | ----- |
| `order_id`     | 주문 번호 |
| `user_id`      | 회원 번호 |
| `product_name` | 상품명   |
| `price`        | 결제 금액 |
| `status`       | 주문 상태 |
| `order_date`   | 주문 일시 |

`status`에는 다음과 같은 값이 들어갈 수 있다.

| 상태값       | 의미    |
| --------- | ----- |
| `SUCCESS` | 결제 성공 |
| `FAILED`  | 결제 실패 |
| `REFUND`  | 환불    |

## 4. SQL 기본 문법 정리

## 4.1 전체 데이터 확인하기 - SELECT *

테이블에 어떤 데이터가 들어 있는지, 전체 구조가 어떤지 확인할 때 사용한다.

```sql
SELECT *
FROM users;
```

해석하면 다음과 같다.

```text
users 테이블에서 모든 컬럼과 모든 행을 조회한다.
```

여기서 `*`는 전체 컬럼을 의미한다.

---

## 4.2 필요한 컬럼만 조회하기 - SELECT 컬럼명

컬럼이 너무 많아 한눈에 들어오지 않을 때, 필요한 컬럼만 골라서 조회할 수 있다.

```sql
SELECT name, email
FROM users;
```

해석하면 다음과 같다.

```text
users 테이블에서 name 컬럼과 email 컬럼만 조회한다.
```

QA에서는 테스트 계정의 이름, 이메일, 가입일처럼 필요한 정보만 빠르게 확인할 때 사용할 수 있다.

---

## 4.3 특정 데이터 하나 찾기 - WHERE =

특정 테스트 계정이나 특정 주문 번호를 정확히 찾아야 할 때 사용한다.

```sql
SELECT *
FROM users
WHERE id = 1;
```

해석하면 다음과 같다.

```text
users 테이블에서 id가 1인 회원 데이터만 조회한다.
```

주의할 점은 세미콜론 `;`의 위치다.

잘못된 예시는 다음과 같다.

```sql
SELECT * FROM users; WHERE id = 1;
```

세미콜론은 SQL 문장이 끝났다는 뜻이므로, `WHERE` 전에 찍으면 안 된다.

---

## 4.4 크기 비교하기 - 비교 연산자

금액, 레벨, 수량처럼 숫자 조건을 비교할 때 사용한다.

예를 들어 5만원 이상 결제한 주문만 조회하려면 다음과 같이 작성한다.

```sql
SELECT *
FROM orders
WHERE price >= 50000;
```

해석하면 다음과 같다.

```text
orders 테이블에서 price가 50000 이상인 주문만 조회한다.
```

자주 사용하는 비교 연산자는 다음과 같다.

| 연산자  | 의미     |
| ---- | ------ |
| `=`  | 같다     |
| `!=` | 같지 않다  |
| `>`  | 크다     |
| `<`  | 작다     |
| `>=` | 크거나 같다 |
| `<=` | 작거나 같다 |

---

## 4.5 여러 조건을 동시에 확인하기 - AND / OR

여러 조건을 함께 확인하고 싶을 때 사용한다.

```sql
SELECT *
FROM orders
WHERE status = 'FAILED'
  AND price > 1000;
```

해석하면 다음과 같다.

```text
orders 테이블에서 결제 상태가 FAILED이고, price가 1000보다 큰 주문만 조회한다.
```

`AND`는 모든 조건을 만족해야 한다.

`OR`는 조건 중 하나만 만족해도 된다.

```sql
SELECT *
FROM orders
WHERE status = 'FAILED'
   OR status = 'REFUND';
```

이 쿼리는 결제 실패이거나 환불 상태인 주문을 조회한다.

---

## 4.6 결과 정렬하기 - ORDER BY

조회 결과를 보기 좋은 순서로 정렬할 때 사용한다.

```sql
SELECT *
FROM orders
ORDER BY order_date DESC;
```

해석하면 다음과 같다.

```text
orders 테이블의 모든 데이터를 주문 일자 기준으로 최신순 정렬한다.
```

정렬 기준은 다음과 같다.

| 문법     | 의미              |
| ------ | --------------- |
| `DESC` | 내림차순, 큰 값부터 정렬  |
| `ASC`  | 오름차순, 작은 값부터 정렬 |

예를 들어 가격이 높은 주문부터 보고 싶다면 다음과 같이 작성한다.

```sql
SELECT *
FROM orders
ORDER BY price DESC;
```

---

## 4.7 일부 데이터만 보기 - LIMIT

데이터가 너무 많을 때 상위 몇 개만 확인할 수 있다.

```sql
SELECT *
FROM users
LIMIT 5;
```

해석하면 다음과 같다.

```text
users 테이블에서 상위 5개 데이터만 조회한다.
```

QA에서는 수만 건의 데이터 중 샘플만 빠르게 확인할 때 유용하다.

---

## 4.8 키워드 검색하기 - LIKE

특정 단어가 포함된 이메일이나 상품명을 찾을 때 사용한다.

```sql
SELECT *
FROM users
WHERE email LIKE '%naver%';
```

해석하면 다음과 같다.

```text
users 테이블에서 email에 naver라는 문자열이 포함된 회원만 조회한다.
```

`%`는 앞뒤에 어떤 글자가 와도 된다는 의미다.

| 조건              | 의미           |
| --------------- | ------------ |
| `LIKE 'test%'`  | test로 시작하는 값 |
| `LIKE '%test'`  | test로 끝나는 값  |
| `LIKE '%test%'` | test가 포함된 값  |

예를 들어 다음 쿼리는 `@test.com`으로 끝나는 이메일을 찾는다.

```sql
SELECT *
FROM users
WHERE email LIKE '%@test.com';
```

이때 `%@test.com`은 `@test.com으로 시작`이 아니라, **@test.com으로 끝나는 값**을 의미한다.

---

## 4.9 필수값 누락 확인하기 - IS NULL

기획서상 필수로 입력되어야 하는 값이 비어 있는지 확인할 때 사용한다.

```sql
SELECT *
FROM users
WHERE signup_date IS NULL;
```

해석하면 다음과 같다.

```text
users 테이블에서 signup_date 값이 비어 있는 데이터를 조회한다.
```

QA 관점에서는 회원가입일이 필수값인데 NULL로 들어간 데이터가 있다면 결함 가능성이 있다.

NULL은 일반적인 값이 아니기 때문에 `= NULL`로 비교하지 않는다.

잘못된 예시는 다음과 같다.

```sql
SELECT *
FROM users
WHERE signup_date = NULL;
```

정확한 방식은 다음과 같다.

```sql
SELECT *
FROM users
WHERE signup_date IS NULL;
```

만약 결과가 없으면, 현재 조건에서는 NULL 데이터가 없다는 의미다.

---

## 4.10 데이터 개수 세기 - COUNT(*)

전체 데이터 개수를 확인할 때 사용한다.

```sql
SELECT COUNT(*) AS total_users
FROM users;
```

해석하면 다음과 같다.

```text
users 테이블에 있는 전체 회원 수를 세고, 결과 컬럼 이름을 total_users로 표시한다.
```

`AS`는 결과 컬럼에 별칭을 붙일 때 사용한다.


```text
price가 1000 이하인 주문만 조회한다.
```

```sql

```

확인 포인트:

```text
DESC는 내림차순이다.
price가 높은 주문부터 조회된다.
```

## 문제 5. 이메일에 naver가 들어가는 회원 조회


```text

```

---

## 문제 6. 상품명이 누락된 주문 확인



```sql

FROM orders

  AND price >= 100000

```

확인 포인트:

```text
status가 SUCCESS이고 price가 100000 이상인 주문만 조회한다.
그 결과를 order_date 기준 최신순으로 정렬한다.
```

## 문제 8. 전체 회원 수 조회

확인 포인트:


## 문제 9. 성공했지만 3만원 미만인 주문 조회

# 6. QA 실무에서 SQL을 사용하는 이유

QA가 SQL을 사용하는 이유는 단순히 데이터를 보는 것이 아니라, **테스트 결과가 실제 DB에 어떻게 저장되었는지 확인하기 위해서**다.

예를 들어 다음과 같은 상황에서 SQL을 사용할 수 있다.

| 상황         | SQL로 확인할 내용                                |
| ---------- | ------------------------------------------ |
| 회원가입 테스트   | 회원 데이터가 정상 생성되었는지 확인                       |
| 결제 테스트     | 결제 상태가 SUCCESS/FAILED/REFUND로 정상 저장되었는지 확인 |
| 환불 테스트     | 환불 처리 후 상태값이 REFUND로 변경되었는지 확인             |
| 이벤트 보상 테스트 | 보상 지급 데이터가 정상 생성되었는지 확인                    |
| 필수값 검증     | NULL 데이터가 발생하지 않았는지 확인                     |
| 로그 확인      | 특정 시간대에 발생한 유저 행동 기록 확인                    |

즉, SQL은 QA가 서버와 DB에 저장된 결과를 확인하는 도구다.

# 7. 명세 리뷰 / 데이터 테이블 검증 / 테스트 케이스 설계의 연결

수업에서는 명세 리뷰와 데이터 테이블 검증 이후, 테스트 케이스 설계 단계로 이어지는 흐름도 배웠다.

흐름을 정리하면 다음과 같다.

```text
명세 리뷰
→ 데이터 테이블 검증
→ 테스트 케이스 설계
→ 단위 테스트
→ 통합 테스트 또는 이후 테스트 단계
```

테스트 케이스 설계는 이후 테스트를 수행하기 위해 테스트를 준비하는 활동이다.

명세 리뷰가 완료된 기획서를 기반으로 테스트 케이스를 설계하며, 보통 기획 단계 말미에서 개발 단계로 넘어가는 시점에 준비된다.

# 8. 테스트 케이스 설계

테스트 케이스를 설계할 때는 기능을 적절한 단위로 쪼개야 한다.

수업에서는 최소 2뎁스, 가능하면 3뎁스까지 기능을 분할해서 보는 것이 좋다고 배웠다.

예를 들어 `길드`라는 큰 기능을 테스트한다고 하면 다음과 같이 나눌 수 있다.

```text
길드
├─ 생성
│  ├─ 생성 가능 레벨
│  ├─ 생성 비용 보유 여부
│  ├─ 길드 이름 중복 여부
│  ├─ 길드 마크 설정
│  ├─ 길드 소개글 입력
│  └─ 최대 인원 설정
├─ 가입 및 탈퇴
├─ 출석
├─ 상점
└─ 던전
```

이런 식으로 기능을 분할하면 테스트 케이스를 더 체계적으로 설계할 수 있다.

테스트 케이스의 분류는 다음처럼 구성할 수 있다.

| 분류   | 예시                           |
| ---- | ---------------------------- |
| 대분류  | 길드                           |
| 중분류  | 길드 생성                        |
| 소분류  | 생성 가능 레벨 확인                  |
| 시나리오 | 생성 가능 레벨 미만인 유저가 길드 생성을 시도한다 |

# 9. 단위 테스트

단위 테스트는 기능이나 화면을 작은 단위로 나누어 테스트하는 방식이다.

## 9.1 코드 단위 테스트

개발 코드 단위로 테스트하는 방식이다.

다만 게임 QA가 직접 코드 단위 테스트를 진행하는 경우는 많지 않다고 배웠다.

## 9.2 기능 단위 테스트

게임을 구성하는 시스템이나 콘텐츠를 기능 단위로 나눈 뒤 테스트하는 방식이다.

예시:

* 길드 생성
* 길드 가입
* 길드 탈퇴
* 길드 출석
* 길드 상점
* 길드 던전

## 9.3 화면 단위 테스트

게임 화면 하나를 하나의 테스트 단위로 보는 방식이다.

예시:

* 대전 상대 목록 화면
* 공격덱 편성 화면
* 전투 준비 화면
* 전투 화면
* 전투 결과 화면
* 보상 수령 화면
* 랭킹 화면

화면 단위 테스트는 직관적이고 간단하다는 장점이 있다.

# 10. 단위 테스트의 특징

단위 테스트는 수행 주기가 중요하다.

수업에서는 단위 테스트가 데일리로 진행될 수 있다고 배웠다.

또한 단위 테스트 수행 환경은 보통 엔진 에디터 상에서 진행될 수 있다.

예시:

* Unity Editor
* Unreal Blueprint

따라서 게임 QA를 준비하는 입장에서는 최소 하나의 게임 엔진에 대한 활용 경험이 있으면 좋다.

이번 과제로는 Unity를 활용한 슈팅 게임 제작 실습이 제시되었다.

# 11. CI/CD 개발 파이프라인 이해

수업에서는 단위 테스트와 함께 CI/CD 개발 파이프라인에 대한 개념도 언급되었다.

이미지 흐름을 기준으로 정리하면 다음과 같다.

```text
작업자
→ 코드 Push
→ 코드 저장소
→ 테스트/빌드 서버
→ Build / Test / Release
→ 배포 서버
→ Application Server
```

각 단계의 의미는 다음과 같다.

| 단계                 | 의미                            |
| ------------------ | ----------------------------- |
| 작업자                | 개발자, 기획자, 아티스트 등 실제 작업을 하는 사람 |
| 코드 Push            | 작업한 코드를 저장소에 올리는 행위           |
| 코드 저장소             | 코드가 관리되는 저장소                  |
| 테스트/빌드 서버          | 코드를 빌드하고 테스트하는 서버             |
| Build              | 실행 가능한 빌드 생성                  |
| Test               | 자동화 또는 수동 테스트 수행              |
| Release            | 배포 가능한 상태로 정리                 |
| Deploy             | 배포 서버로 전달                     |
| Application Server | 실제 서비스가 동작하는 서버               |

QA는 이 파이프라인을 완벽하게 개발자 수준으로 다룰 필요는 없더라도, **어느 단계에서 빌드가 생성되고, 어느 단계에서 테스트가 수행되며, 어느 단계에서 배포되는지**는 이해해야 한다.

# 12. 버전 관리 시스템 예습 키워드

수업에서는 버전 관리 시스템으로 SVN과 Git이 언급되었다.

앞으로 Git을 사용해볼 예정이며, SVN도 별도로 개념을 공부해두면 좋다고 했다.

예습할 키워드는 다음과 같다.

| 키워드           | 간단한 의미                           |
| ------------- | -------------------------------- |
| Repository    | 코드와 파일을 저장하는 저장소                 |
| Commit        | 변경 사항을 저장소에 기록하는 행위              |
| Commit Log    | 커밋 기록과 변경 내용 설명                  |
| Checkout      | 저장소의 특정 버전이나 브랜치를 내려받는 행위        |
| Clone         | 원격 저장소를 내 PC에 복제하는 행위            |
| Main / Master | 기본 브랜치 이름                        |
| Branch        | 독립적으로 작업하기 위한 분기                 |
| Revision      | 변경 이력의 특정 버전                     |
| Push          | 내 작업 내용을 원격 저장소에 올리는 행위          |
| Pull          | 원격 저장소의 최신 내용을 내 작업 환경으로 가져오는 행위 |
| Merge         | 브랜치의 변경 사항을 합치는 행위               |

# 13. 오늘 수업을 듣고 느낀 점

이번 수업을 통해 SQL은 개발자만 사용하는 도구가 아니라, QA에게도 중요한 검증 도구라는 것을 알게 되었다.

처음에는 `SELECT`, `FROM`, `WHERE` 같은 문법이 단순 조회 문법처럼 보였다.

하지만 실습을 해보니 SQL은 테스트 결과를 확인하고, 조건에 맞는 데이터를 찾고, 누락된 값을 검증하는 데 직접적으로 사용될 수 있었다.

특히 `IS NULL`은 QA 관점에서 중요하다고 느꼈다.

기획서상 필수값인데 DB에 NULL로 저장되어 있다면, 겉으로는 화면이 정상처럼 보여도 실제 데이터상으로는 결함일 수 있기 때문이다.

또한 테스트 케이스 설계에서는 기능을 1뎁스로만 보는 것이 아니라, 2뎁스 또는 3뎁스까지 나누어 구조화하는 것이 중요하다는 점을 배웠다.

예를 들어 길드라는 큰 기능을 테스트할 때, 단순히 “길드 테스트”라고 보는 것이 아니라 생성, 가입, 탈퇴, 출석, 상점, 던전처럼 기능을 나누고, 다시 생성 가능 레벨, 비용, 이름 중복, 마크, 소개글 같은 세부 조건까지 내려가야 한다.

이번 수업은 QA가 단순히 게임을 플레이하며 버그를 찾는 역할만 하는 것이 아니라, 데이터, 명세, 테스트 케이스, 빌드 파이프라인까지 함께 이해해야 한다는 것을 느낀 시간이었다.
