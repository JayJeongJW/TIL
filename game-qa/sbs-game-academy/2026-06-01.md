## 4. Git / SVN 기본 개념

### Main, Master, Trunk

Git에서는 보통 `main` 또는 `master`가 주 저장소 역할을 한다.

SVN에서는 `trunk`가 주 저장소 역할을 한다.

| 용어     | 의미                |
| ------ | ----------------- |
| main   | Git의 기본 브랜치       |
| master | Git의 기존 기본 브랜치 명칭 |
| trunk  | SVN의 기본 개발 라인     |

최근에는 GitHub에서 기본 브랜치 이름으로 `main`을 많이 사용한다.

## 5. 메인 레파지토리와 브랜치

메인 레파지토리는 실서비스 환경 또는 라이브 환경의 기준이 되는 저장소다.

라이브 환경은 안정성이 가장 중요하다. 따라서 모든 변경사항이 바로 라이브 환경에 반영되는 것이 아니라, 여러 단계의 테스트 환경을 거쳐 검증된 뒤 반영된다.

### 브랜치의 의미

브랜치는 메인 레파지토리를 기준으로 특정 작업을 하기 위해 분리한 작업 공간이다.

예를 들어 개발팀은 새로운 기능 개발을 위해 별도 브랜치를 만들고, QA팀은 테스트를 위해 QA 전용 브랜치 또는 Dev 환경을 사용할 수 있다.

## 7. 품질 게이트

각 환경 사이에는 품질 게이트가 존재한다.

품질 게이트는 다음 단계로 넘어가기 전에 충족해야 하는 기준이다.

예를 들어 1차 환경에서 2차 환경으로 넘어가기 위해서는 기획자 또는 개발자의 자체 테스트가 완료되어야 하고, QA가 테스트할 수 있는 상태가 되어야 한다.

QA는 이 품질 게이트를 수립하거나 검증하는 역할을 맡을 수 있다.

### 품질 게이트의 목적

* 불안정한 빌드가 다음 환경으로 넘어가는 것을 방지
* 테스트 가능한 상태인지 확인
* 치명적인 결함을 사전에 차단
* Live 환경의 안정성 확보

## 8. Revision 개념

Revision은 버전 관리 시스템에서 사용하는 변경 이력 번호다.

게임 클라이언트 버전과는 다르다.

예시는 다음과 같다.

```text
1.0.7 (12854)
```

여기서 의미는 다음과 같다.

```text
1.0.7  = 개발팀에서 정의한 게임 버전
12854 = Revision number
```

Revision number는 특정 변경 이력 또는 커밋 번호를 의미한다.

수업에서는 커밋이 한 번 진행될 때마다 리비전 넘버가 증가한다고 설명했다.

---

## 9. Revision이 중요한 이유

Revision은 버그 리포트에서 매우 중요하다.

같은 버그처럼 보여도, 어떤 리비전에서 발생했는지에 따라 원인과 수정 여부가 달라질 수 있다.

### 버그 리포트에서 Revision이 필요한 이유

* 문제가 발생한 정확한 버전을 확인할 수 있다.
* 개발자가 동일한 환경에서 문제를 재현할 수 있다.
* 수정된 버전에서 QA가 재검증할 수 있다.
* 서로 다른 빌드에서 발생한 문제를 구분할 수 있다.

### 예시

```text
빌드 버전: 1.0.7
Revision: 12854
발생 환경: QA Dev Server
발생 기능: 로그인 API
```

이런 식으로 기록하면 개발자가 어느 시점의 코드에서 문제가 발생했는지 추적하기 쉽다.

반대로 Revision 정보가 없으면, 개발자와 QA가 서로 다른 버전을 보고 있을 수 있다. 이 경우 문제 재현이 어려워지고, 수정 확인도 정확하게 진행하기 어렵다.

## 10. 프론트엔드 테스트와 백엔드 테스트

### 프론트엔드 테스트

프론트엔드 테스트는 사용자가 직접 보는 화면과 조작 흐름을 확인하는 테스트에 가깝다.

예를 들어 네이버 로그인 화면을 기준으로 보면 다음과 같다.

```text
아이디 입력
→ 비밀번호 입력
→ 로그인 버튼 클릭
→ 화면 변화 확인
```

게임에서는 다음과 같은 테스트가 프론트엔드 테스트에 해당한다.

* UI 버튼 동작 확인
* 화면 전환 확인
* 캐릭터 이동 확인
* 아이템 장착 화면 확인
* 상점 UI 출력 확인
* 튜토리얼 진행 확인

게임 엔진 기준으로는 Unity, Unreal에서 직접 화면을 보며 테스트하는 방식이 여기에 가깝다.

---

### 백엔드 테스트

백엔드 테스트는 화면보다 데이터 흐름과 서버 응답을 확인하는 테스트다.

로그인 기능을 예로 들면 다음과 같다.

```text
클라이언트가 로그인 요청
→ 서버가 아이디/비밀번호 확인
→ 데이터베이스 조회
→ 로그인 성공 또는 실패 응답 반환
→ 클라이언트가 결과 표시
```

QA는 이때 화면만 보는 것이 아니라, 요청과 응답이 정상적으로 처리되는지도 확인해야 한다.

게임에서는 다음과 같은 기능이 백엔드 테스트와 연결된다.

* 로그인
* 출석 이벤트
* 상점 구매
* 아이템 지급
* 랭킹 조회
* 우편 보상
* 이벤트 보상
* 길드 보상
* 결제 보상 처리

## 11. API 개념

API는 클라이언트와 서버 사이에서 요청과 응답을 주고받기 위한 약속 또는 매개체다.

수업에서는 식당 비유로 이해했다.

```text
손님 = 클라이언트
주방장 = 서버
종업원/키오스크 = API
```

손님이 주방장에게 직접 주문하지 않고, 종업원이나 키오스크를 통해 주문하는 것처럼 클라이언트도 서버에 직접 접근하는 것이 아니라 API를 통해 요청을 보낸다.

백엔드 테스트를 한다는 것은 API를 이용해 서버 요청과 응답을 검증한다는 뜻이다.

## 14. Jenkins 개념

Jenkins는 빌드를 생성하고 배포하거나, 테스트를 자동으로 실행하는 데 사용하는 도구다.

게임 QA에서는 Jenkins를 통해 다음 작업을 수행할 수 있다.

* 최신 코드 가져오기
* 빌드 생성
* 테스트 자동 실행
* 결과 리포트 생성
* QA용 빌드 배포
* 정해진 시간마다 반복 실행

이번 실습에서는 Jenkins를 이용해 GitHub 저장소의 파일을 가져오고, Postman 테스트를 Newman으로 실행해 HTML 리포트를 생성하는 흐름을 실습했다.

# 15. 실습 개요

이번 실습의 목표는 다음과 같았다.


## 16. 실습 1 - Mock 서버 실행

먼저 Node.js를 설치하고, 실습 파일이 있는 폴더에서 CMD를 실행했다.

### 실행 명령어

```bash
npm install
```

`npm install`은 `package.json`에 정의된 패키지를 설치하는 명령어다.

실행 결과 다음과 같은 문구가 출력되었다.

```text
added 71 packages
```

이는 필요한 패키지가 설치되었다는 의미다.

추가로 다음과 같은 보안 경고도 출력되었다.

```text
4 vulnerabilities
```

하지만 이번 수업용 Mock 서버 실습에서는 설치 실패가 아니라 경고로 보고 진행했다.

이후 서버를 실행했다.

```bash
node server.js
```

정상 실행 시 다음 문구가 출력되었다.

```text
Mock Game API running on port 3000
```

이는 Mock API 서버가 3000번 포트에서 실행 중이라는 의미다.

## 17. 실습 2 - Postman API 테스트

Postman에서는 환경 파일과 Collection 파일을 Import하여 테스트를 진행했다.

### Environment 설정

`GameQA_ENV.postman_environment.json` 파일을 Import하고 Active 상태로 설정했다.

이 환경 파일에는 API 요청에 필요한 기본 주소 등이 포함되어 있었다.

예상되는 base URL은 다음과 같은 형태다.

```text
http://localhost:3000
```

### Collection Import

다음 Collection 파일을 Postman에 추가했다.

```text
Login_API_Test.postman_collection.json
Ranking_API_Test.postman_collection.json
Item_Grant_API_Test.postman_collection.json
```

## 18. Login API 테스트

먼저 Login API를 실행했다.

```text
Login API Test
→ Login - obtain token
→ POST 요청 실행
```

로그인 API는 인증 토큰을 발급하는 역할을 한다.

정상 결과는 다음과 같다.

* 응답 코드: 200 OK
* 응답 데이터: token 값 발급

이 토큰은 이후 Ranking API 또는 Item Grant API 요청에서 인증 정보로 사용될 수 있다.

## 19. Ranking API 테스트

이후 Ranking API를 실행했다.

```text
Ranking API Test
→ Get Ranking - top 10
→ GET 요청 실행
```

정상 결과는 상위 10명의 랭킹 데이터가 조회되는 것이다.

확인해야 할 항목은 다음과 같다.

* 응답 코드가 200인지
* 랭킹 데이터가 반환되는지
* top 10 조건에 맞는 데이터가 나오는지
* 응답 JSON 구조가 정상인지

## 20. Item Grant API 테스트

마지막으로 아이템 지급 API를 실행했다.

```text
Item Grant API Test
→ Grant Item to Player
→ POST 요청 실행
```

이번 실습에서는 `player_001` 유저에게 회복 물약 1개를 지급하는 요청을 테스트했다.

```json

  "granted": {
    "playerId": "player_001",
    "itemId": "potion_small",
    "amount": 1
  }
}
```

QA 관점에서는 단순히 success가 true인지 보는 것에서 끝나면 안 된다.

다음 항목을 함께 확인해야 한다.

* 요청한 playerId와 응답 playerId가 일치하는가
* 요청한 itemId와 응답 itemId가 일치하는가
* 지급 수량 amount가 정확한가
* 응답 코드가 정상인가
* 실제 서버 데이터에도 반영되는 구조인지 확인이 필요한가

## 26. Jenkinsfile 작성

Jenkins에서 Pipeline Script from SCM을 사용하려면 GitHub 저장소 안에 `Jenkinsfile`이 있어야 한다.

이번 실습에서는 Jenkinsfile을 만들어 Pipeline 실행 절차를 정의했다.

### Jenkinsfile 예시

```groovy
pipeline {
    agent any

    stages {
        stage('Check Node Version') {
            steps {
                bat 'node -v'
                bat 'npm -v'
            }
        }

        stage('Install Dependencies') {
            steps {
                bat 'call npm install'
            }
        }

        stage('Install Newman Locally') {
            steps {
                bat 'call npm install --save-dev newman newman-reporter-htmlextra'
            }
        }

        stage('Start Mock Server') {
            steps {
                bat 'start "" /B node server.js'
                bat 'timeout /t 3 /nobreak'
            }
        }

        stage('Run Login API Test') {
            steps {
                bat 'if not exist newman mkdir newman'
                bat 'call npx newman run Login_API_Test.postman_collection.json -e GameQA_ENV.postman_environment.json -r cli,htmlextra --reporter-htmlextra-export newman\\Login_Report.html --export-environment GameQA_ENV.postman_environment.json'
            }
        }

        stage('Run Ranking API Test') {
            steps {
                bat 'call npx newman run Ranking_API_Test.postman_collection.json -e GameQA_ENV.postman_environment.json -r cli,htmlextra --reporter-htmlextra-export newman\\Ranking_Report.html'
            }
        }

        stage('Run Item Grant API Test') {
            steps {
                bat 'call npx newman run Item_Grant_API_Test.postman_collection.json -e GameQA_ENV.postman_environment.json -r cli,htmlextra --reporter-htmlextra-export newman\\ItemGrant_Report.html'
            }
        }
    }

    post {
        always {
            archiveArtifacts artifacts: 'newman/*.html', allowEmptyArchive: true
        }
    }
}
```

## 27. Jenkinsfile 구조 이해

### Check Node Version

```groovy
bat 'node -v'
bat 'npm -v'
```

Jenkins 환경에서 Node.js와 npm이 정상 인식되는지 확인한다.

### Install Dependencies

```groovy
bat 'call npm install'
```

`package.json`을 기준으로 필요한 패키지를 설치한다.

### Install Newman Locally

```groovy
bat 'call npm install --save-dev newman newman-reporter-htmlextra'
```

Newman과 HTML 리포터를 Jenkins workspace 안에 설치한다.

전역 설치보다 로컬 설치 방식이 Jenkins 환경에서 더 안정적으로 동작할 수 있다.

### Start Mock Server

```groovy
bat 'start "" /B node server.js'
```

Mock API 서버를 백그라운드로 실행한다.

```groovy
bat 'timeout /t 3 /nobreak'
```

서버가 켜질 시간을 확보하기 위해 3초 대기한다.

### Run API Tests

각 Collection을 Newman으로 실행하고 HTML 리포트를 생성한다.

### archiveArtifacts

```groovy
archiveArtifacts artifacts: 'newman/*.html'
```

생성된 HTML 리포트를 Jenkins 빌드 결과의 Artifacts 영역에 보관한다.

# 28. 트러블슈팅 정리

이번 실습에서는 여러 문제가 발생했다.

실제 업무에서도 CI/CD 환경은 한 번에 정상 동작하지 않을 수 있기 때문에, 오류 로그를 보고 원인을 분리하는 과정이 중요하다는 것을 느꼈다.

---

## 28.1 `git add.` 입력 오류

### 발생 상황

처음에 다음과 같이 입력했다.

```bash
git add.
```

### 원인

`git add`와 `.` 사이에 공백이 없었다.

Git은 `add.`라는 명령어를 찾으려고 했고, 해당 명령어가 없기 때문에 오류가 발생했다.

### 해결

```bash
git add .
```

공백을 넣어 다시 실행했다.

---

## 28.2 커밋 메시지 누락

### 발생 상황

다음 명령어를 입력했다.

```bash
git commit -m
```

### 원인

`-m` 옵션 뒤에는 커밋 메시지가 필요하다.

### 해결

```bash
git commit -m "Initial upload"
```

커밋 메시지를 포함해 다시 실행했다.

---

## 28.3 GitHub URL만 단독 입력

### 발생 상황

Git Bash에 다음 주소만 입력했다.

```bash
https://github.com/JayJeongJW/-.git
```

### 원인

GitHub 주소는 명령어가 아니다.

원격 저장소를 연결하려면 `git remote add origin` 명령어와 함께 사용해야 한다.

### 해결

```bash
git remote add origin https://github.com/JayJeongJW/gameqa-jenkins-practice.git
```

## 28.4 `src refspec main does not match any`

### 발생 상황

다음 오류가 발생했다.

```text
error: src refspec main does not match any
```

### 원인

아직 main 브랜치가 없거나, 커밋이 생성되지 않은 상태에서 push를 시도했기 때문이다.

### 해결

```bash
git add .
git commit -m "Initial upload"
git branch -M main
git push -u origin main
```

순서대로 진행했다.

## 28.5 `fetch first` 오류

### 발생 상황

GitHub 저장소에 이미 커밋이 있는 상태에서 push를 시도해 오류가 발생했다.

```text
Updates were rejected because the remote contains work that you do not have locally.
```

### 원인

원격 저장소와 로컬 저장소의 이력이 달랐기 때문이다.

### 해결

처음에는 강제로 push했다.

```bash
git push -u origin main --force
```

하지만 이후 새 저장소를 만들 때는 README를 생성하지 않고 빈 저장소로 만들어 이 문제를 피했다.

## 28.6 Jenkinsfile not found

### 발생 상황

Jenkins 빌드 시 다음 오류가 발생했다.

```text

```

### 원인

Pipeline Script from SCM 방식은 GitHub 저장소에서 `Jenkinsfile`을 찾는다.

처음에는 저장소에 Jenkinsfile이 없었기 때문에 Jenkins가 파이프라인을 실행할 수 없었다.

### 해결

GitHub 저장소 루트에 `Jenkinsfile`을 추가했다.

하지만 GitHub 저장소의 브랜치는 `main`이었다.

## 28.8 Git 실행 파일 경로 문제

### 발생 상황

Jenkins가 Visual Studio 내부 Git 경로를 참조하려고 했다.

### 원인

Jenkins Tools의 Git 경로가 제대로 설정되지 않았다.

### 해결

Jenkins 관리 > Tools에서 Git 실행 파일 경로를 설정했다.

```text
C:\Program Files\Git\cmd\git.exe
```

또는 다음 경로도 사용할 수 있다.

```text
C:\Program Files\Git\bin\git.exe
```

중요한 점은 폴더가 아니라 `git.exe`까지 포함된 경로를 입력해야 한다는 것이다.

## 28.9 node_modules 업로드 문제

### 발생 상황

실습 폴더에 `node_modules`가 포함된 상태로 GitHub 업로드를 시도할 수 있었다.

## 28.10 3000번 포트 충돌 가능성

### 발생 상황

수동으로 Mock 서버를 실행한 상태에서 Jenkins도 서버를 실행하려고 했다.

### 원인

이미 3000번 포트에서 `node server.js`가 실행 중이면 Jenkins에서 다시 서버를 실행할 때 충돌할 수 있다.

### 해결

Jenkins 빌드 전 기존 Node 프로세스를 종료했다.

```cmd
taskkill /F /IM node.exe
```

## 28.11 Jenkins 빌드가 오래 멈추는 문제

### 발생 상황

Jenkins 빌드가 `Check Node Version` 또는 `bat` 단계에서 오래 멈췄다.

### 의심 원인

* Jenkins가 Node.js 경로를 인식하지 못함
* Jenkins Tools의 NodeJS 설정 문제
* NodeJS 자동 설치 과정 지연
* 기존 node 프로세스 충돌
* 배치 파일 내부의 pause 명령어

### 정리

Jenkins에서 빌드가 멈출 경우, 무작정 기다리기보다 Console Output의 마지막 줄을 확인해야 한다.

특히 다음 문구를 확인해야 한다.

```text
node is not recognized
```

```text
EADDRINUSE
```

```text
Press any key to continue . . .
```

```text
Finished: FAILURE
```

문제 원인에 따라 NodeJS Tools 설정, PATH 설정, 포트 종료, 배치 파일 수정 등을 진행해야 한다.

## 29. 실습을 통해 이해한 Jenkins 흐름

이번 실습을 통해 Jenkins가 단순히 버튼 하나로 빌드하는 도구가 아니라는 것을 알게 되었다.

Jenkins는 다음 흐름으로 동작한다.

```text
GitHub 저장소 접근
→ main 브랜치 checkout
→ Jenkinsfile 읽기
→ Pipeline stage 실행
→ npm install
→ Mock 서버 실행
→ Newman 테스트 실행
→ HTML 리포트 생성
→ Artifacts 보관
```

즉, Jenkins에서 문제가 발생하면 다음 중 어느 단계에서 막혔는지를 먼저 확인해야 한다.

* GitHub 접근 문제
* 브랜치 문제
* Jenkinsfile 문제
* Node.js 실행 문제
* npm 패키지 설치 문제
* Mock 서버 실행 문제
* Newman 실행 문제
* 리포트 생성 문제

## 30. QA 관점에서 본 이번 실습의 의미

이번 실습은 단순한 개발 도구 사용법이 아니라, QA 업무에서 실제로 마주칠 수 있는 흐름을 체험한 실습이었다.

### QA 업무와 연결되는 부분

* 테스트 대상 리비전 확인
* API 응답 코드와 JSON 구조 확인
* 자동화 테스트 결과 리포트 확인
* 실패 로그를 보고 원인 범위 좁히기
* 개발자에게 재현 가능한 정보 전달

## 31. 개인 회고

이번 실습은 생각보다 많이 꼬였다.

특히 로컬 폴더, GitHub 저장소, Jenkins workspace가 서로 다르다는 점을 처음에는 명확히 구분하지 못했다.

처음에는 내가 CMD에서 실행한 폴더를 Jenkins도 그대로 보는 줄 알았다. 하지만 Jenkins는 로컬 폴더를 직접 보는 것이 아니라, GitHub 저장소에서 파일을 가져와 workspace에서 실행한다는 것을 알게 되었다.

또한 `node_modules`처럼 로컬에서 생성된 결과물을 그대로 GitHub에 올리는 것이 아니라, 필요한 원본 파일만 저장소에 올리고 빌드 과정에서 다시 설치해야 한다는 점도 배웠다.

Git 명령어도 작은 공백 하나 때문에 오류가 발생했다.

```bash
git add.
```

와

```bash
git add .
```

는 완전히 다르다.

처음에는 이런 부분이 너무 사소하게 느껴졌지만, 실제 실습에서는 이 작은 차이 때문에 다음 단계가 전부 막힐 수 있었다.

이번 실습을 통해 QA도 도구를 단순히 따라 쓰는 것에서 끝나면 안 된다고 느꼈다.

오류가 발생했을 때 Console Output을 보고, 문제가 Git인지, Jenkins인지, Node인지, 파일 경로인지, 포트 충돌인지 구분하는 능력이 필요하다.

아직 익숙하지는 않지만, 이번 실습을 통해 Jenkins와 Postman이 실제 QA 자동화 흐름에서 어떻게 연결되는지 감을 잡을 수 있었다.

## 32. 오늘의 핵심 정리

이번 수업에서 가장 중요하게 남은 내용은 다음과 같다.

```text
Git은 변경 이력을 관리한다.
Branch는 작업 환경을 분리한다.
Revision은 버그 리포트에서 중요하다.
API 테스트는 서버 요청과 응답을 검증한다.
Postman은 API를 수동으로 테스트한다.
Newman은 Postman 테스트를 자동 실행한다.
Jenkins는 빌드와 테스트 실행을 자동화한다.
Jenkins는 GitHub에서 Jenkinsfile을 읽어 Pipeline을 실행한다.
QA는 실패 로그를 보고 문제 원인을 분리할 수 있어야 한다.
```

이번 실습에서 가장 크게 배운 점은 다음이다.

> Jenkins에서 빌드가 실패했을 때는 화면만 보고 판단하지 말고, 반드시 Console Output을 확인해야 한다.

Console Output은 Jenkins에서 발생한 문제를 추적하는 가장 중요한 단서다.
