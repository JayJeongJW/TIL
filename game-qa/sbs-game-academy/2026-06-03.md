# 2026-06-02 QA 수업 정리 - QA 포트폴리오 흐름, ISTQB 개요, BVT/BAT/Smoke Test

## 오늘 수업 목표

오늘 수업에서는 지난 시간에 진행한 Postman, GitHub, Jenkins 실습 흐름을 다시 복기하고, QA 포트폴리오를 구성할 때 어떤 산출물이 연결될 수 있는지 학습했다.

또한 ISTQB Foundation Level의 기본 학습 범위를 간략히 훑어보고, 빌드 검증 단계에서 사용되는 BVT, BAT, Smoke Test 개념을 배웠다.

---

## 지난 실습 복기

지난 시간에는 Postman을 이용해 API 테스트를 수행하고, Newman을 통해 테스트 결과 리포트를 생성했다.

이후 실습 파일을 Git과 GitHub를 이용해 원격 저장소에 업로드하고, Jenkins에서 GitHub 저장소를 불러와 빌드 및 테스트 실행 흐름을 확인했다.

전체 흐름은 다음과 같다.

```text
Postman API 테스트
→ Newman 리포트 생성
→ GitHub 저장소 업로드
→ Jenkins Pipeline 실행
→ 테스트 결과 확인
```

이번 실습을 통해 QA 업무에서도 CI/CD 파이프라인의 흐름을 이해하는 것이 중요하다는 점을 알게 되었다.

---

## BVT / BAT / Smoke Test

BVT, BAT, Smoke Test 개념을 학습했다.

이 테스트들은 배포된 빌드가 본격적인 테스트를 수행할 수 있는 최소 조건을 만족하는지 빠르게 확인하기 위한 테스트다.

### BVT

BVT는 Build Verification Test의 약자다.

배포된 빌드가 테스트 가능한 상태인지 빠르게 확인한다.

예를 들어 길드 정원 확장 기능을 테스트해야 한다면, 최소한 다음 기능들이 정상 동작해야 한다.

```text
길드 생성
길드 가입
길드 탈퇴
길드 추방
```

이 기능들이 동작하지 않으면 본 테스트를 진행하기 어렵다.

따라서 BVT 단계에서 실패로 판단할 수 있다.

---

## BVT에서 중요한 기준

BVT는 전체 품질을 세부적으로 확인하는 테스트가 아니다.

본 테스트를 진행할 수 있는 최소 조건이 충족되었는지 확인하는 테스트다.

따라서 기능이 정상 동작한다면, 버튼 텍스트가 누락되는 정도의 UI 결함은 BVT 실패 사유가 아닐 수 있다.

반대로 테스트 대상 기능의 핵심 흐름이 동작하지 않으면 BVT 실패로 볼 수 있다.

---

## BVT 수행 시 중요한 점

BVT는 빠르게 수행되어야 한다.

목적은 세부 결함을 모두 찾는 것이 아니라, 해당 빌드가 테스트 가능한 상태인지 빠르게 판단하는 것이다.

따라서 체크리스트를 기반으로 짧은 시간 안에 진행하는 것이 중요하다.

---

## 오늘 배운 점

오늘 수업을 통해 QA 포트폴리오가 단순 문서 작성에서 끝나지 않고, 테스트 계획, 테스트케이스, 버그 리포트, 자동화 흐름까지 연결될 수 있다는 점을 알게 되었다.

또한 Jenkins, GitHub, Postman, Newman 같은 도구들은 각각 따로 쓰이는 것이 아니라 하나의 테스트 흐름 안에서 연결될 수 있다는 점을 다시 복습했다.

ISTQB 관련 내용은 아직 전체를 깊게 이해한 단계는 아니지만, 테스트의 기본 개념과 용어를 체계적으로 정리하는 데 도움이 될 수 있다는 것을 알게 되었다.

---

## 오늘의 핵심 정리

```text
BVT/BAT/Smoke Test는 본 테스트 전에 빌드가 테스트 가능한 상태인지 빠르게 확인하는 테스트다.
BVT는 세부 결함을 찾는 테스트가 아니라 최소 동작 조건을 확인하는 테스트다.
```
