# 2026-06-02 QA 수업 정리 - API 테스트, BVT, QA 단계 테스트, 디바이스 호환성

## 오늘 수업 개요

오늘 수업에서는 지난 시간에 다루었던 API 테스트 이론을 보강하고, BVT 이후 실제 QA 단계에서 수행되는 테스트 흐름을 학습했다.

또한 확인 테스트, 리그레션 테스트, 컴플라이언스, 사인오프, 모니터링 등 유지보수 프로젝트에서 반복적으로 수행되는 QA 업무 흐름을 정리했다.

수업 후반에는 비기능 테스트 중 하나인 디바이스 호환성 테스트에 대해 학습했다.

---

## API 테스트 기본 이론

API 테스트를 진행하려면 먼저 API를 호출할 수 있는 기본 정보가 필요하다.

API 테스트에서 확인해야 할 주요 정보는 다음과 같다.

```text
API 엔드포인트 주소
HTTP 메서드
요청 파라미터
요청 바디
응답 상태 코드
응답 데이터 구조
```

### API 엔드포인트

API 엔드포인트는 API를 호출할 수 있는 주소를 의미한다.

예를 들어 로그인 API, 랭킹 조회 API, 아이템 지급 API는 각각 다른 엔드포인트를 가질 수 있다.

QA는 API 테스트를 진행하기 위해 어떤 주소로 요청을 보내야 하는지 알아야 한다.

---

## HTTP 메서드

API는 목적에 따라 다른 HTTP 메서드를 사용한다.

### GET

기존 데이터를 조회할 때 사용한다.

예시는 다음과 같다.

```text
사용자 정보 조회
랭킹 정보 조회
아이템 정보 조회
이벤트 정보 조회
```

### POST

새로운 데이터를 생성할 때 사용한다.

예시는 다음과 같다.

```text
캐릭터 생성
닉네임 생성
아이템 지급
스테이지 추가
```

### PUT / PATCH

기존 데이터를 수정할 때 사용한다.

일반적으로 `PUT`은 전체 수정, `PATCH`는 일부 수정에 가깝다.

예시는 다음과 같다.

```text
닉네임 변경
직업 변경
스테이지 보상 수치 변경
캐릭터 정보 수정
```

### DELETE

기존 데이터를 삭제할 때 사용한다.

예시는 다음과 같다.

```text
캐릭터 삭제
아이템 삭제
게시글 삭제
데이터 제거
```

---

## HTTP 상태 코드

API 테스트에서는 응답 상태 코드도 함께 확인해야 한다.

상태 코드는 요청이 정상적으로 처리되었는지, 실패했다면 어느 쪽 문제인지 판단하는 기준이 된다.

```text
200번대: 정상 처리
300번대: 리디렉션
400번대: 클라이언트 에러
500번대: 서버 에러
```

### 200번대

정상적으로 요청이 처리된 상태다.

예를 들어 `200 OK`, `201 Created`가 있다.

### 400번대

클라이언트 요청에 문제가 있는 상태다.

예를 들어 잘못된 파라미터, 인증 실패, 권한 부족, 존재하지 않는 경로 요청 등이 포함될 수 있다.

### 500번대

서버 내부에서 문제가 발생한 상태다.

서버 로직 오류, DB 오류, 예외 처리 실패 등이 원인이 될 수 있다.

---

## BVT / BAT / Smoke Test

BVT, BAT, Smoke Test는 배포된 빌드가 본격적인 테스트를 진행할 수 있는 최소 조건을 만족하는지 빠르게 확인하는 테스트다.

목적은 세부 결함을 모두 찾는 것이 아니라, **이 빌드로 본 테스트를 진행할 수 있는지 판단하는 것**이다.

---

## BVT에서 중요한 기준

BVT에서 가장 중요한 것은 시간이다.

BVT는 오래 진행하는 테스트가 아니다.

게임 장르에 따라 차이는 있지만, 제한된 시간 안에 핵심 기능이 동작하는지만 빠르게 확인해야 한다.

예시는 다음과 같다.

```text
서브컬처 RPG: 약 2시간 이내
MMORPG: 약 3시간 이내
일반 앱: 약 30분 이내
```

BVT는 체크리스트를 기반으로 수행하는 것이 일반적이다.

결과도 복잡하게 판단하기보다 다음처럼 이분법적으로 확인한다.

```text
동작한다
동작하지 않는다
테스트 가능하다
테스트 불가능하다
```

조직의 성숙도에 따라 수동으로 진행할 수도 있고, 자동화 테스트를 적용할 수도 있다.

---

## QA 단계 테스트

BVT 이후 본격적인 QA 단계에서는 기획 요구사항이 정상적으로 구현되었는지 총체적으로 확인한다.

이 단계에서는 다음과 같은 테스트가 포함될 수 있다.

```text
시스템 테스트
시스템 통합 테스트
기능 전수 검사
업데이트 테스트
패치노트 테스트
N차 테스트
```

QA 단계 테스트는 런칭 스펙 또는 업데이트 스펙에 대한 전수 검사에 가깝다.

가장 많은 리소스가 투입되는 단계이며, 테스트케이스를 기반으로 진행된다.

---

## 테스트 커버리지

테스트 커버리지는 계획한 테스트 대비 실제 수행한 테스트의 양을 의미한다.

```text
테스트 커버리지 = 실제 수행한 테스트 양 / 계획한 테스트 양
```

QA 단계에서는 테스트 커버리지를 최대한 만족하면서, 검출된 결함을 개발팀에 전달하고 수정 여부를 확인해야 한다.

---

## 확인 테스트

확인 테스트는 이전 차수에서 보고한 결함이 정상적으로 수정되었는지 확인하는 테스트다.

예를 들어 1차 테스트에서 결함 100건을 보고했고, 개발팀이 해당 결함을 수정한 빌드 2를 QA팀에 전달했다고 가정한다.

이때 QA팀이 먼저 확인해야 하는 것은 다음이다.

```text
1차 테스트에서 보고한 결함들이 정상적으로 수정되었는가?
```

이처럼 수정된 결함을 다시 확인하는 테스트를 확인 테스트라고 한다.

---

## 리그레션 테스트

리그레션 테스트는 코드 수정이나 기능 추가로 인해 기존 기능에 새로운 문제가 발생하지 않았는지 확인하는 테스트다.

예를 들어 우편함 문제를 수정했는데, 직접 관련 없어 보이는 퀘스트 기능에서 새로운 문제가 발생할 수 있다.

이런 의도하지 않은 영향을 사이드 이펙트라고 한다.

```text
사이드 이펙트:
코드 변경이나 추가 작업으로 인해 의도하지 않은 문제가 다른 영역에서 발생하는 현상
```

---

## 리그레션 테스트가 필요한 이유

수정된 부분만 확인하면 다른 영역에서 발생한 문제를 놓칠 수 있다.

이상적으로는 처음부터 끝까지 모든 기능을 다시 테스트하는 것이 가장 안전하다.

하지만 현실적으로 시간과 인력이 제한되어 있기 때문에, 효과적인 리그레션 시나리오를 만들어 매 빌드마다 반복적으로 수행하는 방식이 필요하다.

리그레션 테스트는 반복적인 성격을 가진다.

따라서 자주 수행되는 시나리오는 자동화를 적용하기 좋다.

---

## 컴플라이언스 테스트

컴플라이언스 테스트는 플랫폼 정책을 검증하는 테스트다.

게임은 출시 플랫폼에 따라 지켜야 하는 정책이 다르다.

예시는 다음과 같다.

```text
Google Play 정책
Apple App Store 정책
Steam 정책
Console 플랫폼 정책
```

콘솔 플랫폼의 경우 각 플랫폼마다 별도의 인증 기준이 존재한다.

```text
Sony: TRC
Microsoft: XR / TCR
Nintendo: LOT Check
```

플랫폼 정책 검증은 실제 서비스 출시나 심사 과정에서 중요하게 작용할 수 있다.

---

## 사인오프

사인오프는 테스트가 종료된 이후 또는 주어진 리소스를 모두 소진한 이후, 서비스를 진행할지 중단할지 논의하는 과정이다.

즉, 현재 빌드나 서비스 상태를 기준으로 `Go / Stop`을 판단하는 논의체에 가깝다.

---

## 모니터링

테스트 이후에는 모니터링 업무도 진행된다.

주요 모니터링 항목은 다음과 같다.

```text
게임 모니터링
커뮤니티 모니터링
테스트웨어 정리
```

테스트웨어에는 테스트 계정, 테스트 아이템, 치트성 아이템 등이 포함될 수 있다.

테스트 종료 후에는 이러한 테스트용 요소를 원복, 삭제, 탈퇴 처리하는 마감 작업이 필요하다.

---

## 유지보수 프로젝트의 반복 테스트 흐름

수업에서는 위 테스트 흐름이 유지보수 프로젝트에서 루틴하게 반복될 수 있다고 설명했다.

예를 들어 2주 단위 업데이트가 있는 프로젝트라면 다음 흐름이 반복될 수 있다.

```text
BVT
QA 단계 테스트
확인 테스트
리그레션 테스트
컴플라이언스 확인
사인오프
모니터링
테스트웨어 정리
```

# 비기능 테스트

수업 후반에는 루틴하게 진행되지 않는 비기능 테스트에 대해 학습했다.

비기능 테스트는 기능 자체가 정상 동작하는지만 보는 것이 아니라, 성능, 호환성, 안정성, 사용성 등 기능 외적인 품질 특성을 확인하는 테스트다.

이번 시간에는 그중 디바이스 호환성 테스트를 중심으로 학습했다.

---

## 디바이스 호환성 테스트

디바이스 호환성 테스트는 특정 디바이스에 게임을 설치하고 실행하고 운용하는 과정에서 특이사항이 발생하지 않는지 확인하는 테스트다.

게임은 다양한 하드웨어, OS, 드라이버, 화면 비율, 입력 환경에서 실행될 수 있다.

따라서 QA는 여러 환경에서 게임이 정상적으로 실행되는지 확인해야 한다.

---

## PC 환경에서 확인할 항목

PC 게임에서는 다음과 같은 항목을 기준으로 호환성을 확인할 수 있다.

```text
CPU
GPU
RAM
모니터 비율
단일/다중 모니터 환경
OS 버전
GPU Driver
브라우저
보안 프로그램 충돌
```

예를 들어 웹 런처를 통해 실행되는 게임은 브라우저별 테스트가 필요할 수 있다.

```text
Chrome
Edge
Whale
Firefox
Safari
```

또한 게임 내 보안 프로그램과 금융권 보안 프로그램이 충돌하는지도 확인 대상이 될 수 있다.

---

## 모바일 환경에서 확인할 항목

모바일 게임에서는 다음과 같은 항목을 확인할 수 있다.

```text
제조사
CPU
GPU
RAM
화면 비율
OS 버전
Google Play Games 환경
인터럽트
포그라운드 / 백그라운드 전환
앱 공존성
```

모바일 환경은 디바이스 종류가 매우 다양하기 때문에, 대표 기기군을 선정해 테스트하는 것이 중요하다.

---

## 인터럽트 테스트

인터럽트 테스트는 게임 플레이 중 외부 이벤트가 발생했을 때 게임에 부정적인 영향이 없는지 확인하는 테스트다.

예시는 다음과 같다.

```text
전화 수신
문자 또는 앱 알림
시스템 알림
배터리 부족 알림
권한 팝업
```

게임 도중 이러한 인터럽트가 발생했을 때 게임이 멈추거나, 보상이 사라지거나, 세션이 비정상 종료되지 않는지 확인해야 한다.

---

## 포그라운드 / 백그라운드 전환 테스트

모바일 게임에서는 앱이 포그라운드와 백그라운드를 오가는 상황이 자주 발생한다.

확인해야 할 항목은 다음과 같다.

```text
빠른 전환 반복 시 게임이 정상 유지되는가?
백그라운드 일정 시간 유지 후 복귀 시 세션 처리가 정상인가?
전투 중 백그라운드 전환 후 복귀 처리가 정상인가?
보상 수령 중 앱 전환 시 데이터가 정상 유지되는가?
```

수업에서는 빠른 전환을 여러 번 반복하는 방식도 언급되었다.

---

## 앱 공존성 확인

앱 공존성 확인은 백그라운드에 여러 앱이 실행 중인 상태에서 게임을 설치하고 실행하고 운용했을 때 문제가 없는지 확인하는 테스트다.

특히 메모리가 낮은 디바이스에서는 메모리 부족으로 게임이 종료되거나 성능 문제가 발생할 수 있다.

따라서 저사양 디바이스 환경에서도 확인이 필요하다.

---

## 오늘의 핵심 정리

```text
API 테스트는 엔드포인트, 메서드, 상태 코드, 응답 데이터를 확인해야 한다.
GET은 조회, POST는 생성, PUT/PATCH는 수정, DELETE는 삭제에 사용된다.
BVT는 본 테스트를 진행할 수 있는 최소 조건을 빠르게 확인하는 테스트다.
QA 단계 테스트는 업데이트 스펙이 정상 구현되었는지 테스트케이스 기반으로 확인한다.
확인 테스트는 수정된 결함이 정상적으로 고쳐졌는지 확인하는 테스트다.
```
