## 2. 네트워크 호환성 테스트

게임은 사용자의 네트워크 환경에 따라 속도, 안정성, 응답 시간이 달라질 수 있다.

따라서 정상적인 고속 네트워크뿐 아니라 느린 속도, 높은 지연, 패킷 손실, 네트워크 전환 및 단절 상황에서도 게임이 안정적으로 동작하는지 확인해야 한다.

### 무선 네트워크 환경

* Wi-Fi 5
* Wi-Fi 6
* Wi-Fi 6E
* Wi-Fi 7
* 2.4GHz
* 5GHz
* 6GHz

### 이동통신 환경

* 3G
* 4G / LTE
* 5G

통신사별로 모든 테스트를 반복하기보다는, 대표적인 네트워크 환경에서 설치·실행·유지 여부를 확인하는 방식으로 접근할 수 있다.

---

## 3. 저속 네트워크 환경을 테스트하는 이유

모바일 데이터가 부족하거나 통신 상태가 좋지 않으면 사용자의 실제 네트워크 속도가 크게 제한될 수 있다.

이때 게임이 단순히 느려지는 것에서 끝나는 것이 아니라 다음과 같은 문제가 발생할 수 있다.

* 게임 크래시
* 순간적인 랙
* 접속 끊김
* 서버 응답 대기 상태 지속
* 결제 결과 누락
* 강화 재료만 차감되는 문제
* PvP 매칭 상태 비정상 처리
* 중복 보상 또는 보상 누락

따라서 저속 환경과 불안정한 환경에서도 서버 요청이 안전하게 처리되는지 확인해야 한다.

단, 데이터 소진 이후 반드시 3G 통신망으로 전환되는 것은 아니다. 실제로는 LTE나 5G 연결 상태를 유지하면서 통신 속도만 제한되는 경우도 있으므로, 실무에서는 특정 통신 세대보다 실제 속도와 지연 조건을 기준으로 테스트하는 것이 중요하다.

## 4. 네트워크 전환 테스트

게임 플레이 중 다음과 같은 네트워크 전환 상황이 발생할 수 있다.

```text
Wi-Fi → 모바일 데이터
모바일 데이터 → Wi-Fi
5GHz Wi-Fi → 2.4GHz Wi-Fi
네트워크 연결 → 일시 단절
일시 단절 → 재연결
```

특히 다음 상황에서 네트워크가 변경되거나 끊기면 서버 데이터가 잘못 처리될 가능성이 있다.

* PvP 매칭 진행 중
* 실시간 전투 진행 중
* 결제 프로세스 진행 중
* 아이템 강화 진행 중
* 보상 수령 중
* 스테이지 결과 저장 중
* 캐릭터 생성 또는 닉네임 변경 중

QA는 단순히 다시 연결되는지만 확인하면 안 된다.

네트워크 단절 전후로 다음 항목도 함께 확인해야 한다.

* 재화가 중복 차감되지 않는가
* 보상이 중복 지급되지 않는가
* 진행 결과가 누락되지 않는가
* 서버와 클라이언트 데이터가 일치하는가
* 적절한 오류 메시지나 재시도 안내가 제공되는가
* 세션이 기획된 정책대로 유지되거나 종료되는가

## 5. 약전계 환경

약전계 환경은 전파가 약해 네트워크 연결이 불안정한 환경을 의미한다.

예시는 다음과 같다.

* 오피스 건물 엘리베이터 내부
* 비상구 계단
* 깊은 지하 주차장
* 섬 지역
* 산간 지역
* 건물 내부 음영 지역

약전계 환경에서는 요청이 즉시 실패하지 않고 일정 시간 동안 응답을 기다리는 상태가 발생할 수 있다.

게임에서는 이때 로딩 스피너가 계속 표시되거나, 일정 횟수 또는 제한 시간을 넘은 뒤 오류 팝업이 나타날 수 있다.

예를 들어 클라이언트와 서버 사이의 요청 제한이 설정되어 있다면 다음과 같은 정책을 확인해야 한다.

```text
요청 재시도 제한 횟수 도달
→ 추가 요청 중단
→ 네트워크 오류 팝업 표시
→ 재시도 또는 로비 이동 제공
```

구체적인 재시도 횟수와 제한 시간은 프로젝트의 개발 정책에 따라 달라질 수 있다.

## 6. 네트워크 관련 주요 개념

### 패킷

패킷은 네트워크를 통해 전달되는 작은 데이터 단위다.

택배에 비유하면, 하나의 큰 물건을 여러 개의 작은 상자로 나누어 보내는 것과 비슷하다.

### 패킷 손실률

전송된 패킷 중 목적지에 도착하지 못한 패킷의 비율이다.

```text
패킷 손실률이 낮음
→ 비교적 안정적인 통신

패킷 손실률이 높음
→ 끊김, 재전송, 데이터 누락 가능성 증가
```

### 레이턴시

요청을 보낸 뒤 응답이 돌아오기까지 걸리는 지연 시간이다.

택배가 이틀 안에 도착해야 하는데 일주일 뒤 도착하는 것에 비유할 수 있다.

단위는 보통 `ms`를 사용한다.

### 대역폭

일정 시간 동안 전송할 수 있는 데이터의 양이다.

도로에 비유하면 다음과 같다.

```text
낮은 대역폭
→ 좁은 1차선 도로

높은 대역폭
→ 넓은 8차선 도로
```

대역폭이 낮으면 같은 양의 데이터를 전송하는 데 더 오랜 시간이 걸릴 수 있다.

## 7. Network Emulator for Windows Toolkit 실습

Windows 환경에서 Network Emulator for Windows Toolkit을 설치하고 약전계 환경을 인위적으로 구성하는 실습을 진행했다.

### 실습 전 확인 항목

* 현재 사용 중인 네트워크 어댑터
* IPv4 주소
* 적용할 네트워크 프로토콜
* 에뮬레이션 대상 프로그램
* 패킷 손실률
* 레이턴시
* 대역폭

실습에서는 네트워크 필터를 생성하고, 선택한 어댑터에 링크를 연결한 뒤 네트워크 제한값을 설정했다.

### 약전계 환경 설정값

```text
Packet Loss
- Random Loss
- Loss Rate: 0.3

Latency
- Fixed
- 80ms

Bandwidth & Queue
- Bandwidth
- 200kbps
```

`Loss Rate 0.3`은 도구가 소수 비율을 사용하는 경우 약 30%의 손실을 의미할 수 있다.

이는 일반적인 약한 네트워크보다 상당히 가혹한 조건이므로, 의도한 단위와 입력 형식을 반드시 확인해야 한다.

### 트래픽 확인

설정을 적용한 뒤 다음 기능을 이용해 트래픽 상태를 확인했다.

```text
Action
→ Toggle Tracing 활성화
→ RT Traffic Monitor 확인
```

## 8. IP 주소 확인 시 주의사항

실습 중 검색 사이트를 통해 공인 IP 주소를 확인했다.

하지만 네트워크 도구의 `Local IPv4` 항목이 실제 PC의 로컬 어댑터 주소를 요구하는 경우에는 검색 사이트에서 보이는 공인 IP가 아니라, `ipconfig`에서 확인되는 IPv4 주소를 사용해야 할 수 있다.

예시는 다음과 같다.

```text
공인 IP
→ 인터넷 외부에서 보이는 주소

로컬 IPv4
→ 공유기 내부에서 PC에 할당된 주소
→ 예: 192.168.x.x
```

실습 도구가 어떤 주소를 요구하는지 확인하지 않고 공인 IP를 입력하면 필터가 의도한 방식으로 적용되지 않을 수 있다.

## 9. 글로벌 네트워크 환경

서비스 지역이 해외로 확장되면 현지 네트워크 환경에서도 테스트가 필요하다.

주요 확인 대상은 다음과 같다.

* 게임 설치
* 게임 실행
* 로그인
* PvP
* 보스전
* 실시간 동기화 콘텐츠
* 채팅
* 서버 응답 지연
* 연결 해제 및 재접속

글로벌 테스트 환경을 만들 때는 단순히 국가 평균값만 사용하는 것이 아니라, 실제 서비스 대상 지역의 불리한 조건도 고려할 수 있다.

실습에서는 다음 조건을 가정했다.

```text
Country     : Brazil
City        : Nova Iguaçu
ISP         : Nio

Packet Loss : 2%
Latency     : 80ms RTT
Download    : 5Mbps
Upload      : 1Mbps
```

위 값은 실습을 위해 설정한 가정값이며, 실제 서비스 테스트에서는 신뢰할 수 있는 측정 자료와 사용자 분포를 기반으로 값을 선정해야 한다.

## 10. 글로벌 환경 실습 문제

글로벌 환경값을 적용했을 때 게임뿐 아니라 인터넷 연결 자체가 동작하지 않는 문제가 발생했다.

약전계 설정을 제거한 뒤 글로벌 환경값으로 다시 구성했지만 같은 현상이 발생했다.

또한 패킷 손실, 레이턴시, 대역폭 설정을 각각 비활성화했음에도 인터넷이 정상적으로 복구되지 않았다.

### 발생 결과

* 게임 실행 불가
* 인터넷 사용 불가
* 설정값 비활성화 후에도 동일 현상
* 수강생 약 3분의 2에서 유사 현상 발생
* 강사 환경에서도 즉시 원인 확인 불가

### 의심 가능한 원인

정확한 원인은 확인하지 못했지만 다음 가능성을 생각할 수 있다.

* 프로그램과 현재 Windows 환경의 호환성 문제
* 네트워크 필터 드라이버 충돌
* 잘못된 네트워크 어댑터 선택
* 공인 IP와 로컬 IP 혼동
* Include / Exclude 필터 설정 문제
* 이전 약전계 설정이 완전히 해제되지 않음
* 보안 프로그램 또는 방화벽과의 충돌
* 가상 네트워크 어댑터와의 충돌

이번 실습은 최종적으로 네트워크 에뮬레이션 기능과 설정 방법을 확인하는 수준에서 종료했다.

정상적인 테스트 결과를 얻지 못했기 때문에 성공한 네트워크 테스트로 기록하지 않고, 원인을 확인하지 못한 트러블슈팅 사례로 남겼다.

## 11. 외부 기기 연결 테스트

게임은 외부 기기와 연결되거나 연결이 해제될 때도 정상적으로 동작해야 한다.

주요 테스트 대상은 다음과 같다.

* 유선 이어폰
* 무선 이어폰
* 게임 컨트롤러
* 스마트워치 등 웨어러블 기기

### 오디오 기기 테스트

게임 실행 중 외부 스피커에서 이어폰으로 출력 장치가 변경될 때 다음 문제가 발생할 수 있다.

* 전체 소리가 출력되지 않음
* 배경음은 들리지만 효과음이 들리지 않음
* 효과음은 들리지만 배경음이 들리지 않음
* 이어폰 연결 해제 후 외부 스피커로 소리가 복구되지 않음
* 블루투스 재연결 후 음량이 비정상적으로 변경됨

따라서 게임 실행 전후 및 플레이 중 연결·해제를 모두 확인해야 한다.

## 12. 클라이언트 성능 테스트

클라이언트 성능 테스트는 게임 클라이언트가 하드웨어 자원을 얼마나 효율적으로 사용하는지 확인하는 테스트다.

주요 측정 항목은 다음과 같다.

* 평균 프레임
* 하위 1% 프레임
* CPU 사용량
* GPU 사용량
* RAM 사용량
* 디바이스 발열
* 배터리 소모량
* 로딩 시간
* 프레임 드롭 구간

평균 프레임만 높더라도 순간적으로 프레임이 크게 떨어지면 플레이 경험이 불안정할 수 있다.

따라서 평균 프레임과 함께 하위 프레임도 확인할 필요가 있다.

### 사용 도구

모바일 환경에서는 다음 도구를 사용할 수 있다.

* ADB
* Android Studio
* Xcode

PC 환경에서는 다양한 벤치마크 및 모니터링 도구를 활용할 수 있다.

현업 모바일 QA에서는 ADB와 Xcode를 활용할 수 있다.

## 13. ADB 실습 준비

Android 환경에서 ADB를 사용하기 위해 다음 조건이 필요하다.

### 실제 디바이스 사용

* 데이터 통신이 가능한 USB 케이블
* Android SDK Platform Tools
* 개발자 옵션 활성화
* USB 디버깅 활성화
* PC와 디바이스 연결 승인

### 앱 플레이어 사용

실습에서는 Nox 앱플레이어를 이용하는 방법도 안내되었다.

```text
설정
→ 시스템
→ 휴대전화 또는 태블릿 정보
→ 소프트웨어 정보
→ 빌드 번호 여러 번 클릭
→ 개발자 옵션 활성화
→ USB 디버깅 활성화
```

### 패키지명 확인

ADB로 특정 게임을 분석하려면 해당 게임의 패키지명이 필요하다.

Google Play 웹페이지 주소의 `id=` 뒤에서 패키지명을 확인할 수 있다.

예시:

```text
https://play.google.com/store/apps/details?id=com.example.game&hl=ko
```

패키지명:

```text
com.example.game
```

`&hl=ko`와 같이 `&` 뒤에 붙는 값은 추가 URL 파라미터이므로 패키지명에서 제외한다.

## 14. 개인 실습 진행 상태

이번 시간에는 데이터 통신이 가능한 USB 케이블을 준비하지 못했다.

또한 Nox 앱플레이어를 실행하면 PC가 완전히 종료되는 문제가 발생해 앱플레이어를 통한 실습도 진행하지 못했다.

따라서 다른 수강생의 실습 과정을 관찰하며 ADB 설정 방법과 패키지명 확인 방법을 메모했다.

다음 수업 전까지 데이터 케이블을 준비한 뒤 실제 Android 디바이스로 ADB 실습을 진행할 예정이다.

Nox 실행 시 PC가 강제 종료되는 현상은 일반적인 프로그램 오류보다 시스템 안정성과 관련될 수 있으므로, 원인을 확인하기 전까지 반복 실행하지 않는 것이 안전하다.

## 15. 서버 성능 테스트

서버 성능 테스트는 많은 사용자가 동시에 접속하거나 요청을 보낼 때 서버가 안정적으로 처리할 수 있는지 확인하는 테스트다.

게임 서버 성능 테스트는 다음과 같은 지식이 필요할 수 있다.

* 서버 구조
* 네트워크
* 동시 접속
* 요청 처리량
* 응답 시간
* DB 처리
* 부하 분산

대표 도구로는 다음이 언급되었다.

* JMeter
* 상용 서버 성능 테스트 도구

서버 성능 테스트는 수행 난도가 높고 개발자 수준의 서버·네트워크 지식이 필요할 수 있다.

게임 QA 단일팀에서 독립적으로 수행하기보다는 서버 개발팀, 인프라팀, 성능 전문팀과 협업하는 경우가 많다.

공공 서비스, 금융 서비스, AI 서비스처럼 대규모 요청 처리와 장애 예방이 중요한 환경에서 특히 중요하게 사용된다.

## 16. 오늘 배운 점

오늘 수업을 통해 네트워크 테스트는 단순히 인터넷이 연결되는지만 확인하는 작업이 아니라는 점을 알게 되었다.

패킷 손실, 지연 시간, 대역폭, 네트워크 전환과 같은 조건이 게임의 결제, 강화, PvP, 보상 처리에 직접적인 영향을 줄 수 있다.

또한 네트워크가 끊기는 것 자체보다, 끊기는 순간에 서버 데이터가 중복되거나 누락되지 않는지 확인하는 것이 더 중요하다는 점을 배웠다.

실습은 프로그램 또는 환경 문제로 정상 완료하지 못했지만, 실패한 결과를 통해 네트워크 어댑터, IP 주소, 필터 규칙, 프로그램 호환성을 먼저 확인해야 한다는 점을 알 수 있었다.

클라이언트 성능 테스트는 다음 수업에서 ADB를 이용해 이어서 진행할 예정이다.

```text
패킷 손실은 전송 데이터 일부가 목적지에 도착하지 못하는 현상이다.

레이턴시는 요청과 응답 사이의 지연 시간이다.

글로벌 환경 테스트는 서비스 대상 지역의 실제 네트워크 조건을 고려해야 한다.

외부 기기 테스트는 연결과 해제 전후의 오디오 및 입력 상태를 확인한다.

ADB는 Android 디바이스 연결과 상태 확인, 성능 정보 수집 등에 사용할 수 있다.

정상적으로 완료되지 않은 실습은 성공 사례가 아니라 트러블슈팅 기록으로 남겨야 한다.
```
