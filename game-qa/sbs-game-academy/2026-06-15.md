# 2026-06-15 QA 수업 정리 - ADB를 활용한 모바일 게임 성능 테스트

## 1. 오늘 수업 개요

오늘 수업에서는 Android 디바이스와 ADB를 이용해 모바일 게임의 성능 정보를 확인하는 방법을 실습했다.

성능 테스트는 게임이 정상적으로 실행되는지만 확인하는 것이 아니라, 플레이 중 기기 자원을 얼마나 사용하는지 확인하는 테스트다.

이번 수업에서는 다음 항목을 중심으로 학습했다.

* ADB 연결 확인
* 게임 패키지명 확인
* CPU 사용량 확인
* 프레임 / 렌더링 정보 확인
* 메모리 사용량 확인
* 발열 온도 확인
* 배터리 사용량 확인
* Error 로그 확인
* 성능 테스트 결과 리포트 작성 방향

---

## 2. 실습 대상

실습에서는 모바일 게임 `블루 아카이브`를 대상으로 진행했다.

```text
게임명: 블루 아카이브
패키지명: com.nexon.bluearchive
```

성능 테스트 중에는 게임이 실제로 실행 중이어야 한다.

단순히 앱을 켜놓은 Idle 상태보다는, 전투나 로비 이동처럼 실제 플레이가 진행되는 상태에서 측정하는 것이 더 적절하다.

---

## 3. ADB 실행 위치

ADB 명령어는 반드시 `adb.exe`가 있는 폴더에서 실행해야 한다.

Android 실제 기기를 사용하는 경우:

```text
Android SDK Platform Tools 폴더
```

Nox 앱플레이어를 사용하는 경우:

```text
Program Files (x86) → Nox → bin
```

CMD에서 `adb devices`를 입력했을 때 기기가 `device` 상태로 표시되면 정상 연결된 것이다.

```cmd
adb devices
```

예시:

```text
List of devices attached
R3CW60641PB    device
```

---

## 4. 패키지명 확인

ADB로 특정 게임의 성능 정보를 확인하려면 게임의 패키지명이 필요하다.

Google Play 웹페이지 주소에서 `id=` 뒤부터 `&` 앞까지가 패키지명이다.

예시:

```text
https://play.google.com/store/apps/details?id=com.nexon.bluearchive&hl=ko
```

패키지명:

```text
com.nexon.bluearchive
```

`&hl=ko` 이후의 값은 언어, 국가, 유입 경로 등의 URL 옵션이므로 패키지명에 포함하지 않는다.

---

## 5. CPU 사용량 확인

게임 실행 중 CPU 사용량은 다음 명령어로 확인할 수 있다.

```cmd
adb shell dumpsys cpuinfo com.nexon.bluearchive
```

CPU 사용량은 게임 프로세스가 CPU를 얼마나 사용하고 있는지 보여준다.

Android 기기는 여러 CPU 코어를 사용하기 때문에 CPU 사용량이 100%를 넘을 수 있다.
예를 들어 180%로 표시되었다면, 하나의 코어만 사용하는 것이 아니라 여러 코어를 동시에 사용하고 있다는 의미로 볼 수 있다.

CPU 사용량만으로 성능 문제를 바로 단정할 수는 없다.
게임의 플레이 상황, 로딩 여부, 백그라운드 앱, 발열 상태 등을 함께 확인해야 한다.

---

## 6. 프레임 / 렌더링 정보 확인

프레임과 렌더링 관련 정보는 다음 명령어로 확인할 수 있다.

```cmd
adb shell dumpsys gfxinfo com.nexon.bluearchive
```

이 명령어는 CPU 사용량을 확인하는 명령어가 아니라, 화면 렌더링과 프레임 안정성을 확인하는 명령어다.

주로 다음 항목을 볼 수 있다.

* 프레임 처리 시간
* Jank 발생 여부
* 렌더링 지연
* GPU 처리 시간

Jank가 많다는 것은 화면이 부드럽게 이어지지 않고 끊김이 발생할 가능성이 있다는 의미다.

---

## 7. 메모리 사용량 확인

게임의 메모리 사용량은 다음 명령어로 확인할 수 있다.

```cmd
adb shell dumpsys meminfo com.nexon.bluearchive
```

메모리 사용량이 높으면 장시간 플레이 중 앱 종료, 프레임 저하, 로딩 지연 등의 문제가 발생할 수 있다.

단, 메모리 사용량 역시 단일 수치만 보고 문제를 단정하면 안 된다.
로비, 전투, 반복 전투, 백그라운드 복귀 등 여러 상황에서 비교해야 한다.

---

## 8. 발열 온도 확인

배터리 온도는 다음 명령어로 확인할 수 있다.

```cmd
adb shell dumpsys battery | findstr /i "temperature"
```

발열은 장시간 플레이 품질에 영향을 줄 수 있다.

발열이 높아지면 다음 문제가 발생할 수 있다.

* 프레임 저하
* 터치 반응 저하
* 배터리 소모 증가
* 기기 성능 제한
* 앱 강제 종료 가능성 증가

따라서 성능 테스트에서는 CPU, 메모리와 함께 발열도 함께 확인해야 한다.

---

## 9. 배터리 사용량 확인

배터리 관련 통계는 다음 명령어로 확인할 수 있다.

```cmd
adb shell dumpsys batterystats
```

특정 앱 기준으로 확인할 때는 다음 명령어를 사용할 수 있다.

```cmd
adb shell dumpsys batterystats --charged com.nexon.bluearchive
```

결과에서 테스트 대상 앱의 UID를 찾고, 해당 UID의 전력 사용량을 확인한다.

예시:

```text
UID u0a493
Estimated power use
```

배터리 사용 비율은 다음과 같이 계산할 수 있다.

```text
게임 배터리 사용 비율 = 게임 배터리 소모량 ÷ 전체 배터리 소모량 × 100
```

예시:

```text
게임 소모량: 138
전체 소모량: 2583

138 ÷ 2583 × 100 = 약 5.34%
```

이 값은 전체 배터리 소모량 중 해당 게임이 차지한 비중을 의미한다.

전체 배터리 용량 기준으로 계산할 수도 있다.

```text
게임 소모량 ÷ 배터리 전체 용량 × 100
```

예시:

```text
138 ÷ 4855 × 100 = 약 2.84%
```

이 값은 전체 배터리 용량 중 해당 게임이 사용한 비율을 의미한다.

단, `batterystats`는 누적 통계의 영향을 받을 수 있으므로 결과를 해석할 때 주의해야 한다.

---

## 10. Error 로그 확인

게임 실행 중 발생하는 Error 로그는 `logcat`을 통해 확인할 수 있다.

먼저 테스트 대상 게임의 PID를 확인한다.

```cmd
adb shell ps | findstr com.nexon.bluearchive
```

PID를 확인한 뒤, 해당 프로세스의 Error 로그만 필터링해서 확인할 수 있다.

```cmd
adb logcat --pid=PID값 *:E
```

예시:

```cmd
adb logcat --pid=6891 *:E
```

이 명령어를 실행한 상태로 게임을 플레이하면, 특정 시점에 발생한 Error 로그를 확인할 수 있다.

버그 리포트를 작성할 때는 다음 정보를 함께 기록하는 것이 좋다.

* 어떤 화면에서 발생했는가
* 어떤 조작을 했는가
* 발생 시점의 로그는 무엇인가
* 크래시, ANR, OutOfMemoryError가 있었는가
* 반복 재현이 가능한가

## 11. 현업에서의 성능 테스트 방식

수업에서는 현업에서는 명령어를 사람이 매번 직접 입력하기보다, 자동화 스크립트를 만들어 결과를 수집한다고 설명했다.

예를 들어 15분 또는 30분 동안 게임을 플레이하면서 일정 간격으로 다음 정보를 수집할 수 있다.

* CPU 사용량
* 프레임 / 렌더링 정보
* 메모리 사용량
* 발열 온도
* 배터리 사용량
* Error 로그
* 플레이 화면 스크린샷

예시 흐름:

```text
30분 플레이
→ 3분 간격으로 성능 정보 수집
→ 주요 구간 스크린샷 저장
→ 로그 파일 정리
→ 성능 테스트 결과 보고서 작성
```

## 12. 배치 파일이란?

배치 파일은 여러 CMD 명령어를 한 번에 실행할 수 있도록 만든 자동 실행 파일이다.

확장자는 `.bat`을 사용한다.

매번 ADB 명령어를 직접 입력하는 대신, 배치 파일 안에 명령어를 미리 적어두면 더블클릭 또는 CMD 실행만으로 명령어를 자동 실행할 수 있다.

예시:

```bat
@echo off
adb devices
adb shell dumpsys cpuinfo com.nexon.bluearchive > cpuinfo.txt
adb shell dumpsys gfxinfo com.nexon.bluearchive > gfxinfo.txt
adb shell dumpsys meminfo com.nexon.bluearchive > meminfo.txt
adb shell dumpsys battery > battery.txt
pause
```

성능 테스트에서는 배치 파일을 이용해 반복 측정을 자동화할 수 있다.

## 13. 성능 테스트 결과 보고서 구성

성능 테스트 결과는 단순히 로그만 저장하는 것이 아니라, 보고서 형태로 정리해야 한다.

보고서에는 다음 항목이 포함될 수 있다.

* 표지
* 목차
* 게임 소개
* 테스트 개요 및 환경
* 테스트 시나리오
* 결과 및 분석
* CPU 사용량 분석
* 프레임 안정성 분석
* 메모리 사용량 분석
* 발열 분석
* 배터리 사용량 분석
* Error 로그 분석
* 스크린샷 설명
* 종합 평가
* 개선 방안 제안
* 마무리

다만 성능 결과를 해석할 때는 단일 수치만 보고 문제를 단정하지 않아야 한다.
동일한 조건에서 반복 측정하고, 여러 항목을 함께 비교해야 신뢰도 있는 분석이 된다.

---

## 14. 오늘 배운 점

오늘 수업을 통해 ADB를 활용하면 Android 게임의 CPU, 프레임, 메모리, 발열, 배터리, 로그 정보를 확인할 수 있다는 것을 배웠다.

성능 테스트는 단순히 “게임이 실행되는가”를 보는 것이 아니라, 실제 플레이 중 기기 자원이 어떻게 사용되는지 확인하는 과정이다.

또한 성능 테스트 결과는 원본 로그만으로는 의미가 부족하며, 테스트 조건과 해석을 함께 정리해야 보고서로 사용할 수 있다.

이번 실습을 통해 모바일 게임 성능 테스트의 기본 흐름을 경험했고, 이후에는 배치 파일을 이용해 일정 간격으로 성능 데이터를 자동 수집하는 방법을 추가로 학습할 필요가 있다.

## 15. 핵심 정리

```text
adb devices
→ 기기 연결 확인

adb shell dumpsys cpuinfo 패키지명
→ CPU 사용량 확인

adb shell dumpsys gfxinfo 패키지명
→ 프레임 / 렌더링 / Jank 확인

adb shell dumpsys meminfo 패키지명
→ 메모리 사용량 확인

adb shell dumpsys battery | findstr /i "temperature"
→ 배터리 온도 확인

adb shell dumpsys batterystats --charged 패키지명
→ 배터리 사용 통계 확인

adb shell ps | findstr 패키지명
→ PID 확인

adb logcat --pid=PID값 *:E
→ 특정 게임의 Error 로그 확인
```

이번 수업의 핵심은 ADB 명령어를 외우는 것이 아니라, 수집한 데이터를 보고서 형태로 해석하고 정리하는 것이다.
