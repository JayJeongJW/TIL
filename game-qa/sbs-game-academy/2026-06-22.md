# 2026-06-22 QA 수업 정리 - 동등분할과 경계값 분석을 활용한 테스트 시나리오 설계

## 1. 오늘 수업 핵심

오늘 수업에서는 테스트케이스를 설계할 때 사용하는 대표적인 기법인 **동등분할**과 **경계값 분석**을 학습했다.

이번 수업에서 중요한 점은 단순히 요구사항에 적힌 조건만 보고 테스트케이스를 만드는 것이 아니라, **어떤 조건은 동등분할로 묶고, 어떤 조건은 경계값을 기준으로 더 촘촘하게 확인해야 하는지 판단하는 것**이었다.

핵심은 다음과 같다.

```text
동일한 결과가 도출될 것으로 예상되는 입력값들을 그룹핑한 후,
해당 그룹에서 경계에 위치한 값을 대표값으로 선정해 테스트를 수행하고,
그 결과가 나머지 입력값들을 대변한다고 보는 방식이다.
```

즉, 테스트를 무작정 많이 만드는 것이 아니라, **같은 결과가 예상되는 값은 묶고, 결과가 바뀌는 경계값을 대표로 확인하는 것**이 중요하다.

---

## 2. 동등분할이란?

**동등분할**은 같은 결과가 나올 것으로 예상되는 입력값이나 조건을 하나의 그룹으로 묶는 테스트 설계 기법이다.

예를 들어 ID 길이 조건이 다음과 같다고 가정한다.

```text
ID는 4글자 이상 10글자 이하로만 생성 가능하다.
```

이때 모든 글자 수를 하나하나 테스트하지 않는다.

```text
1글자
2글자
3글자
4글자
5글자
...
10글자
11글자
```

이렇게 전부 테스트하는 것이 아니라, 결과가 같은 그룹으로 나눈다.

| 구분    | 조건             | 결과    |
| ----- | -------------- | ----- |
| 유효 그룹 | 4글자 이상 10글자 이하 | 생성 가능 |
| 무효 그룹 | 4글자 미만         | 생성 불가 |
| 무효 그룹 | 10글자 초과        | 생성 불가 |

동등분할은 모든 입력값을 테스트하는 것이 아니라, **같은 결과가 예상되는 입력값들을 대표 그룹으로 묶어 테스트하는 방식**이다.

---

## 3. 경계값 분석이란?

**경계값 분석**은 유효와 무효가 바뀌는 경계 근처의 값을 집중적으로 확인하는 테스트 설계 기법이다.

동등분할이 그룹을 나누는 방식이라면, 경계값 분석은 그 그룹이 바뀌는 **기준점 근처**를 확인하는 방식이다.

```text
동등분할:
유효 그룹 / 무효 그룹을 나눈다.

경계값 분석:
유효와 무효가 바뀌는 기준점 바로 앞, 기준점, 기준점 바로 뒤를 확인한다.
```

버그는 완전히 멀리 떨어진 값보다, 조건이 바뀌는 경계에서 자주 발생할 수 있다.

따라서 조건의 경계가 명확한 컨텐츠는 경계값 분석을 적용하는 것이 좋다.

---

## 4. 동등분할과 경계값 분석의 차이

| 구분     | 목적                       | 예시                 |
| ------ | ------------------------ | ------------------ |
| 동등분할   | 같은 결과가 예상되는 입력값을 그룹으로 묶음 | 4~10글자 유효 / 그 외 무효 |
| 경계값 분석 | 유효와 무효가 바뀌는 경계값을 확인      | 3, 4, 10, 11       |

정리하면 다음과 같다.

```text
동등분할로 그룹을 나누고,
경계값 분석으로 그 그룹의 경계에 위치한 값을 확인한다.
```

---

## 5. 어떤 경우에 경계값 분석을 적용하면 좋은가?

수업에서 나온 핵심은 다음과 같다.

```text
시간 베이스,
PVP 등수별 보상처럼
경계가 명확한 컨텐츠들은 경계값을 분석하는 것이 좋다.
```

반대로 조건이 명확하게 숫자나 시간으로 나뉘지 않는 경우에는 동등분할을 적용하는 것이 좋다.

```text
모호한 것들은 동등분할 적용
컨텐츠에 따라 어떤 것을 적용할지 결정
```

즉, 무조건 하나의 기법만 사용하는 것이 아니라, 컨텐츠 성격에 따라 판단해야 한다.

| 컨텐츠 성격                          | 적합한 방식        |
| ------------------------------- | ------------- |
| 등수, 시간, 날짜, 수치처럼 경계가 명확한 경우     | 경계값 분석        |
| 상태, 유형, 조건처럼 그룹으로 나누는 것이 중요한 경우 | 동등분할          |
| 조건 그룹과 경계가 모두 중요한 경우            | 동등분할 + 경계값 분석 |

## 6. 회원가입 ID 예제 - 동등분할

수업에서 다룬 회원가입 ID 요구사항은 다음과 같았다.

```text
1. ID는 4글자 이상 10글자 이하로만 생성 가능하다.
2. ID는 영문과 숫자의 조합으로만 생성 가능하다.
3. ID에는 욕설이나 음란어에 해당하는 단어 포함이 불가능하다.
```

이 요구사항은 처음 보면 경우의 수가 많아 보인다.

```text
4글자 미만
10글자 초과
영문만 입력
숫자만 입력
특수문자 포함
한글 포함
욕설 포함
음란어 포함
무입력
```

하지만 동등분할 관점에서는 핵심 시나리오를 5개 그룹으로 나눌 수 있다.

| 번호 | 조건                        | 기대 결과                    |
| -- | ------------------------- | ------------------------ |
| 1  | 길이 정상 / 영문+숫자 조합 / 금칙어 없음 | 가입 가능                    |
| 2  | 길이 조건 불만족                 | 가입 불가, 길이 관련 안내 메시지 노출   |
| 3  | 영문+숫자 조합 조건 불만족           | 가입 불가, 조합 관련 안내 메시지 노출   |
| 4  | 욕설 또는 음란어 등 금칙어 포함        | 가입 불가, 부적절한 단어 안내 메시지 노출 |
| 5  | null 또는 무입력               | 가입 불가, 입력 필요 안내 메시지 노출   |

같은 안내 메시지로 처리되는 조건은 하나의 무효 그룹으로 묶을 수 있다.

예를 들어 다음 조건들은 세부 입력값은 다르지만 같은 요구사항을 위반한다.

```text
영문만 입력
숫자만 입력
특수문자 포함
한글 포함
```

이들은 모두 다음 조건을 만족하지 못한다.

```text
ID는 영문과 숫자의 조합으로만 생성 가능하다.
```

따라서 동등분할 기준에서는 하나의 시나리오 그룹으로 볼 수 있다.

## 7. 여러 조건을 동시에 위반하는 경우

여러 조건을 동시에 위반하는 경우도 고려해야 한다.

예를 들어 어떤 ID가 다음 조건을 모두 위반한다고 가정할 수 있다.

```text
글자 수 조건 불만족
영문+숫자 조합 조건 불만족
금칙어 포함
```

이 경우 어떤 메시지를 먼저 노출해야 하는지가 문제가 된다.

```text
글자 수 오류 메시지?
영문+숫자 조합 오류 메시지?
부적절한 단어 포함 메시지?
```

이 부분은 QA가 임의로 정할 수 없다.

여러 조건을 동시에 위반했을 때 어떤 피드백을 먼저 노출할지는 **기획에서 정의해야 하는 정책**이다.

QA는 다음과 같이 정리해야 한다.

```text
여러 조건을 동시에 위반했을 때 노출할 피드백 우선순위가 명시되어 있지 않음
→ 기획 확인 필요 사항으로 전달
```

조건을 여러 개 위반했다고 해서 팝업이 여러 개 동시에 뜨는 식으로 처리하는 것이 일반적이지는 않다.
보통은 우선순위가 정해진 하나의 메시지를 먼저 보여준다.

## 8. 회원가입 ID 예제 - 경계값 분석

이전 문제를 경계값으로 보면 다음과 같이 정리할 수 있다.

```text
ID 네 글자 이상 열 글자 이하 생성
→ 0, 3, 4, 10, 11
```

각 값의 의미는 다음과 같다.

| 입력값  | 기대 결과 | 이유            |
| ---- | ----- | ------------- |
| 0글자  | 무효    | null 또는 무입력   |
| 3글자  | 무효    | 최소 허용값 4보다 작음 |
| 4글자  | 유효    | 최소 허용값        |
| 10글자 | 유효    | 최대 허용값        |
| 11글자 | 무효    | 최대 허용값 초과     |

경계값 분석에서는 단순히 실패하는 값만 보는 것이 아니라, **정확히 허용되는 경계값도 확인해야 한다.**

즉, 3글자와 11글자만 확인하면 부족하다.
4글자와 10글자가 정상적으로 허용되는지도 확인해야 한다.

## 9. PVP 시즌 보상 - 경계값 분석

수업에서 다룬 PVP 시즌 보상 구간은 다음과 같았다.

```text
1등
2등
3등
4-5등
6-10등
11-20등
21-30등
31-50등
51-70등
71-100등
101-300등
301-500등
501-1000등
1001-
```

PVP 시즌 보상은 등수별 보상 구간이 명확하게 나뉘어 있다.
이런 컨텐츠는 경계값 분석을 적용하기 좋다.

경계값 분석을 적용하면 각 보상 구간의 **최소값과 최대값**을 확인해야 한다.

---

## 10. PVP 시즌 보상 경계값

각 구간에서 확인해야 할 대표값은 다음과 같다.

| 보상 구간     | 확인할 경계값   |
| --------- | --------- |
| 1등        | 1         |
| 2등        | 2         |
| 3등        | 3         |
| 4-5등      | 4, 5      |
| 6-10등     | 6, 10     |
| 11-20등    | 11, 20    |
| 21-30등    | 21, 30    |
| 31-50등    | 31, 50    |
| 51-70등    | 51, 70    |
| 71-100등   | 71, 100   |
| 101-300등  | 101, 300  |
| 301-500등  | 301, 500  |
| 501-1000등 | 501, 1000 |
| 1001등 이상  | 1001      |

총 테스트 대표값은 다음과 같다.

```text
1
2
3
4, 5
6, 10
11, 20
21, 30
31, 50
51, 70
71, 100
101, 300
301, 500
501, 1000
1001
```

개수로 세면 총 24개다.

```text
1등, 2등, 3등 = 3개
4-5등부터 501-1000등까지 각 구간 최소/최대 = 20개
1001등 이상 구간의 최소값 = 1개

총 24개
```

따라서 PVP 시즌 보상을 경계값 분석으로 테스트하면 **24개의 대표값**으로 테스트 커버리지를 구성할 수 있다.

## 11. 핫타임 이벤트 - 경계값 분석

핫타임 이벤트 요구사항은 다음과 같았다.

```text
6월 25일부터 6월 30일까지
매일 오후 2시부터 오후 4시까지
경험치 2배 이벤트 진행
```

이 요구사항은 날짜와 시간이 명확하게 정해져 있기 때문에 경계값 분석을 적용하기 좋다.

확인해야 할 경계는 크게 두 가지다.

```text
이벤트 날짜 경계
이벤트 시간 경계
```

## 13. 핫타임 이벤트 시간 경계

```text
오후 4시 1초 이상
```

## 15. 핫타임 이벤트 테스트 개수

핫타임 이벤트의 경계값 테스트 개수는 다음과 같이 계산할 수 있다.

```text
이벤트 기간 외 날짜:
6월 24일 오후 2시
7월 1일 오후 2시
→ 2개

이벤트 기간:
6월 25일 ~ 6월 30일 = 6일

각 날짜별 시간 경계:
오후 1시 59분 59초 이하
오후 2시
오후 4시
오후 4시 1초 이상
→ 하루 4개

6일 × 4개 = 24개

총합:
24개 + 2개 = 26개
```

따라서 핫타임 이벤트를 경계값 분석으로 테스트하면 **총 26개**의 테스트 시나리오가 나온다.

## 16. 이벤트 쿠폰 지급 응용 문제

응용 문제는 다음과 같았다.

```text
회원가입을 완료한지 30일 이상이 경과 되었으면서
AND
현금으로 물품을 구매한 금액이 10만원 초과를 만족하는 모든 사용자에게
이벤트 기간 종료 후 1만원 할인 쿠폰 일괄 지급
```

이 문제에서 중요한 조건 축은 4가지다.

```text
1. 회원
2. 가입기간
3. 결제수단
4. 결제금액
```

이 조건들을 기준으로 동등분할을 해야 한다.

---

## 17. 조건 1 - 회원

회원이라고 해서 모두 같은 상태는 아니다.

```text
일반회원 / 정상회원
비회원 / 회원가입하지 않은 사용자
휴면회원
휴면 후 복귀 회원
탈퇴 회원
탈퇴 유예 회원
미성년자 회원
외국인 회원
```

각 회원 상태는 쿠폰 지급 대상 여부에 영향을 줄 수 있다.

예를 들어 탈퇴 회원은 사용자 DB에서 바로 즉시 삭제되지 않을 수 있다.
탈퇴 신청 후 일정 기간 동안 기록을 삭제하거나 복구할 수 있는 탈퇴 유예 상태가 존재할 수도 있다.

따라서 다음을 확인해야 한다.

```text
탈퇴 회원도 지급 대상인가?
탈퇴 유예 회원은 정상 회원으로 보는가?
휴면 회원도 지급 대상인가?
휴면 후 복귀 회원의 가입기간은 어떻게 계산하는가?
미성년자 회원도 이벤트 대상인가?
외국인 회원도 이벤트 대상인가?
```

---

## 18. 조건 2 - 가입기간

가입기간도 단순히 30일 이상 / 미만으로만 끝나지 않는다.

```text
이벤트 기간 내 30일
이벤트 시작 전 10일 + 이벤트 중 20일
휴면 회원 전환 되고 나서 10일
탈퇴 이후 재가입
```

특히 탈퇴 이후 재가입한 경우는 애매하다.

```text
유저 DB가 남아있는 상태에서 재가입한 경우
→ 신규 회원인지 기존 회원인지 개발 환경에 따라 다를 수 있음
```

이런 내용은 QA가 임의로 판단할 수 없다.
기획 및 개발 확인이 필요하다.

---

## 19. 조건 3 - 결제수단

요구사항에는 “현금으로 물품을 구매한 금액”이라고 되어 있다.

여기서 중요한 질문은 다음과 같다.

```text
무엇을 현금으로 볼 것인가?
```

현금으로 볼 수 있는 후보는 다음과 같다.

```text
계좌이체
상품권
포인트
휴대폰 결제
오픈 결제 / 간편결제
네이버페이
삼성페이
애플페이
```

예를 들어 네이버포인트처럼 1포인트가 현금 가치 1원에 해당하는 경우도 있다.
이런 포인트를 현금으로 볼 것인지 아닌지는 기획에서 정해야 한다.

현금이 아닌 수단의 예시는 다음과 같다.

```text
신용카드
```

다만 실제 서비스 정책에 따라 신용카드를 어떻게 분류할지도 확인이 필요하다.

QA는 결제수단을 임의로 판단하는 것이 아니라, 리스크를 드러내야 한다.

```text
현금으로 인정되는 결제수단의 범위가 명확하지 않음
→ 기획 확인 필요
```

## 20. 조건 4 - 결제금액

결제금액도 단순히 10만원 초과 / 이하만 보면 부족하다.

```text
단일 결제 10만원?
누적 결제 10만원?
10만 2천원 결제 품목 → 3천원 할인쿠폰 사용 → 9만 9천원
9만 9천원 품목 + 배송비 5천원 → 10만 4천원
쿠폰 받고 환불하는 경우
```

예를 들어 다음 케이스가 있다.

```text
10만 2천원 상품 구매
3천원 할인쿠폰 사용
실결제금액 9만 9천원
```

이 경우 상품 금액 기준이면 10만원 초과지만, 실결제금액 기준이면 10만원 이하이다.

또 다른 경우도 있다.

```text
9만 9천원 상품 구매
배송비 5천원
총 결제금액 10만 4천원
```

이 경우 배송비를 포함하면 10만원 초과지만, 상품 금액만 보면 10만원 이하이다.

따라서 다음 기준을 확인해야 한다.

```text
단일 결제 기준인가?
누적 결제 기준인가?
상품 금액 기준인가?
실결제금액 기준인가?
배송비 포함 기준인가?
할인쿠폰 적용 전 기준인가?
할인쿠폰 적용 후 기준인가?
쿠폰 지급 후 환불 시 회수하는가?
```

## 22. 기획 확인 필요 사항

이번 수업에서 중요한 점은, 테스트케이스를 만드는 것만큼 **기획 확인이 필요한 부분을 찾아내는 것**도 QA의 역할이라는 점이다.

### 회원 기준

```text
휴면회원도 지급 대상인가?
휴면 후 복귀 회원은 최초 가입일 기준인가, 복귀일 기준인가?
탈퇴 회원도 지급 대상인가?
탈퇴 유예 회원은 정상 회원으로 보는가?
미성년자 회원도 이벤트 대상인가?
외국인 회원도 이벤트 대상인가?
비회원 데이터가 지급 대상 조회에 포함될 수 있는가?
```

### 가입기간 기준

```text
30일 이상 기준은 이벤트 시작일 기준인가?
이벤트 종료일 기준인가?
쿠폰 지급 배치 실행일 기준인가?

휴면 기간도 가입기간에 포함되는가?
탈퇴 후 재가입 시 신규 회원인가, 기존 회원인가?
```


## 23. 오늘 배운 점

오늘 수업을 통해 테스트 시나리오 설계는 단순히 조건을 나열하는 일이 아니라는 것을 배웠다.

특히 동등분할과 경계값 분석은 각각 역할이 다르다.

```text
동등분할:
같은 결과가 예상되는 입력값을 그룹으로 묶는다.

경계값 분석:
그룹이 바뀌는 경계에 위치한 값을 확인한다.
```

또한 컨텐츠 성격에 따라 어떤 테스트 설계 기법을 적용할지 판단해야 한다.

```text
시간 베이스 컨텐츠
PVP 등수별 보상
날짜/시간 이벤트
수치 기준 보상
→ 경계값 분석 적용이 좋음

회원 상태
결제수단
회원 유형
정책이 필요한 조건
→ 동등분할 적용이 좋음
```

오늘 수업에서 가장 크게 느낀 점은 다음과 같다.

```text
QA는 요구사항을 그대로 받아 적는 사람이 아니라,
요구사항을 테스트 가능한 조건으로 나누고,
어떤 부분이 명확하고 어떤 부분이 기획 확인이 필요한지 찾아내는 사람이다.
```

## 24. 핵심 정리

```text
동등분할은 같은 결과가 예상되는 입력값을 하나의 그룹으로 묶는 기법이다.

경계값 분석은 유효와 무효가 바뀌는 기준점 근처의 값을 테스트하는 기법이다.

경계가 명확한 컨텐츠는 경계값 분석을 적용하기 좋다.

모호한 조건이나 정책 판단이 필요한 컨텐츠는 동등분할을 적용하기 좋다.

PVP 시즌 보상처럼 등수 구간이 명확한 경우 각 구간의 최소값과 최대값을 확인한다.

핫타임 이벤트처럼 날짜와 시간이 정해진 경우 시작 전, 시작 시점, 종료 시점, 종료 후를 확인한다.

이벤트 쿠폰 지급처럼 회원 상태, 가입기간, 결제수단, 결제금액이 얽힌 문제는 조건 축을 나눈 뒤 동등분할로 테스트 시나리오를 설계한다.

요구사항만으로 기대 결과를 확정할 수 없는 경우 QA가 임의로 판단하지 않고 기획 확인 필요 사항으로 정리한다.
```

오늘 수업의 핵심을 한 문장으로 정리하면 다음과 같다.

> QA는 모든 경우의 수를 무작정 테스트하는 사람이 아니라,
> 조건을 그룹으로 나누고, 경계값을 찾아내며, 모호한 정책을 기획 확인 사항으로 정리하는 사람이다.
