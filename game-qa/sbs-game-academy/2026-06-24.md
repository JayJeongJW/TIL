# 2026-06-24 QA 수업 정리 - 동등분할, 경계값 분석, 페어와이즈 테스트

## 1. 오늘 수업 핵심

오늘 수업에서는 하나의 요구사항 안에서 **동등분할**과 **경계값 분석**을 함께 사용하는 방식과, 여러 입력값의 조합이 많아질 때 사용하는 **페어와이즈 테스트 기법**을 학습했다.

핵심은 다음과 같다.

```text
그룹 간 경계가 명확한 조건은 경계값 분석을 사용한다.
그룹 간 경계가 모호하거나 상태/유형으로 나뉘는 조건은 동등분할을 사용한다.
입력값 조합이 많아지는 경우에는 페어와이즈를 사용해 테스트 케이스 수를 줄인다.
```

# 2. 이벤트 쿠폰 지급 문제

## 문제

```text
12월 1일부터 31일까지 한 달간 진행되는 이벤트

매일 오후 6시부터 8시까지 선착순 1000명에게 할인 쿠폰 지급 이벤트

이벤트 참여 방식은 이벤트 페이지 접속 후 쿠폰 발급 버튼을 클릭해서 발급받는 방식

쿠폰 발급은 주 1회만 가능
주마다 발급 기록은 초기화
즉 쿠폰 발급 가능 최대 수량은 4장

쿠폰은 멤버십 등급에 따라 할인쿠폰 금액에 차이 있음

일반 멤버는 1천원 할인 쿠폰
실버 멤버는 3천원 할인 쿠폰
골드 멤버는 5천원 할인 쿠폰
플래티넘 멤버는 1만원 할인 쿠폰

동일 상품에 쿠폰을 중복 적용은 불가능

쿠폰은 발급일 기준으로 6개월의 유효기간 존재
```

요구사항에서 유추할 수 있는 많은 정보를 고려해 테스트 시나리오를 구성해야 한다.

---

# 3. 경계값 분석 적용 항목

경계값 분석은 날짜, 시간, 수량, 횟수, 유효기간처럼 **기준이 명확한 값**에 적용한다.

---

## 3-1. 키워드 1 - 날짜 / 시간

### 조건

```text
12월 1일부터 12월 31일까지
매일 오후 6시부터 오후 8시까지
```

### 테스트 값

```text
11월 30일 오후 6시
1월 1일 오후 6시

12월 1일 ~ 12월 31일
- 오후 5시 59분 59초
- 오후 6시
- 오후 8시
- 오후 8시 1초
```

### 테스트 개수

```text
12월 1일 ~ 12월 31일 = 31일

하루 기준 테스트 시점 4개
31일 × 4개 = 124개

이벤트 기간 밖 테스트 2개
11월 30일 오후 6시
1월 1일 오후 6시

총 126개
```

### 예외 케이스

```text
예외 케이스 1.
한국 시간 기준으로 적용할지, 타국 지역 시간도 포함되는지 확인

예외 케이스 2.
오전 시간대에서도 쿠폰이 잘못 발급되지 않는지 확인

예외 케이스 3.
오후 8시까지 선착순 1000명이 도달하지 않았을 때
남은 수량이 다음 날로 이월되는지, 삭제되는지 확인
```

날짜와 시간은 단순히 시작/종료 시각만 확인하는 것이 아니라, **시간대 기준, 오전 오발급, 잔여 수량 처리 정책**까지 확인해야 한다.

# 5. 테스트 시나리오 설계 태도

수업에서 강조된 내용은 다음과 같다.

```text
예상 가능한 시나리오는 테스트할 확률이 1%라도 있으면 고려해야 한다.
```

하지만 모든 테스트를 매번 수행해야 하는 것은 아니다.

```text
그렇다고 매번 테스트를 할 필요가 있나?
→ 그것은 아님
```

테스트는 리스크와 중요도에 따라 주기를 정해야 한다.

```text
보잘것없어 보이는 테스트라도 찜찜하면
한 달에 한두 번
두 달에 한두 번
최소 분기별 한 번씩 확인할 수 있음
```

반대로 다음과 같은 태도는 위험하다.

```text
어제 했으니 오늘은 패스
한 달 빡세게 했으니 당분간 이쪽 테스트는 안 해도 될 것 같아
```

이런 식으로 타협하면 안 된다.

QA는 가능성을 무시하는 것이 아니라, **될지 안 될지 확인하는 역할**을 해야 한다.

# 7. 페어와이즈 테스트

## 7-1. 페어와이즈란?

페어와이즈는 여러 입력값을 조합해서 테스트해야 하는 경우에 활용하는 테스트 기법이다.

모든 조합을 전부 테스트하면 테스트 케이스 수가 너무 많아질 수 있다.
이때 모든 조합을 다 확인하는 대신, **입력값 2개씩의 조합이 최소 한 번씩 등장하도록 테스트 케이스를 구성**한다.

```text
페어와이즈 = 입력값 조합 폭발을 줄이기 위한 테스트 기법
```

---

## 7-2. 페어와이즈가 필요한 예시

### PC 게임 그래픽 옵션

PC 게임은 그래픽 옵션 조합이 매우 많다.

```text
모니터가 지원하는 해상도
안티앨리어싱 적용 여부
안티앨리어싱 배율 2 / 4 / 6 / 8 / 16
프레임 설정
FSR
LS1
품질 / 성능 / 울트라 / 균형 옵션
블러 사용 여부
블러 값
```

옵션별로 모두 커버하려면 테스트 가지수가 매우 많아진다.
이런 경우 페어와이즈가 유용하다.

### 밸런스 테스트

밸런스 테스트에서도 페어와이즈를 활용할 수 있다.

```text
장비 세팅
스킬 세팅
전투력에 미치는 영향 요소
신규 캐릭터
카운터 신규 조합
```

예를 들어 PVP 밸런스 테스트에서 직업, 레벨, 스킬, 장비, 버프, 진형이 모두 조합되면 전체 테스트 케이스 수가 매우 커진다.

# 8. 페어와이즈의 단점과 샘플링

페어와이즈도 입력값이 많아지면 테스트 케이스가 많이 나온다.
따라서 적절한 샘플링이 중요하다.

수업에서는 Allpairs 결과의 `pairings` 값을 기준으로 샘플링을 설명했다.

```text
우측 pairings 항목의 스코어가 큰 수치는 유의미한 테스트
스코어가 낮으면 유의미함이 떨어짐
```

Allpairs 결과에서 `~` 물결표가 붙는 경우가 있다.

```text
각 입력값 앞에 물결 표시
```

이 물결표는 쌍을 맞추기 위해 들어간 중복 테스트, 즉 구색 맞추기 성격으로 볼 수 있다.

```text
27번까지는 1:1로 진행
28번부터는 다시 테스트를 해야 함
쌍으로 조합이 되어야 하기 때문
쌍을 맞추기 위해 물결표가 붙음
물결표가 많아질수록 테스트의 유의미함이 떨어짐
```

따라서 페어와이즈 결과를 전부 그대로 쓰는 것이 아니라, pairings 점수와 급감 구간을 보고 샘플링할 수 있다.

```text
입력값이 많으면 샘플링을 한다.
큰 폭으로 급격하게 급감하는 구간을 기준으로 1차 샘플링을 할 수 있다.

예:
64 → 28
큰 폭으로 줄어드는 구간을 1차 샘플링 기준으로 잡음
```

2차 샘플링도 가능하다.

```text
첫 번째 pairings 급감 수치보다
그다음 급감 수치에 해당하는 구간까지 필터링

도출된 테스트 케이스 숫자가 수행 가능한 수준이면 테스트 진행
```

```text
직업
```

```text
직업    레벨    스킬    장비    버프        진형

```

이 입력값은 PVP 컨텐츠 대상의 테스트 조합을 만들기 위한 값이다.

```text
진형 = 배치 위치
```

# 12. Allpairs 출력 결과 해석

Allpairs를 실행하면 다음과 같은 형태의 테스트 케이스가 생성된다.

```text
case    직업    레벨    스킬    장비    버프    진형    pairings
1       검사    55      패시브  일반    무료아이템  중열    15
2       검사    35      2번     전설    펫        전열    15
3       검사    45      3번     고급    유료아이템  후열    15
...
```

출력 결과에서 중요한 항목은 다음과 같다.

```text
case = 테스트 케이스 번호
직업/레벨/스킬/장비/버프/진형 = 조합된 테스트 입력값
pairings = 해당 테스트 케이스가 커버하는 조합 점수
```

pairings 값이 높을수록 더 많은 유의미한 조합을 포함하고 있다고 볼 수 있다.

# 13. 물결표의 의미

출력 결과 후반부에는 값 앞에 `~`가 붙는 경우가 있다.

예시:

```text
28    힐러    75    3번    에픽    길드    ~전열    8
40    전사    35    ~4번   에픽    펫      ~후열    2
43    ~도적   ~55   ~1번   희귀    팀조합  ~전열    1
```

수업에서는 이 물결표를 다음처럼 설명했다.

```text
쌍을 맞추기 위해 다시 들어간 값
중복 테스트
구색 맞추기
```

즉, 물결표가 많아질수록 그 테스트 케이스의 유의미함은 낮아질 수 있다.

따라서 테스트 케이스를 전부 수행하기 어렵다면, pairings 값과 물결표 여부를 보고 우선순위를 정할 수 있다.

# 14. 오늘 배운 점

오늘 수업에서 가장 중요한 점은 테스트 설계 기법을 하나만 고정적으로 쓰는 것이 아니라, 조건의 성격에 따라 다르게 적용해야 한다는 것이다.

```text
경계가 명확한 조건
→ 경계값 분석

유형이나 상태로 나뉘는 조건
→ 동등분할

입력값 조합이 많아지는 조건
→ 페어와이즈
```

또한 테스트는 단순히 가능한 모든 조합을 나열하는 것이 아니다.

```text
예상 가능한 시나리오는 고려해야 한다.
하지만 모든 테스트를 매번 수행할 수는 없다.
따라서 리스크와 우선순위에 따라 테스트 주기를 정해야 한다.
```

QA는 “어제 했으니 오늘은 패스”처럼 감으로 타협하는 것이 아니라,
왜 테스트를 해야 하는지, 왜 이번에는 우선순위를 낮춰도 되는지 기준을 가져야 한다.

# 15. 핵심 정리

```text
동등분할은 모호하거나 상태/유형으로 나뉘는 조건을 그룹화하는 기법이다.

경계값 분석은 날짜, 시간, 수량, 횟수, 유효기간처럼 기준이 명확한 조건에 적용한다.

페어와이즈는 입력값 조합이 너무 많을 때 2개씩의 조합을 커버하도록 테스트 케이스를 줄이는 기법이다.

Allpairs는 페어와이즈 테스트 케이스를 생성할 수 있는 도구이다.

Allpairs 입력 파일은 탭으로 열을 구분하고, 엔터로 행을 구분한다.

Allpairs 출력 결과의 pairings 값은 테스트 케이스의 유의미함을 판단하는 기준으로 활용할 수 있다.

출력 결과의 물결표는 쌍을 맞추기 위해 들어간 중복/구색 맞추기 성격의 값으로 볼 수 있다.

입력값이 많아질수록 테스트 케이스도 늘어나므로, pairings 급감 구간과 물결표 여부를 기준으로 샘플링할 수 있다.

QA는 테스트 가능성이 낮아 보이는 시나리오도 고려해야 하지만, 리스크와 주기에 따라 수행 기준을 세워야 한다.
```
