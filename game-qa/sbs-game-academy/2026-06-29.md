# 2026-06-30 QA 수업 정리 - Jira 결함 관리와 테스트 케이스 포트폴리오 구성

## 1. 오늘 수업 핵심

오늘 수업에서는 Jira를 활용한 결함 관리 방식과, QA 포트폴리오용 테스트 케이스 문서 구성 방법을 학습했다.

핵심은 다음과 같다.

```text
1. 버그는 생성부터 종료까지 상태값을 통해 수명 주기를 관리한다.
2. Jira 이슈는 요약, 설명, 환경, 재현 스텝, 기대 결과, 실제 결과가 명확해야 한다.
3. 우선순위와 심각도는 서로 다른 기준으로 판단해야 한다.
4. 테스트 케이스 포트폴리오는 표지, 대시보드, 테스트 케이스 시트로 구성한다.
5. 테스트 케이스는 제3자가 봐도 수행 가능한 수준으로 명확하게 작성해야 한다.
6. AI는 테스트 케이스 초안 작성에 도움을 줄 수 있지만, 그대로 제출하면 안 된다.
```

# 2. Jira 스페이스 구성

Jira 스페이스를 만든 뒤에는 결함 관리에 필요한 상태값, 이슈 필드, 레이블, 첨부 파일 관리 방식을 정의해야 한다.

Jira는 단순히 버그를 적는 공간이 아니라, **버그가 어떤 상태에 있고, 누가 처리 중이며, 어느 환경까지 확인되었는지 관리하는 도구**다.

---

# 3. 버그의 수명 주기

## 상태값 정의

| 상태              | 의미                                      |
| --------------- | --------------------------------------- |
| To Do           | 신규 생성된 결함이며, 해당 결함을 처리해야 하는 담당자가 할당된 상태 |
| In Progress     | 작업 또는 디버깅이 진행 중인 상태                     |
| Dev Resolve     | 개발 환경에서 수정 사항 반영이 완료된 상태                |
| Staging Resolve | 스테이징 환경에서 수정 사항 반영이 완료된 상태              |
| Dev QA          | 개발 환경에서 수정된 사항이 정상적으로 확인된 상태            |
| Staging QA      | 스테이징 환경에서 수정된 사항이 정상적으로 확인된 상태          |
| Closed          | 라이브 환경에서 수정 사항이 정상 반영되어 종료된 상태          |

버그는 단순히 “수정됨”으로 끝나는 것이 아니라, 개발 환경과 스테이징 환경, 라이브 환경에서 각각 확인이 필요하다.

---

## Closed 처리 유형

Closed 상태 안에서도 결함이 어떤 이유로 종료되었는지 구분할 수 있다.

| 종료 유형      | 의미                                 |
| ---------- | ---------------------------------- |
| By Design  | 정상적인 기획 의도                         |
| Duplicated | 이미 등록된 중복 이슈                       |
| 수정하지 않음    | 수정해서 얻는 이득보다 수정하지 않는 쪽의 이득이 더 큰 경우 |
| 정보 부족      | 정확한 재발생 스텝이 파악되지 않아 수정이 어려운 경우     |

### 수정하지 않음 예시

```text
버그이지만 유저들이 해당 버그성 플레이를 좋아하는 경우
이를 막았을 때 유저 이탈 가능성이 더 큰 경우
```

또는 다음과 같은 경우도 있을 수 있다.

```text
수정이 필요하긴 하지만 안드로이드 운영체제 자체의 고질적인 문제인 경우
게임사 단독으로 해결하기 어려운 경우
```

### 정보 부족 예시

```text
버그가 발생했다는 제보는 있으나,
정확한 재현 경로가 확인되지 않아 개발자가 수정하기 어려운 경우
```

# 4. 수정 확인 후 상태 변경

각 환경에 접속해서 문제 수정 여부를 확인해야 한다.

만약 수정이 제대로 되지 않았다면, 상태값을 다시 `In Progress`로 변경하고 댓글을 남겨야 한다.

예시:

```text
수정 사항이 정상 반영되지 않았습니다.
동일한 재현 스텝으로 다시 확인했을 때 문제가 재발생합니다.
재확인 부탁드립니다.
```

QA는 단순히 이슈를 등록하는 것에서 끝나는 것이 아니라, 수정 후 재확인까지 진행해야 한다.

# 5. Jira 요약 항목 작성

요약 항목은 Jira 이슈에서 가장 먼저 보이는 제목 역할을 한다.

수업에서 강조된 내용은 다음과 같다.

```text
버그의 구체적인 정보는 들어가 있어야 한다.
하지만 문장은 컴팩트해야 한다.
요약글이 길면 안 된다.
어디서 어떤 문제가 발생했는지 담겨야 한다.
문장 어미는 "~하는 현상"을 사용한다.
```

## 좋은 요약의 조건

| 조건       | 설명                       |
| -------- | ------------------------ |
| 발생 위치 포함 | 어느 화면, 기능, 상황에서 발생했는지 포함 |
| 문제 현상 포함 | 어떤 비정상 동작이 발생했는지 포함      |
| 간결함      | 너무 길지 않게 작성              |
| 객관적 표현   | 감정적 표현이나 추측 표현 배제        |
| 현상 중심    | “문제”, “버그”보다 실제 현상을 표현   |

## 요약 작성 예시

```text
저장 슬롯 정보와 실제 로드 후 퀘스트 진행 상태가 일치하지 않는 현상
```

```text
특정 해상도에서 그래픽 옵션 변경 후 UI 버튼이 화면 밖으로 밀려나는 현상
```

```text
전투 중 회피 입력 시 캐릭터가 즉시 반응하지 않는 현상
```

## 주의할 표현

```text
이러한 문제
이러한 버그
이러한 결함
```

이런 표현은 구체성이 떨어질 수 있다.

다만 버그로 단정하기 어려운 경우에는 다음처럼 작성할 수 있다.

```text
버그로 의심될 수 있는 상황이며 담당자 확인이 필요함
```

# 6. Jira 설명 항목 구성

설명 항목은 이슈에 대한 상세 정보를 담는 필드다.

수업에서는 설명 항목을 6가지로 구성한다고 정리했다.

```text
1. 테스트 환경
2. 상세 현상
3. 재발생 스텝
4. 발생 빈도
5. 기대 결과
6. 실제 결과
```

---

## 6-1. 테스트 환경

테스트 환경에는 문제를 재현한 기본 정보를 작성한다.

포함할 수 있는 항목은 다음과 같다.

```text
PC
Mobile
Console
환경에 대한 기본 스펙
테스트 계정 정보
닉네임
UID
테스트 게임 버전
```

예시:

```text
플랫폼: PC
OS: Windows 11
GPU: RTX 3060
게임 버전: v1.0.3
테스트 계정: QA_Test_01
UID: 123456789
```

---

## 6-2. 상세 현상

상세 현상에는 실제로 어떤 문제가 발생했는지 구체적으로 작성한다.

요약 항목이 짧은 제목이라면, 상세 현상은 문제를 이해할 수 있도록 충분히 풀어 쓰는 부분이다.

예시:

```text
수동 저장 슬롯에는 퀘스트가 진행된 상태로 표시되지만,
해당 슬롯을 로드하면 실제 게임 내 퀘스트 진행 상태는 이전 단계로 복원되는 현상이 발생한다.
이로 인해 유저가 저장 시점과 실제 복원 시점을 잘못 판단할 수 있다.
```

---

## 6-3. 재발생 스텝

재발생 스텝은 개발자나 담당자가 그대로 따라 했을 때 문제를 다시 확인할 수 있도록 작성해야 한다.

수업에서 강조된 내용은 다음과 같다.

```text
글 읽기 귀찮아하는 사람도 재발생 스텝만 따라 할 수 있어야 한다.
상세하면 상세할수록 좋다.
6~7단계 이상으로 정리할 수 있으면 좋다.
```

예시:

```text
1. 게임 실행 후 테스트 계정으로 로그인한다.
2. 메인 화면에서 퀘스트 메뉴로 진입한다.
3. 특정 퀘스트를 수락한다.
4. 퀘스트 목표를 1회 이상 진행한다.
5. 수동 저장 슬롯에 현재 진행 상태를 저장한다.
6. 게임을 종료한 뒤 다시 접속한다.
7. 저장한 슬롯을 로드한다.
8. 로드 후 퀘스트 진행 상태를 확인한다.
```

---

## 6-4. 발생 빈도

발생 빈도는 문제가 얼마나 자주 발생하는지 작성하는 항목이다.

| 발생 빈도   | 의미                             |
| ------- | ------------------------------ |
| 100% 발생 | 동일 조건에서 항상 발생                  |
| 랜덤 발생   | 1%~99% 확률로 발생                  |
| 1회 발생   | 어떻게든 재현하려고 했지만 이후에는 재현되지 않은 경우 |

랜덤 발생인 경우에는 반드시 시도 횟수와 재현 횟수를 적어야 한다.

예시:

```text
10회 시도 중 3회 발생
발생 빈도: 약 30%
```

1회 발생의 경우에는 단정하지 않고 추가 확인이 필요하다는 식으로 정리해야 한다.

---

## 6-5. 기대 결과

기대 결과는 정상적으로 동작했을 때의 결과를 작성한다.

예시:

```text
수동 저장 슬롯에 표시된 퀘스트 진행 상태와 실제 로드 후 퀘스트 진행 상태가 일치해야 한다.
```

---

## 6-6. 실제 결과

실제 결과는 테스트 중 확인된 비정상 결과를 작성한다.

예시:

```text
수동 저장 슬롯에는 퀘스트 진행 상태가 표시되지만,
로드 후 실제 퀘스트 진행 상태는 이전 단계로 복원된다.
```

# 7. 우선순위와 심각도

Jira 이슈에서는 우선순위와 심각도를 구분해야 한다.

## 우선순위

우선순위는 **비즈니스 관점의 평가**다.
즉, 유저에게 미치는 영향도와 얼마나 먼저 수정해야 하는지를 기준으로 판단한다.

수업에서 언급된 우선순위 값은 다음과 같다.

```text
아주 높음
높음
보통
낮음
아주 낮음
```

## 심각도

심각도는 **문제 자체의 심각성과 발생 빈도**를 기준으로 판단한다.

일반적으로 사용하는 심각도 값은 다음과 같다.

```text
Blocker
Major
Minor
Trivial
```

---

## 우선순위와 심각도가 다를 수 있는 예시

예시:

```text
모바일 게임 실행 시 스플래시 이미지에서 회사 CI가 잘못 노출되고 있는 현상
```

이 경우 기능적으로 게임 플레이가 막히는 것은 아니므로 심각도는 낮게 볼 수 있다.

```text
심각도: Trivial
```

하지만 회사 CI가 잘못 노출되는 문제는 브랜드 이미지와 외부 노출 측면에서 중요할 수 있다.

```text
우선순위: 높음 또는 아주 높음
```

즉, 문제 자체는 작아 보여도 비즈니스 영향이 크면 우선순위는 높아질 수 있다.

# 8. 레이블

레이블은 테스트 케이스와 Jira 이슈를 연결하는 데 중요하다.

수업에서 강조된 내용은 다음과 같다.

```text
테스트 케이스 1뎁스 명칭과 레이블 명칭은 동일해야 한다.
```

예시:

```text
TC 1뎁스: 저장/로드
Jira 레이블: 저장/로드
```

또는:

```text
TC 1뎁스: 전투
Jira 레이블: 전투
```

레이블이 통일되어야 나중에 이슈를 필터링하거나 대시보드로 집계하기 쉽다.

# 9. 첨부 파일 관리

Jira 이슈에는 스크린샷, GIF, 영상 파일, PDF 등을 첨부할 수 있다.

다만 멀티미디어 파일은 용량 문제 때문에 Google Drive에 올리고, 링크를 본문에 넣는 방식으로 관리할 수 있다.

```text
첨부 가능 자료:
- 스크린샷
- GIF
- 영상 파일
- PDF
- 로그 파일
```

영상이나 대용량 파일은 다음처럼 관리한다.

```text
1. Google Drive에 영상 업로드
2. 공유 링크 생성
3. Jira 본문 또는 댓글에 링크 삽입
```

# 11. 테스트 케이스 포트폴리오 구성

테스트 케이스 포트폴리오는 크게 3가지 시트로 구성한다.

```text
1. 표지 시트
2. 테스트 수행 결과 대시보드 시트
3. 테스트 케이스 시트
```

---

## 11-1. 표지 시트

표지 시트에는 단순 제목만 넣는 것이 아니라, 왜 해당 콘텐츠로 테스트 케이스를 설계했는지 작성해야 한다.

포함할 내용:

```text
왜 해당 콘텐츠를 TC 설계 대상으로 선택했는가?
TC 작성 시 어떤 부분을 유의했는가?
테스트 범위는 어디까지인가?
```

예시:

```text
본 테스트 케이스는 저장/로드 시스템의 상태 복원 안정성을 확인하기 위해 작성하였다.
저장 슬롯 정보, 퀘스트 진행 상태, NPC 상태, 보상 수령 여부가 서로 일치하는지 확인하는 데 중점을 두었다.
```

---

## 11-2. 테스트 수행 결과 대시보드 시트

대시보드는 테스트 결과를 숫자와 그래프로 보여주는 시트다.

수업에서는 대시보드를 간단하게 위쪽에 배치하는 방식도 가능하다고 했다.
실무에서는 별도 시트로 만들고, 테스트 케이스 시트 위쪽에도 요약을 같이 배치하는 경우가 있다.

대시보드에 들어갈 수 있는 항목:

```text
전체 TC 수
Pass 수
Fail 수
Blocked 수
N/A 수
Not Run 수
Pass 비율
Fail 비율
검증률
이슈 유형별 수
우선순위별 수
```

대시보드는 수동으로 계산하는 것이 아니라, 함수를 이용해 자동으로 집계되도록 구성해야 한다.

---

## 11-3. 테스트 케이스 시트

테스트 케이스 시트는 실제 TC를 작성하는 핵심 시트다.

수업에서 강조된 분량 기준은 다음과 같다.

```text
테스트 케이스 150개 ~ 200개 라인 사이
```

하나의 라인에는 한 가지 테스트 항목만 포함되어야 한다.

```text
하나의 라인 = 하나의 테스트 시나리오
```

화면을 보고 테스트 케이스를 설계하면 UI 확인 시나리오가 많아져 테스트 케이스가 단순해질 수 있다.

180개 라인을 기준으로 보면 비율은 다음 정도가 좋다.

```text
UI 검증 시나리오: 30%
기능 검증 시나리오: 70%
```

---

# 12. 긍정 케이스와 부정 케이스

테스트 케이스는 긍정 케이스와 부정 케이스를 함께 구성해야 한다.

수업 메모 기준으로는 다음 비율을 언급했다.

```text
긍정 케이스: 60~70% 정도
부정 케이스: 30~40% 정도
```

긍정 케이스는 정상 동작을 확인하는 시나리오다.

예시:

```text
정상 조건에서 쿠폰이 발급되는지 확인
```

부정 케이스는 비정상 조건, 예외 조건, 제한 조건을 확인하는 시나리오다.

예시:

```text
이미 이번 주에 쿠폰을 발급받은 상태에서 재발급을 시도할 경우 발급이 제한되는지 확인
```

---

# 13. 테스트 케이스 컬럼 구성

테스트 케이스 컬럼은 사람마다 다를 수 있고, 구글링하면 다양한 템플릿을 찾을 수 있다.

수업에서 강사님 스타일로 언급된 컬럼은 다음과 같다.

```text
TC ID
1뎁스
2뎁스
3뎁스
케이스 우선순위
사전 조건
테스트 데이터
테스트 수행 스텝
기대 결과
실제 결과
결과
BTS ID
수행자
Notes
```

예시 TC ID:

```text
CRIM-GUILD-001
```

TC ID는 게임명 또는 콘텐츠명을 기준으로 일관된 규칙을 가지는 것이 좋다.

---

# 14. 사전 조건 / 수행 스텝 / 기대 결과의 관계

수업에서는 테스트 케이스를 상태 전이 관점으로 설명했다.

```text
Before → Event → After
```

이를 테스트 케이스 컬럼에 연결하면 다음과 같다.

| 상태 전이  | 테스트 케이스 컬럼 |
| ------ | ---------- |
| Before | 사전 조건      |
| Event  | 테스트 수행 스텝  |
| After  | 기대 결과      |

예시:

```text
Before:
캐릭터가 전투 상태이며 회피 가능 스태미나를 보유하고 있음

Event:
공격 모션 중 Alt 키를 입력함

After:
기획 의도에 따라 회피가 즉시 발동되거나, 공격 모션 종료 후 발동되어야 함
```

---

# 15. 결과값과 BLOCKED 처리

결과 컬럼에는 Pass, Fail, Blocked, N/A, Not Run 등을 사용할 수 있다.

수업에서 언급된 BLOCKED는 다음과 같은 경우에 사용할 수 있다.

```text
테스트를 해야 하지만 라이브 환경이라 수행하기 어렵거나 까다로운 경우
```

이때는 단순히 BLOCKED만 적으면 안 되고, Notes에 사유를 작성해야 한다.

예시:

```text
결과: BLOCKED
Notes: 라이브 환경에서 유료 결제 테스트가 필요해 실제 수행 불가. 테스트 서버 또는 결제 샌드박스 환경 필요.
```

---

# 16. 테스트 시나리오 작성 기준

수업에서 강조된 테스트 시나리오 작성 기준은 다음과 같다.

```text
게임 화면에 보이는 모든 기능들을 최소 한 번씩은 확인할 수 있도록 구성한다.
화면에 보이지 않지만 연동되는 기능도 최대한 고민해 커버리지를 확보한다.
모든 테스트 시나리오는 제3자가 봐도 이해될 수 있도록 명확하고 구체적인 지시문 형태로 작성한다.
```

즉, 단순히 “버튼 확인”이라고 적는 것이 아니라, 어떤 상태에서 어떤 버튼을 눌렀을 때 어떤 결과가 나와야 하는지 작성해야 한다.

나쁜 예:

```text
상점 버튼 확인
```

좋은 예:

```text
메인 로비 화면에서 우측 하단 상점 버튼을 클릭했을 때 상점 화면으로 정상 진입하는지 확인
```

# 17. 엑셀 작성 기준

테스트 케이스 포트폴리오는 단순히 내용만 좋으면 되는 것이 아니라, 엑셀 문서 자체의 완성도도 중요하다.

수업에서 언급된 엑셀 작성 기준은 다음과 같다.

```text
결과 부분은 드롭다운 형태로 작성
조건부 서식 색깔 필수
대시보드는 함수로 자동 계산 및 집계
대시보드는 그래프와 숫자 중심으로 구성
엑셀의 기본 작업은 모든 라인을 흰색으로 처리한 상태에서 작업
```

모든 라인을 흰색으로 처리한다는 것은 기본 격자선이 보이지 않도록 하여 문서처럼 깔끔하게 보이게 만드는 것을 의미한다.

# 18. AI 활용 시 주의점

AI를 이용하면 테스트 케이스를 쉽고 빠르게 만들 수 있다.

하지만 수업에서 강조된 주의점은 다음과 같다.

```text
AI를 통해 만든 테스트 케이스를 그대로 포트폴리오에 제출하면 안 된다.
```

AI가 만든 시나리오는 반드시 직접 검토해야 한다.

확인해야 할 내용은 다음과 같다.

```text
이 시나리오가 왜 필요한가?
이 시나리오가 어떤 리스크를 검증하는가?
이 시나리오가 실제 게임 화면과 맞는가?
이 시나리오가 중복되거나 과장되지 않았는가?
면접에서 설명할 수 있는가?
```

면접에서는 “왜 이 테스트 케이스를 만들었는지”를 물어볼 수 있다.
따라서 AI 결과물을 그대로 넣는 것이 아니라, 본인이 이해하고 검수한 뒤 최종 산출물로 만들어야 한다.

---

## AI 활용 시 좋은 방식

```text
1. 게임 이미지를 상세하게 캡처한다.
2. 테스트 케이스 템플릿을 제시한다.
3. 구체적인 지시사항을 입력한다.
4. AI가 만든 결과를 직접 검토한다.
5. 누락된 시나리오, 중복 시나리오, 실제 게임과 맞지 않는 내용을 수정한다.
6. 최종 TC는 본인이 설명할 수 있는 수준으로 정리한다.
```

결과가 마음에 들지 않으면 프롬프트를 수정해야 한다.

# 19. 오늘 수업에서 가져갈 포인트

오늘 수업의 핵심은 Jira와 테스트 케이스 문서가 따로 떨어져 있는 것이 아니라는 점이다.

```text
테스트 케이스에서 Fail 발생
→ BTS/Jira 이슈 등록
→ 요약/설명/환경/재현 스텝 작성
→ 심각도/우선순위 판단
→ 담당자 확인 및 수정
→ 개발/스테이징/라이브 환경에서 QA 확인
→ Closed 처리
```

즉, TC는 테스트 수행을 위한 문서이고, Jira는 발견된 문제를 추적하고 해결하기 위한 도구다.

QA는 단순히 테스트만 수행하는 사람이 아니라, 문제를 재현 가능한 형태로 기록하고, 수정 여부를 확인하며, 최종 반영까지 추적하는 역할을 한다.

# 20. 핵심 정리

```text
Jira의 상태값은 버그의 수명 주기를 관리하기 위한 기준이다.

To Do는 신규 결함이 생성되고 담당자가 할당된 상태다.

In Progress는 개발자가 수정 작업을 진행 중인 상태다.

Dev Resolve와 Staging Resolve는 각각 개발 환경과 스테이징 환경에 수정 사항이 반영된 상태다.

Dev QA와 Staging QA는 QA가 각 환경에서 수정 여부를 확인한 상태다.

Closed는 라이브 환경까지 수정 사항이 정상 반영된 상태다.

By Design은 정상적인 기획 의도다.

Duplicated는 중복 이슈다.

수정하지 않음은 수정 비용이나 리스크보다 유지하는 편이 더 나은 경우다.

정보 부족은 정확한 재현 스텝이 없어 수정이 어려운 경우다.

Jira 요약은 짧지만 문제 위치와 현상이 드러나야 한다.

Jira 설명은 테스트 환경, 상세 현상, 재발생 스텝, 발생 빈도, 기대 결과, 실제 결과로 구성한다.

우선순위는 비즈니스 영향도 기준이다.

심각도는 문제 자체의 심각성과 발생 빈도 기준이다.

TC 포트폴리오는 표지, 대시보드, 테스트 케이스 시트로 구성한다.

테스트 케이스는 150~200개 라인 정도가 적절하다.

UI 검증과 기능 검증의 비율을 고려해야 한다.

하나의 라인에는 하나의 테스트 시나리오만 작성한다.

사전 조건은 Before, 테스트 수행 스텝은 Event, 기대 결과는 After에 해당한다.

AI는 테스트 케이스 초안 작성에는 도움이 되지만, 그대로 제출해서는 안 된다.

최종 테스트 케이스는 본인이 직접 리뷰하고 면접에서 설명할 수 있어야 한다.
```
